## Q101：系统这么多异步环节（HTTP → Kafka → gRPC → DB），出了问题怎么定位是哪一步的？链路追踪怎么做的？

**[进阶] [分布式链路追踪]**

### 回答

当前系统有一条关键异步链路：

```
API 创建 Job → Kafka → Stage 生成 → Kafka → Task 生成 → Kafka → Action 写入 DB
    → Redis 缓存 → UDP Push / Agent Pull → Agent 执行 → gRPC 上报
    → Kafka → 批量聚合 → DB UPDATE → Stage 推进 → 下一轮
```

横跨 **HTTP → Kafka → gRPC → Redis → MySQL** 五种协议，日志里只能看到单点信息，无法关联一个 Job 的完整生命周期。

#### 技术选型：OpenTelemetry + Jaeger

选型理由：
- **OpenTelemetry** 是 CNCF 毕业项目，Go SDK 是官方一级维护，对 gRPC、HTTP、Kafka、SQL 都有成熟的 instrumentation 库
- **Jaeger** 同为 CNCF 毕业项目，与 OTel 无缝对接
- 厂商中立——不绑定特定供应商，后续可换 Tempo、Zipkin 等

#### 核心难点：Kafka 异步链路的上下文传播

HTTP 和 gRPC 的 Trace 传播是自动的（中间件/拦截器注入），但 Kafka 是异步的——Producer 和 Consumer 不在同一进程，需要**手动通过 Kafka Message Headers 传播 SpanContext**：

```go
// 自定义 kafkaHeaderCarrier 实现 OTel 的 TextMapCarrier 接口
type kafkaHeaderCarrier []kafka.Header

func (c *kafkaHeaderCarrier) Get(key string) string { /* 从 Headers 读 */ }
func (c *kafkaHeaderCarrier) Set(key, value string) { /* 向 Headers 写 */ }
func (c *kafkaHeaderCarrier) Keys() []string         { /* 列出所有 Key */ }

// Producer 侧：发送前注入
func InjectToKafka(ctx context.Context, msg *kafka.Message) {
    carrier := kafkaHeaderCarrier(msg.Headers)
    otel.GetTextMapPropagator().Inject(ctx, &carrier)
    msg.Headers = carrier
}

// Consumer 侧：消费时提取
func ExtractFromKafka(ctx context.Context, msg kafka.Message) context.Context {
    carrier := kafkaHeaderCarrier(msg.Headers)
    return otel.GetTextMapPropagator().Extract(ctx, &carrier)
}
```

这样即使一个 Job 跨了 5 个 Kafka Topic，整条链路仍然是一个完整的 Trace。

#### 采样策略

三层采样，兼顾成本和可用性：

| 场景 | 采样率 | 理由 |
|------|--------|------|
| 健康检查、心跳 | 0% | 每 5s 一次心跳 × 6000 Agent = 1200 QPS，全采会爆量 |
| 正常业务链路 | 10% | 控制存储成本 |
| 异常链路（耗时 >5s / ERROR） | 100% | 故障链路绝不能丢 |

实现方式是自定义 `BusinessSampler`，`ParentBased` 确保子 Span 继承父 Span 的采样决定，保持链路完整性。

### 面试加分点

- **追问"为什么不用 SkyWalking？"** → SkyWalking 的 Go SDK 不如 OTel 成熟，Java Agent 很强但 Go 需要手动埋点，选 OTel 更适合 Go 项目
- **追问"Kafka Headers 会影响性能吗？"** → traceparent 只有一个 Header（约 55 字节），相比消息体可忽略不计

## Q102：Kafka 链路追踪的 SpanContext 传播具体怎么实现？跟 HTTP/gRPC 的有什么区别？

**[进阶] [异步链路追踪]**

### 回答

三种协议的 Trace 传播机制对比：

| 协议 | 传播载体 | 实现方式 | 是否自动 |
|------|---------|---------|---------|
| **HTTP** | `traceparent` Header | `otelgin` 中间件自动注入/提取 | ✅ 自动 |
| **gRPC** | gRPC Metadata | `otelgrpc.NewServerHandler()` 拦截器 | ✅ 自动 |
| **Kafka** | Kafka Message Headers | 自定义 `kafkaHeaderCarrier` 手动注入/提取 | ❌ 手动 |

Kafka 的关键区别：

1. **时间差**：Producer 发送和 Consumer 消费可能隔了几秒甚至几分钟，Span 的 startTime 需要在 Consumer 侧重新创建
2. **跨进程**：Producer 和 Consumer 不在同一个进程，甚至不在同一台机器
3. **无回包**：Kafka 没有 Response 的概念，Producer Span 在发送成功后就结束，Consumer 创建新的子 Span

代码流程：

```go
// 1. API Handler 创建根 Span（otelgin 自动）
// ctx 中已有 Span

// 2. Producer: 发消息前注入 TraceContext
msg := kafka.Message{Topic: "job_topic", Key: []byte(jobID), Value: data}
tracing.InjectToKafka(ctx, &msg) // traceparent 写入 msg.Headers
writer.WriteMessages(ctx, msg)

// 3. Consumer: 消费时提取父上下文，创建子 Span
ctx = tracing.ExtractFromKafka(context.Background(), msg)
ctx, span := tracer.Start(ctx, "consumer.ProcessJob",
    trace.WithAttributes(attribute.String("job.id", string(msg.Key))))
defer span.End()
// 后续 DB/Kafka 操作以 ctx 为父，自动成为子 Span
```

最终在 Jaeger UI 里看到的效果：一个 Job 从 API 创建到最终 Agent 上报完成，整条链路在一个 Trace 里，每个 Kafka Topic 的生产/消费都是独立的 Span 对。

## Q103：DB 挂了系统会怎样？怎么防止雪崩？

**[高频] [熔断降级]**

### 回答

先分析雪崩链条：

```
MySQL 响应变慢（主从切换/死锁高发）
→ 6 Worker 每 200ms~1s 扫描 DB → DB 查询超时 5s
→ goroutine 堆积（等待 DB 响应）
→ 连接池耗尽（max_open_connections）
→ 所有后续请求都在等连接
→ gRPC Server 响应变慢（Agent Fetch/Report 超时）
→ Agent 大面积进入重连模式
→ 整个系统雪崩
```

#### 解法：基于 gobreaker 的熔断器

三组独立的熔断器保护三个外部依赖：

| 依赖 | 触发条件 | 熔断时长 | 探测请求数 |
|------|---------|---------|-----------|
| **MySQL** | 连续 5 次失败或失败率 >60% | 30s | 3 |
| **Redis** | 连续 3 次失败或失败率 >50% | 15s | 5 |
| **Kafka** | 连续 5 次失败或失败率 >50% | 60s | 2 |

熔断器状态机：Closed（正常） → Open（熔断，直接拒绝请求） → Half-Open（允许少量探测） → Closed 或回到 Open。

关键在于 **ReadyToTrip 的双条件**：既看连续失败次数（防止突发错误），又看失败率（防止高失败率但被成功请求冲淡）：

```go
ReadyToTrip: func(counts gobreaker.Counts) bool {
    if counts.ConsecutiveFailures >= 5 { return true }
    if counts.Requests >= 10 {
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return failureRatio >= 0.6
    }
    return false
},
```

#### 降级行为

| 熔断对象 | 降级方式 | 用户影响 |
|---------|---------|---------|
| **MySQL** | Worker 跳过本轮扫描，API 返回 503 | Job 创建暂不可用，已运行任务不受影响 |
| **Redis** | Agent Pull 降级为直查 DB | 延迟从 100ms 级增到秒级，但不中断 |
| **Kafka** | Job 创建写入本地 WAL，恢复后重放 | 有延迟但不丢失 |

设计哲学：**Redis 是加速层不是必须层**——这个决定在架构设计之初就考虑了，所以 Redis 熔断时有天然的降级路径。

## Q104：为什么选 gobreaker 而不是 Hystrix-go？熔断器的参数怎么调的？

**[进阶] [技术选型]**

### 回答

| 对比维度 | gobreaker | Hystrix-go |
|---------|-----------|-----------|
| 维护状态 | Sony 维护，活跃 | 已归档（archived），Netflix 停止维护 |
| 依赖量 | 零外部依赖，纯 Go 实现 | 依赖较多 |
| API 复杂度 | 简单直接，一个 `Execute` 搞定 | 概念多（Command Pattern、Thread Pool） |
| 泛型支持 | v2 支持泛型 | 不支持 |
| 功能完整度 | 纯熔断器 | 熔断 + 限流 + 隔离全家桶 |

选 gobreaker 的理由：限流我用 `golang.org/x/time/rate` 单独做，不需要 Hystrix 的全家桶。gobreaker 代码不到 500 行，出问题可以直接看源码。

参数调优思路：

- **MySQL 30s 超时**：MySQL 主从切换通常 10-20s，30s 足够等待恢复
- **Redis 15s 超时**：Redis Sentinel 故障转移通常 5-10s
- **Kafka 60s 超时**：Kafka Broker 选举和 ISR 恢复可能需要 30-60s
- **失败阈值**：MySQL 5 次（允许偶发超时），Redis 3 次（Redis 操作简单，连续 3 次失败说明大概率故障）
- **Half-Open 探测数**：MySQL 3 个（重要依赖多探测几次确认恢复），Kafka 2 个（Kafka 恢复比较确定性）

## Q105：6000 Agent 同时拉任务，Server 怎么扛住？限流怎么做的？

**[高频] [限流策略]**

### 回答

限流分三层：

#### 第一层：HTTP API（对外）

用 Gin 中间件 + 令牌桶：

```go
// 全局限流
router.Use(ratelimit.NewGlobalLimiter(100, 200)) // 100 QPS, burst 200

// CreateJob 单独限流（成本高：事务 + Kafka 投递）
jobGroup.Use(ratelimit.NewGlobalLimiter(10, 20)) // 10 QPS, burst 20
```

100 QPS 全局限流是因为管控平台不是高频 API；CreateJob 10 QPS 是因为每次创建涉及 DB 事务 + Kafka 投递 + Stage/Task/Action 生成，成本高。

#### 第二层：gRPC（Agent 侧）

用服务端一元拦截器：

```go
grpcServer := grpc.NewServer(
    grpc.UnaryInterceptor(ratelimit.UnaryServerInterceptor(
        rate.NewLimiter(2000, 3000), // CmdFetch: 2000 QPS, burst 3000
    )),
)
```

参数依据：6000 Agent 每 5s 拉一次 = 1200 QPS，2000 QPS 留了约 2 倍余量。如果开启心跳捎带优化，Fetch 频率降低到 1000 QPS 以下。

#### 第三层：客户端退避

限流返回 gRPC `ResourceExhausted`，Agent 收到后进入指数退避重试（0.5s → 1s → 2s），跟网络超时的重试策略共用同一套逻辑，不需要额外代码。

#### 限流参数汇总

| 限流点 | QPS | Burst | 设计依据 |
|-------|-----|-------|---------|
| HTTP 全局 | 100 | 200 | 管控平台非高频 |
| CreateJob | 10 | 20 | 单次创建成本高 |
| gRPC CmdFetch | 2000 | 3000 | 6000 Agent × 0.2/s = 1200 QPS + 余量 |
| gRPC CmdReport | 5000 | 8000 | 批量上报突发高 |
| gRPC Heartbeat | 2000 | 2000 | 6000 × 0.2/s = 1200 QPS |

## Q106：为什么用令牌桶不用滑动窗口？IP 限流怎么做的？

**[进阶] [限流算法]**

### 回答

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **令牌桶（✅ 采用）** | 允许突发、实现简单、Go 标准库 | 需要选 rate 和 burst 两个参数 | 允许一定突发的 API 限流 |
| 滑动窗口 | 精确控制窗口内请求数 | 实现复杂、不允许突发 | 严格的计费限流 |
| 漏桶 | 平滑输出 | 突发被丢弃、不适合 API | 消息队列消费 |
| 固定窗口 | 最简单 | 窗口边界突发问题 | 简单统计 |

选令牌桶的核心原因：
1. `golang.org/x/time/rate` 是 Go 官方维护的标准库，不需要引入第三方依赖
2. 允许突发（burst 参数）——Agent 重启后会集中拉取，需要允许短暂的突发流量
3. 令牌桶天然适合 API 限流场景

IP 限流用 `sync.Map` 为每个 IP 维护独立的限流器：

```go
type IPRateLimiter struct {
    limiters sync.Map // map[string]*rate.Limiter
    rate     rate.Limit
    burst    int
}

func (l *IPRateLimiter) getLimiter(ip string) *rate.Limiter {
    if v, ok := l.limiters.Load(ip); ok {
        return v.(*rate.Limiter)
    }
    limiter := rate.NewLimiter(l.rate, l.burst)
    l.limiters.Store(ip, limiter)
    return limiter
}
```

注意事项：`sync.Map` 不会自动清理过期条目，生产环境应加 TTL 清理机制（定时遍历删除长时间未访问的 limiter），避免内存泄漏。

## Q107：Agent 怎么找到 Server？Server 扩容了 Agent 能自动感知吗？

**[中频] [服务发现]**

### 回答

当前 Agent 是硬编码 IP 连接 Server，问题很明显：扩缩容需要改 6000 台 Agent 的配置。

#### 方案：DNS-Based 服务发现

Kubernetes 环境下用 Headless Service + gRPC 内置 DNS resolver：

```
Server 部署为 StatefulSet + Headless Service
→ woodpecker-server-0.woodpecker-server.default.svc.cluster.local
→ woodpecker-server-1.woodpecker-server.default.svc.cluster.local

Agent 配置 Service DNS 名（而非 IP）
→ woodpecker-server.default.svc.cluster.local:9090
```

Agent 端只需改一行连接代码：

```go
conn, _ := grpc.Dial(
    "dns:///woodpecker-server.default.svc.cluster.local:9090",
    grpc.WithDefaultServiceConfig(`{"loadBalancingConfig": [{"round_robin": {}}]}`),
)
```

`dns:///` 前缀告诉 gRPC 使用 DNS resolver，自动解析为所有 Pod IP，round-robin 负载均衡。Server 扩缩容对 Agent 完全透明——K8s 更新 DNS 记录，gRPC resolver 自动刷新。

#### 为什么不用 etcd/Consul？

对我们的场景来说 DNS-based 够用：
- K8s DNS 是零额外组件（CoreDNS 已内置）
- 更新延迟（DNS TTL）对非实时场景可接受
- 不值得为服务发现多一个组件的运维成本

如果是非 K8s 环境，用 manual resolver + 配置文件多地址作为降级方案：

```ini
# agent.ini
[server]
addresses = 10.0.1.10:9090,10.0.1.11:9090,10.0.1.12:9090
```

## Q108：配置改了要重启服务吗？配置热加载怎么做的？

**[中频] [配置中心]**

### 回答

不需要重启。实现了基于 **fsnotify 文件监听 + atomic.Value 原子更新** 的轻量热加载方案：

```go
type HotConfig struct {
    value  atomic.Value // 存储 *RuntimeConfig，无锁读取
    path   string
    logger *zap.Logger
}

// 读取：atomic.Load，无锁
func (hc *HotConfig) Get() *RuntimeConfig {
    return hc.value.Load().(*RuntimeConfig)
}

// 文件变更监听：fsnotify
func (hc *HotConfig) watchLoop() {
    watcher, _ := fsnotify.NewWatcher()
    watcher.Add(hc.path)
    for event := range watcher.Events {
        if event.Op&fsnotify.Write == fsnotify.Write {
            cfg, err := loadRuntimeConfig(hc.path)
            if err != nil { continue } // 加载失败不更新，保留旧配置
            hc.value.Store(cfg) // 原子替换
        }
    }
}
```

支持热更新的参数包括：限流阈值、超时时间、熔断器参数、链路追踪采样率、Action 加载间隔等。

#### 设计要点

1. **读无锁**：`atomic.Value.Load()` 是 lock-free 的，不影响业务请求性能
2. **加载失败不更新**：解析出错时保留旧配置，不会因为配置写错导致服务异常
3. **不用 Apollo/Nacos**：对面试展示项目来说引入太重了；生产环境建议接入公司配置中心，用 etcd watch 实时推送

### 追问应对

**"多实例配置怎么同步？"** → 多实例场景下，方案一：共享存储（NFS 挂载同一个 INI 文件）；方案二：ConfigMap + Volume Mount（K8s 原生，ConfigMap 更新自动同步到所有 Pod）；方案三：如果需要灰度配置，升级到配置中心。

## Q109：滚动升级时怎么保证不丢任务？优雅关闭的顺序是什么？

**[高频] [优雅上下线]**

### 回答

#### Server 优雅关闭顺序

收到 SIGTERM 后，按**从外到内**的固定顺序关闭：

```
1. HTTP Server.Shutdown()     — 停止接收新 HTTP 请求，等待在途请求完成（最多 10s）
2. gRPC GracefulStop()        — 停止接收新 RPC，等待在途 RPC 完成
3. Dispatcher.Stop()          — 停止调度引擎（等当前 Worker tick 完成）
4. Election.Stop()            — 释放 Redis Leader 锁（让 Standby 快速接管）
5. KafkaModule.Stop()         — 关闭 Consumer（commit offset）+ Producer（flush 缓冲区）
6. DBModule.Stop()            — 关闭 DB 连接池
7. TracingShutdown()          — 刷出 OpenTelemetry traces
```

**第 4 步是关键**：主动释放 Leader 锁后，Standby 实例立即接管（而不是等 30s TTL 过期）。否则集群会有 30s 的调度空窗期。

#### Agent 优雅关闭

```
1. 停止 fetchLoop（不再拉新任务）
2. 等待 WorkPool 中正在执行的 Action 完成（最多 60s）
3. resultQueue 中未上报的结果写入 WAL 文件
4. 刷出 DedupModule 的 finished_set 到磁盘
5. 关闭 gRPC 连接
```

WAL 文件保证了即使 Agent 关闭时有未上报的结果，重启后也能回放上报。

#### K8s 配置

```yaml
terminationGracePeriodSeconds: 30
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 5"]
```

preStop 加 5s sleep 的原因：SIGTERM 和 Service Endpoints 更新是异步的——如果不 sleep，Endpoints 还没更新完，新请求还在打到正在关闭的 Pod 上。sleep 5s 确保 Endpoints 更新完毕再开始关闭。

## Q110：错误码怎么设计的？为什么不直接用 HTTP 状态码？

**[中频] [错误处理]**

### 回答

HTTP 状态码只有几十个，表达不了业务语义。比如 500 Internal Server Error——到底是 DB 超时、Kafka 发送失败、还是代码 bug？客户端拿到 500 什么都判断不了。

#### 5 位数字错误码

```
格式：XYYZZ
X  = 错误类别  1=客户端错误  2=服务端错误  3=外部依赖错误  4=限流/熔断
YY = 模块编号  10=API  20=调度  30=Agent通信  40=存储  50=配置
ZZ = 具体序号
```

示例：

| 错误码 | 含义 | HTTP 状态码 |
|--------|------|------------|
| 11001 | 请求参数无效 | 400 |
| 11002 | Job 不存在 | 400 |
| 21001 | 内部服务器错误 | 500 |
| 22001 | 调度失败 | 500 |
| 34001 | 数据库连接失败 | 503 |
| 42900 | 请求被限流 | 429 |
| 45030 | 熔断器打开 | 503 |

#### 错误码到 HTTP 状态码的映射

```go
func mapToHTTPStatus(code int) int {
    switch code / 10000 {
    case 1: return 400  // 客户端错误
    case 2: return 500  // 服务端错误
    case 3: return 503  // 外部依赖错误
    case 4: return 429  // 限流/熔断
    }
}
```

#### 统一响应格式

```json
// 成功
{"code": 0, "message": "success", "data": {...}}

// 错误
{"code": 34001, "message": "database connection failed"}
```

好处：
1. 客户端可以根据错误码精确判断错误类型和重试策略
2. 监控告警可以按错误码分类统计
3. 错误码文档化后，前后端对齐高效

## Q111：系统的可观测性怎么做的？Metrics、Logging、Tracing 三者之间怎么关联？

**[进阶] [可观测性]**

### 回答

三支柱现状：

| 支柱 | 工具 | 完成度 | 说明 |
|------|------|--------|------|
| **Metrics** | Prometheus + Grafana | 80% | step-8 已定义 13+ 业务指标 |
| **Logging** | Logrus/Zap → Filebeat → ELK | 70% | 日志采集链路已设计 |
| **Tracing** | OpenTelemetry + Jaeger | 本轮补齐 | 链路追踪完整方案 |

三者孤立的问题：Grafana 看到延迟飙升 → 不知道哪个请求 → 打开 Jaeger → 不知道搜什么 → 打开 Kibana → 不知道哪条日志关联。

#### 关联方案一：Trace-Log 关联

在日志中注入 traceId，实现 Jaeger ↔ ELK 互跳：

```go
func LogWithTrace(ctx context.Context, logger *zap.Logger) *zap.Logger {
    span := trace.SpanFromContext(ctx)
    if !span.SpanContext().IsValid() { return logger }
    return logger.With(
        zap.String("traceId", span.SpanContext().TraceID().String()),
        zap.String("spanId", span.SpanContext().SpanID().String()),
    )
}

// 日志输出：{"traceId":"abc123","spanId":"789xyz","msg":"processing job event","jobId":"job_001"}
```

效果：Kibana 按 traceId 搜索 → 看到该链路所有环节的日志；Jaeger 点击"View Logs"→ 跳转到 Kibana 对应 traceId 的日志视图。

#### 关联方案二：Prometheus Exemplar（Metric-Trace 关联）

```go
// Histogram 指标携带 traceID
histogram.(prometheus.ExemplarObserver).ObserveWithExemplar(
    value,
    prometheus.Labels{"traceID": span.SpanContext().TraceID().String()},
)
```

效果：Grafana 中看到延迟异常点 → 鼠标悬停看到 Exemplar 中的 traceID → 点击直接跳转 Jaeger 查看该 Trace 的完整链路。

#### 排障流程

```
Grafana 延迟告警
→ 看到 P99 飙升到 5s
```

## Q112：SLA 指标是怎么定义的？达不到怎么办？

**[中频] [SLA 设计]**

### 回答

定义了 6 个核心 SLA 指标：

| SLA 指标 | 目标 | 计算方式 | 告警阈值 |
|---------|------|---------|---------|
| Job 成功率 | ≥ 99.5% | success_jobs / total_jobs | < 99% → P0 |
| Action 下发延迟 | P99 < 200ms | Action 写入 DB → Agent 拉取 | P99 > 500ms → P1 |
| 端到端延迟 | P95 < 60s | Job 创建 → 最后一个 Action 完成 | P95 > 120s → P1 |
| Agent 在线率 | ≥ 99.9% | online_agents / total_agents | < 99% → P0 |
| API 可用性 | ≥ 99.95% | successful_requests / total_requests | < 99.9% → P0 |
| gRPC 可用性 | ≥ 99.9% | successful_rpcs / total_rpcs | < 99.5% → P1 |

#### 三级降级策略

| 级别 | 触发场景 | 处理方式 | 用户影响 |
|------|---------|---------|---------|
| **L1 轻微** | Redis 不可用、Tracing 后端挂 | 自动降级，系统自愈 | 无感知或轻微延迟 |
| **L2 中等** | MySQL 慢查询、Kafka 故障、部分 Agent 离线 | 熔断器介入，任务排队 | 部分功能受限 |
| **L3 严重** | MySQL 完全不可用、多组件同时故障 | 人工介入，P0 告警 | 核心功能不可用 |

设计哲学：**核心链路和辅助链路分开**——Redis、Tracing 是辅助的，挂了不影响核心功能；MySQL 是核心的，挂了必须有熔断保护和告警。Agent 本地有 WAL + 缓存，即使 Server 全挂，已下发的任务继续执行，恢复后上报结果。

## Q113：Gin 的全局错误恢复怎么做的？panic 了服务会挂吗？

**[基础] [错误处理]**

### 回答

不会挂。Gin 的 Recovery 中间件能捕获 handler 中的 panic，返回错误响应而不是让整个进程崩溃：

```go
func Recovery(logger *zap.Logger) gin.HandlerFunc {
    return func(c *gin.Context) {
        defer func() {
            if r := recover(); r != nil {
                logger.Error("panic recovered",
                    zap.Any("error", r),
                    zap.String("path", c.Request.URL.Path),
                    zap.String("stack", string(debug.Stack())),
                )
                errors.Error(c, errors.New(errors.ErrInternalServer, "internal server error"))
            }
        }()
        c.Next()
    }
}
```

关键点：
1. `recover()` 捕获 panic，记录完整堆栈到日志
2. 返回统一的 500 错误码（不暴露内部信息给客户端）
3. **只影响当前请求的 goroutine**——其他请求不受影响，服务继续运行

注意：panic recovery 只是最后一道防线。正确做法是在业务代码中 **主动 check error**，而不是依赖 recover。panic 应该只在"不可能发生的情况"（如空指针、数组越界）时触发，正常的业务错误应该通过 error 返回值传递。

## Q114：API 版本怎么管理？gRPC Proto 怎么保证向后兼容？

**[中频] [API 版本管理]**

### 回答

#### HTTP API 版本管理

采用 URL 路径版本：`/api/v1/*`、`/api/v2/*`

规则：
1. 新增字段不破坏旧客户端（向后兼容）
2. 不兼容变更走新版本号（v1 → v2）
3. 弃用 API 通过 Response Header `Deprecation: true` 提前通知
4. 最多同时维护 2 个版本（当前版本 + 上一版本）

#### gRPC Proto 向后兼容

```protobuf
message HeartbeatResponse {
    bool accepted = 1;
    bool has_actions = 2;          // v1.1 新增
    repeated string action_ids = 3; // v1.2 新增
    // reserved 4;                  // 保留已弃用的字段编号
}
```

规则：
1. 只添加新字段，不删除/重命名已有字段
2. 新字段默认值 = 零值（Proto3 特性），旧客户端不解析新字段也不会出错
3. 已弃用的字段编号用 `reserved` 保留，防止将来复用导致兼容问题

#### Agent-Server 版本协商

Agent 连接时上报自己的版本号，Server 根据版本决定返回格式：

```go
func (s *HeartbeatService) Heartbeat(ctx context.Context, req *pb.HeartbeatRequest) (*pb.HeartbeatResponse, error) {
    if semver.Compare(req.AgentVersion, "v1.2.0") >= 0 {
        // v1.2+ Agent 支持心跳捎带 Action ID 列表
        resp.ActionIds = s.getPendingActionIds(req.Hostname)
    } else {
        // 旧版 Agent 只支持 has_actions 布尔标记
        resp.HasActions = s.hasPendingActions(req.Hostname)
    }
    return resp, nil
}
```

好处：新旧 Agent 共存时服务不中断，支持灰度升级。

## Q115：如果让你从零搭建这套服务治理体系，优先级怎么排？

**[高频] [架构思维]**

### 回答

基于投入产出比，分三批：

#### P0 — 上线前必须有（防止线上事故）

| 能力 | 理由 | 预估工时 |
|------|------|---------|
| **熔断器** | 没有熔断器，任何一个依赖故障都可能雪崩 | 2天 |
| **优雅上下线** | 没有优雅关闭，滚动升级必丢消息 | 1天 |
| **统一错误码** | 没有错误码，排障全靠猜 | 1天 |

这三个是"**没有会出事**"的能力。

#### P1 — 上线后第一个月补齐（提升运维效率）

| 能力 | 理由 | 预估工时 |
|------|------|---------|
| **分布式链路追踪** | 异步链路排障效率从小时级降到分钟级 | 3天 |
| **限流** | Agent 规模上来后的保护伞 | 1天 |
| **可观测性整合** | Metrics-Log-Trace 三者关联，排障闭环 | 2天 |

这三个是"**有了效率翻倍**"的能力。

#### P2 — 稳定运行后逐步完善

| 能力 | 理由 | 预估工时 |
|------|------|---------|
| **服务发现** | K8s 环境下用 DNS 就够，非 K8s 用多地址配置 | 1天 |
| **配置热加载** | 当前规模改配置重启也能接受 | 1天 |
| **API 版本管理** | 客户端少的时候直接协调升级 | 0.5天 |
| **SLA 定义** | 有了指标体系后再量化 | 0.5天 |

这三个是"**有了更好但没有也不致命**"的能力。

#### 总结

> "服务治理不是一口气全搞完的——P0 先搞好熔断和优雅关闭，确保系统不雪崩、升级不丢数据；P1 补齐链路追踪和限流，让运维效率上台阶；P2 根据实际需求逐步完善。每个阶段都有明确的止损点，而不是做一个大而全的治理框架。"
