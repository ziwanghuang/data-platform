# 八、微服务拆分与架构演进（Q85-93）

> 本文档覆盖面试问题 85-93，聚焦从单体架构到微服务的拆分设计、服务边界划分、通信模式选型、数据治理、部署策略和迁移方案。

## Q85：从单体到微服务，你拆了哪些服务？每个服务的职责是什么？

**[中等] [微服务设计]**

### 回答

将原有的 `woodpecker-server` 单体拆分为 **6 个微服务 + 1 个 Agent**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    TBDS 管控平台微服务架构                             │
│                                                                      │
│  ① ctrl-gateway        API 网关，用户请求入口                         │
│  ② ctrl-orchestrator   流程编排引擎，Job/Stage/Task/Action 生成       │
│  ③ ctrl-dispatcher     指令下发服务，Action 写入 DB+Redis+通知 Agent  │
│  ④ ctrl-bridge         通信桥接服务，Agent gRPC 网关                  │
│  ⑤ ctrl-tracker        状态追踪服务，结果聚合+进度检测+Stage 推进     │
│  ⑥ ctrl-guardian       守护服务，超时检测+失败重试+冷热分离+补偿扫描  │
│  ⑦ ctrl-agent          节点执行器，每个节点一个                       │
│                                                                      │
│  命名规范：ctrl- 前缀（Control 缩写）+ 英文职责名词                   │
└─────────────────────────────────────────────────────────────────────┘
```

**拆分总览**：

| 序号 | 服务名 | 语言 | 端口 | 核心职责 | 实例数 |
|------|--------|------|------|---------|--------|
| ① | ctrl-gateway | Go | 8080 | HTTP API、鉴权限流、请求路由、响应聚合 | 2~3 |
| ② | ctrl-orchestrator | Go | 8081 | Job 创建、Stage 编排、Task 生成、Action 生成 | 2~3 |
| ③ | ctrl-dispatcher | Go | 8082 | Action 批量写入 DB、Redis 加载、UDP 通知 Agent | 3~5 |
| ④ | ctrl-bridge | Go | 9090 | gRPC 服务（CmdFetch/CmdReport/HeartBeat） | 3~5 |
| ⑤ | ctrl-tracker | Go | 8083 | Action 结果聚合、Task/Stage/Job 进度检测 | 2~3 |
| ⑥ | ctrl-guardian | Go | 8084 | 超时检测、失败重试、冷热分离、补偿扫描 | 1~2 |

### 为什么要拆分？

原有 `woodpecker-server` 是一个**巨型单体**，存在 5 个核心问题：

| 问题 | 说明 |
|------|------|
| **扩展性差** | 只有 Leader 执行调度，增加实例无法提升吞吐 |
| **故障域大** | 任何一个模块 OOM，整个 Server 不可用 |
| **资源浪费** | CPU 密集型（编排）和 IO 密集型（gRPC）混在一起，无法独立扩缩容 |
| **发布耦合** | 修改 API 接口需要重新部署整个 Server，影响正在执行的调度任务 |
| **开发效率低** | 所有功能在一个代码库，编译慢、测试慢、冲突多 |

拆分后，每个服务可以**独立部署、独立扩缩容、独立发布**，最关键的是实现了**故障隔离**——比如 dispatcher 的 DB 写入出现慢查询不会影响 bridge 的 Agent 通信。

## Q86：ctrl-gateway、ctrl-orchestrator、ctrl-dispatcher、ctrl-bridge、ctrl-tracker、ctrl-guardian 分别做什么？

**[中等] [服务边界]**

### 回答

逐个展开每个服务的详细职责：

### ① ctrl-gateway — API 网关

| 属性 | 值 |
|------|---|
| **框架** | Gin |
| **资源特征** | IO 密集型（接收请求、转发） |

**职责**：
1. **接收用户 HTTP 请求**（创建 Job、查询状态、取消 Job）
2. **鉴权与权限校验**（Token 验证、操作权限检查）
3. **请求限流与熔断**（防止突发流量打垮后端服务）
4. **请求路由**（将请求转发到对应的后端服务）
5. **响应聚合**（查询 Job 状态时聚合多个服务的数据）

**对外接口**：
```
POST   /api/v1/jobs                  → 创建 Job（转发到 orchestrator）
GET    /api/v1/jobs/{jobId}          → 查询 Job（聚合 orchestrator + tracker）
POST   /api/v1/jobs/{jobId}/cancel   → 取消 Job（转发到 orchestrator）
GET    /api/v1/jobs/{jobId}/stages   → Stage 列表（转发到 tracker）
GET    /api/v1/jobs/{jobId}/actions  → Action 列表（转发到 tracker）
```

**拆分理由**：发布解耦（修改 API 不影响调度）、安全边界集中（鉴权/限流/审计）、灰度发布支持、协议转换预留（未来 gRPC-Web、GraphQL）。

### ② ctrl-orchestrator — 流程编排引擎

| 属性 | 值 |
|------|---|
| **框架** | Gin + Kafka Consumer |
| **资源特征** | CPU 密集型（模板解析、Action 生成） |

**职责**：
1. **Job 创建**：根据 JobCode 查找流程模板，生成 Job + Stage 列表
2. **Stage 编排**：消费 Kafka `stage_topic`，为 Stage 生成 Task
3. **Task 生成**：消费 Kafka `task_topic`，调用 TaskProducer 生成 Action
4. **Action 投递**：将 Action 按 IP 哈希分片投递到 `action_batch_topic`
5. **流程模板管理**：维护 ProcessTemplate 和 TaskProducer 注册表

**消费**：job_topic(4)、stage_topic(8~16)、task_topic(16~32)
**生产**：action_batch_topic(16~32, IP 哈希)、stage_topic(8~16)

**拆分理由**：CPU 密集型操作（6000 节点 × 50 Action = 30 万 Action 生成），需要独立计算资源。多实例组成 Consumer Group 自动负载均衡。编排逻辑独立演进（未来可替换 Activiti）。

### ③ ctrl-dispatcher — Action 下发服务

| 属性 | 值 |
|------|---|
| **框架** | Kafka Consumer |
| **资源特征** | IO 密集型（DB 批量写入 + Redis 写入） |

**职责**：
1. **Action 批量写入 DB**：消费 `action_batch_topic`，每批 200 条 INSERT
2. **Action 加载到 Redis**：ZADD 到以 `hostuuid` 为 Key 的 Sorted Set
3. **通知 Agent**：Action 写入 Redis 后通过 UDP 通知对应节点
4. **补偿加载**：定时检查 state=cached 但 Redis 中不存在的 Action

**拆分理由**：**系统最大瓶颈**。6000 节点 × 50 Action = 30 万 Action 写入 DB + Redis。拆分后多实例按 IP 分片并行写入，从分钟级降至秒级。独立扩缩容，突发大量 Action 时可临时增加实例。

### ④ ctrl-bridge — Agent 通信网关

| 属性 | 值 |
|------|---|
| **框架** | gRPC Server |
| **资源特征** | IO 密集型（网络 IO 为主） |

**职责**：
1. **任务拉取**：CmdFetchChannel — 从 Redis 读取 Action 列表返回
2. **结果上报**：CmdReportChannel — 接收结果投递到 Kafka `action_result_topic`
3. **心跳接收**：HeartBeatV2 — 写入 Redis Hash（TTL=30s）
4. **连接管理**：gRPC 连接健康检查

**拆分理由**：承受 6000 节点 × 10 次/s = **6 万 QPS**，是并发最高的服务。完全无状态（从 Redis 读、往 Kafka 写），可任意水平扩展。结果上报从同步写 DB 改为投递 Kafka 后立即返回，大幅降低延迟。

### ⑤ ctrl-tracker — 状态追踪服务

| 属性 | 值 |
|------|---|
| **框架** | Kafka Consumer + Gin |
| **资源特征** | IO 密集型（DB 批量更新 + 进度计算） |

**职责**：
1. **Action 结果聚合**：消费 `action_result_topic`，200ms 窗口内存聚合，批量 UPDATE
2. **Task 进度检测**：消费 `task_status_detect_topic`，统计 Action 完成度
3. **Stage 进度检测**：消费 `stage_status_detect_topic`，统计 Task 完成度
4. **Stage 推进**：当前 Stage 完成后投递下一个 Stage 到 `stage_topic`
5. **状态查询**：提供 HTTP 接口供 gateway 查询状态

**拆分理由**：批量聚合优化——原来逐条 UPDATE（10 万次/批），拆分后内存聚合 200ms 窗口批量 UPDATE（数百次/批），DB 压力降低 99%+。进度检测从"定时扫描"变为"事件触发"，延迟从秒级降至毫秒级。

### ⑥ ctrl-guardian — 守护服务

| 属性 | 值 |
|------|---|
| **框架** | Gin + 定时任务 |
| **资源特征** | 低资源消耗 |

**职责**：
1. **超时检测**：每 10s 扫描 state=executing 且超过 120s 的 Action
2. **失败重试**：每 30s 扫描 retryCount < retryLimit 的 Task
3. **冷热分离**：每天凌晨迁移已完成 Action 到归档表
4. **补偿扫描**：每 5~10s 扫描超时未完成的 Stage/Task，重新投递 Kafka
5. **节点管理**：离线检测、节点信息查询

**拆分理由**：低优先级后台任务，不在核心路径上。冷热分离涉及大量数据迁移可能导致 DB 锁表，独立后不影响核心链路。单实例 + 分布式锁即可。

## Q87：拆分的依据是什么？怎么确定服务边界的？

**[进阶] [领域划分]**

### 回答

拆分依据是**基于数据流分析**的方法论，结合四个维度判断：

### 方法论：追踪核心数据流

```
用户请求 ──→ [API 网关] ──→ 创建 Job
                              │
                              ▼
                        [流程编排引擎] ──→ 生成 Stage → Task → Action
                              │
                              ▼ (Kafka)
                        [Action 下发服务] ──→ DB → Redis → 通知 Agent
                              │
                              ▼ (gRPC)
                        [Agent 通信网关] ←──→ Agent（拉取/上报/心跳）
                              │
                              ▼
                        [状态聚合服务] ──→ 批量更新 DB → 检测进度 → 触发下一 Stage
```

每个 `[...]` 就是一个**天然的服务边界**。数据流在这些节点之间有明确的输入输出、可以通过消息队列或 RPC 解耦。

### 四个判断维度

**维度一：资源特征差异**

| 服务 | 资源特征 | 与其他服务的差异 |
|------|---------|----------------|
| orchestrator | **CPU 密集型**（模板解析、Action 生成） | 需要更多 CPU |
| bridge | **IO 密集型**（6 万 QPS 网络 IO） | 需要更多网络带宽 |
| dispatcher | **IO 密集型**（DB+Redis 写入） | 需要更多 DB 连接 |

不同资源特征的模块混在一起无法独立调参和扩缩容。CPU 密集型需要多 CPU、IO 密集型需要多连接和带宽，混合部署会互相干扰。

**维度二：扩展性需求不同**

- bridge 需要 3~5 个实例应对 6 万 QPS → **需要水平扩展**
- dispatcher 需要 3~5 个实例并行分片写入 → **需要水平扩展**
- guardian 只需 1~2 个实例（低频后台任务） → **不需要多实例**

如果混在一起，为了扩展 bridge 必须同时扩展 guardian，造成资源浪费。

**维度三：故障隔离需求**

关键场景：如果 dispatcher 的 DB 写入出现慢查询或死锁：
- **拆分前**：整个 Server 都受影响，包括 Agent 的心跳上报、gRPC 通信
- **拆分后**：只有 Action 写入受影响，Agent 通信（bridge）、状态追踪（tracker）正常运行

特别是 guardian 的冷热分离涉及大量 DELETE 操作，可能导致 DB 锁竞争。独立后即使它出问题也不影响核心调度链路。

**维度四：发布频率和独立性**

- API 层（gateway）变更最频繁（接口迭代、鉴权策略调整）
- 编排逻辑（orchestrator）变更中频（新增流程模板、TaskProducer）
- 通信层（bridge）变更最少（gRPC 协议稳定）

独立部署后修改 API 不需要重启整个调度引擎，降低发布风险。

### 遵循的拆分原则

| 原则 | 说明 | 应用 |
|------|------|------|
| **单一职责** | 每个服务只做一件事 | 编排 ≠ 下发 ≠ 通信 ≠ 追踪 |
| **数据自治** | 每个服务拥有自己的数据 | Action 表归 dispatcher 管 |
| **按业务边界拆分** | 而非按技术层 | 不是拆成"DB 层"+"缓存层"+"API 层" |
| **高内聚低耦合** | 服务内部高度相关 | 编排和下发通过 Kafka 解耦 |
| **独立部署** | 可独立发布、扩缩容 | bridge 独立扩容应对 Agent 高并发 |
| **渐进式拆分** | 不一步到位 | 先拆最痛的点，再拆其他 |

## Q88：服务间用什么方式通信？同步（gRPC）和异步（Kafka）各用在什么场景？

**[中等] [通信模式]**

### 回答

### 通信拓扑总览

```
ctrl-gateway ──HTTP──→ ctrl-orchestrator（创建/取消 Job）
ctrl-gateway ──HTTP──→ ctrl-tracker（查询状态）
ctrl-gateway ──HTTP──→ ctrl-guardian（查询节点）

ctrl-orchestrator ──Kafka──→ ctrl-dispatcher（Action 批量写入）
ctrl-orchestrator ──Kafka──→ ctrl-orchestrator（Stage/Task 自消费）

ctrl-dispatcher ──UDP──→ ctrl-agent（通知有新任务）

ctrl-agent ──gRPC──→ ctrl-bridge（拉取/上报/心跳）

ctrl-bridge ──Kafka──→ ctrl-tracker（Action 结果）

ctrl-tracker ──Kafka──→ ctrl-orchestrator（触发下一 Stage）

ctrl-guardian ──Kafka──→ ctrl-orchestrator（补偿触发）
```

### 三种通信方式的选型逻辑

| 通信场景 | 方式 | 选型理由 |
|---------|------|---------|
| gateway → 后端服务 | **HTTP 同步** | 用户请求需要同步响应，延迟要求 <500ms |
| orchestrator → dispatcher | **Kafka 异步** | Action 生成是批量操作，异步解耦提升吞吐量 |
| agent → bridge | **gRPC 同步** | 任务拉取需要实时响应，gRPC 高效序列化 |
| bridge → tracker | **Kafka 异步** | 结果上报不需要同步等待 DB 更新 |
| tracker → orchestrator | **Kafka 异步** | Stage 推进是事件驱动，天然适合消息队列 |
| dispatcher → agent | **UDP 单向通知** | 仅通知"有新任务"，不需要可靠传输 |

### 核心决策原则

**同步（HTTP/gRPC）的使用场景**：

1. **用户面对的请求**——gateway 接收用户的创建 Job 请求后需要立即返回 jobId，不能"投递到队列后告诉用户去查"
2. **实时性要求高**——Agent 拉取任务需要立即拿到 Action 列表，不能等消息队列投递
3. **请求-响应模式**——调用方需要知道操作是否成功

**异步（Kafka）的使用场景**：

1. **批量操作**——一次 Job 生成数万个 Action，逐个同步调用 dispatcher 会很慢
2. **解耦生产和消费速率**——orchestrator 生成 Action 的速度和 dispatcher 写入 DB 的速度可能不匹配，Kafka 做缓冲
3. **一对多扇出**——tracker 完成 Stage 后既要通知 orchestrator（生成下一 Stage）又要更新 Job 状态
4. **容错**——Kafka 消息持久化，即使消费者临时不可用，消息不丢失

**UDP 的特殊场景**：

- 仅用于 dispatcher → agent 的"有新任务"通知
- 通知包极小（几十字节），不需要确认——Agent 收到就立即拉取，没收到有 2s 定时轮询兜底
- 5000+ 节点场景下 UDP 不需要维护连接状态，资源开销最低

### 为什么不全用 gRPC？为什么不全用 Kafka？

**全用 gRPC 的问题**：
- orchestrator → dispatcher 如果用 gRPC 同步调用，30 万 Action 需要逐批同步写入，orchestrator 被 dispatcher 的 DB 写入速度拖慢
- 服务间耦合度高——dispatcher 挂了直接影响 orchestrator

**全用 Kafka 的问题**：
- gateway → orchestrator 如果用 Kafka，用户发起创建 Job 后无法立即拿到 jobId
- agent → bridge 如果用 Kafka，任务拉取延迟增加（多一跳消息队列），Agent 无法立即拿到 Action

**结论**：同步和异步各有适用场景，关键看两点——**调用方是否需要立即拿到结果**，以及**生产和消费速率是否匹配**。

## Q89：数据所有权怎么划分的？哪些数据归哪个服务管？

**[中等] [数据治理]**

### 回答

### 数据库拆分策略

采用**逻辑拆分**（同一个 MySQL 实例，不同的 Schema/权限），而非物理拆分：

| 服务 | 拥有的表 | 读写权限 |
|------|---------|---------|
| **ctrl-orchestrator** | job, stage, task, host, cluster | 读写 |
| **ctrl-dispatcher** | action（创建和 init→cached 状态） | 读写 |
| **ctrl-bridge** | — | Redis 读(Action)、Redis 写(心跳)、MySQL 读(Action 详情) |
| **ctrl-tracker** | action（状态更新）, task/stage/job（进度更新） | 读写 |
| **ctrl-guardian** | action_archive, host | 读写 |

### 关键问题：Action 表被多个服务访问

Action 表是系统最核心的"热表"，被 3 个服务访问：

```
ctrl-dispatcher  → INSERT Action（创建）、UPDATE state: init→cached
ctrl-bridge      → SELECT Action（Agent 拉取时查询详情）
ctrl-tracker     → UPDATE Action state: processing→success/failed
ctrl-guardian    → DELETE Action（冷热分离归档）
```

### 共享数据的处理策略（三阶段演进）

| 方案 | 说明 | 适用阶段 |
|------|------|---------|
| **共享数据库** | 多个服务直接访问同一张 Action 表 | 初期（快速落地） |
| **数据同步** | 通过 Kafka 事件同步关键数据到各服务本地表 | 中期（数据量增大） |
| **API 调用** | 通过服务间 API 查询数据 | 长期（完全解耦） |

**初期策略（推荐）**：共享同一个 MySQL 实例，通过**读写权限**控制数据归属。虽然不是"教科书式"的每个服务独占数据库，但在团队规模小、业务快速迭代阶段是务实的选择。

**中期演进**：当 Action 表读写压力继续增大时，可以将 bridge 对 Action 详情的查询改为从本地缓存读取（通过 Kafka 事件同步 Action 数据到 bridge 的本地存储）。

### Redis 使用分配

| Key 模式 | 写入服务 | 读取服务 | 说明 |
|---------|---------|---------|------|
| `{hostuuid}` (Sorted Set) | ctrl-dispatcher | ctrl-bridge（读）, ctrl-tracker（ZREM） | Action 缓存 |
| `heartbeat:{hostuuid}` (Hash) | ctrl-bridge | ctrl-guardian | 心跳信息 |
| `ctrl:leader:{service}` (String) | 各服务自己 | 各服务自己 | 分布式锁 |

### 数据归属的核心原则

1. **写入方即 Owner**——谁 INSERT 谁负责这张表的 Schema 定义和数据完整性
2. **尽量通过事件传递变化**——而非直接查对方的表
3. **允许初期共享**——不必一开始就做到完美解耦，渐进式拆分

## Q90：微服务拆分后，分布式事务怎么处理？有没有跨服务的事务场景？

**[进阶] [分布式事务]**

### 回答

### 核心原则：避免跨服务事务

**我们的系统中没有传统意义上的分布式事务（如 2PC/TCC/Saga）。**

原因在于整个任务调度的数据流是**单向流动**的：

```
Job → Stage → Task → Action → 执行 → 结果上报
```

每一步都是"上游完成后触发下游"，不存在"要么全部成功、要么全部回滚"的场景。每一步的原子性在单个服务内就能保证（单库事务）。

### 潜在的跨服务一致性场景

**场景一：orchestrator 写入 DB 后发 Kafka 失败**

```
orchestrator: 
  1. 事务写入 Job + Stage 到 DB  ← 成功
  2. 投递第一个 Stage 到 Kafka   ← 失败！
```

**处理方式**：最终一致性 + 补偿

- Stage 数据已经在 DB 中（状态为"待处理"）
- guardian 的补偿扫描每 5~10s 检查"已创建但超时未开始"的 Stage
- 发现后重新投递到 Kafka
- 整个过程最多延迟 5~10s，不需要分布式事务

**场景二：tracker 更新 Action 状态后投递 Stage 推进消息失败**

```
tracker:
  1. 批量 UPDATE Action 状态     ← 成功
  2. 检测到 Stage 完成
  3. 投递下一个 Stage 到 Kafka   ← 失败！
```

**处理方式**：同样是补偿机制兜底

- Action 状态已更新，Stage 实际已完成（可以通过查询 Action 状态确认）
- guardian 的补偿扫描检测到"所有 Action 完成但 Stage 未推进"的情况
- 重新投递下一个 Stage

**场景三：dispatcher 写入 DB 后 Redis 写入失败**

```
dispatcher:
  1. 批量 INSERT Action 到 DB    ← 成功（state=init）
  2. ZADD 到 Redis               ← 失败！
```

**处理方式**：dispatcher 的补偿加载

- dispatcher 有定时补偿任务，检查 state=init 但 Redis 中不存在的 Action
- 重新加载到 Redis
- Agent 的 2s 定时轮询也是兜底机制

### 为什么不用 2PC/TCC/Saga？

| 方案 | 分析 |
|------|------|
| **2PC（两阶段提交）** | 任务调度系统不需要"全部成功或全部回滚"的强一致语义 |
| **TCC（Try-Confirm-Cancel）** | 过于复杂，且 Action 执行本身不可回滚（Shell 命令已执行） |
| **Saga** | 更适合有多个步骤都需要回滚的场景（如订单→支付→库存） |

**我们的场景特点**：

1. 数据流单向——没有"回滚已执行的 Action"的需求
2. 每个环节都有补偿机制——最终一致性完全满足
3. 任务本身有幂等性——重复触发不会造成问题

> **工程判断**：在任务调度场景中，"先写 DB 后发 Kafka + 补偿扫描"的最终一致性方案，复杂度远低于分布式事务，且完全满足业务需求。引入 2PC/TCC 属于过度设计。

### 各服务资源需求

| 服务 | CPU | 内存 | 网络 | 说明 |
|------|-----|------|------|------|
| ctrl-gateway | 1 核 | 512MB | 低 | 轻量级代理转发 |

### 扩缩容策略

- **查询压力增大**→ 先扩 gateway 和 tracker

## Q92：从单体迁移到微服务的过程中，怎么保证业务不中断？灰度策略是什么？

**[进阶] [迁移策略]**

### 回答

### 核心策略：渐进式拆分，三阶段演进

**不建议一步到位拆成 6 个服务**，而是分 3 个阶段，每个阶段都保留回滚能力：

```
Phase 1（2~3 周）：拆出 Agent 通信网关 + Action 下发
├── 拆出 ctrl-bridge（最痛的点：6 万 QPS）
├── 拆出 ctrl-dispatcher（第二痛的点：百万级 Action 写入）
└── 其余功能暂时保留在 ctrl-server（过渡态）

Phase 2（2~3 周）：拆出流程编排 + 状态追踪
├── 拆出 ctrl-orchestrator（引入 Kafka 事件驱动）
├── 拆出 ctrl-tracker（引入批量聚合）
└── ctrl-server 退化为 ctrl-gateway + ctrl-guardian

Phase 3（1~2 周）：拆出 API 网关 + 运维服务
├── 拆出 ctrl-gateway
├── 拆出 ctrl-guardian
└── 废弃 ctrl-server
```

### Phase 1 详细方案（以拆出 bridge 为例）

**Step 1：代码层面抽取**

将 `AgentCmdServiceModule`（gRPC 服务实现）从 woodpecker-server 中抽取，编译为独立的 ctrl-bridge 二进制。

**Step 2：双写验证阶段**

```
Agent → ctrl-bridge (新) ──→ Kafka[action_result]
                              同时
Agent → woodpecker-server (老) ──→ 直接写 DB

两套同时运行，对比结果
```

- 新老同时运行，Agent 连接新的 ctrl-bridge
- ctrl-bridge 的结果上报走 Kafka → 但同时老 Server 也在消费（对比验证）
- 验证数据一致后，关闭老 Server 的 gRPC 服务

**Step 3：流量切换**

- Agent 的 gRPC 连接地址从 woodpecker-server 切换到 ctrl-bridge
- 通过 DNS 或配置中心切换，不需要重启 Agent
- 先切 10% 的 Agent（灰度验证），观察 1~2 天无问题后全量切换

**Step 4：回滚准备**

- 如果发现问题，将 Agent 连接地址切回 woodpecker-server
- 回滚时间 <5 分钟（仅需修改配置）

### 每个 Phase 的回滚策略

| Phase | 回滚方式 | 回滚时间 |
|-------|---------|---------|
| Phase 1 | Agent 连接地址切回 server | <5 分钟 |
| Phase 2 | 关闭 Kafka 消费者，启用 server 的 Worker | <10 分钟 |
| Phase 3 | 将 gateway 和 guardian 合并回 server | <30 分钟 |

### 灰度策略

**按集群灰度**（不是按 Agent 百分比）：

1. 选择一个**非核心的测试集群**，先将该集群的 Agent 切到新架构
2. 运行 1~2 天，对比新老架构的任务执行结果（成功率、延迟、日志）
3. 确认无误后扩大到 3~5 个集群
4. 全量切换

**为什么按集群而非百分比？** 因为同一个集群的 Agent 之间有协作关系（比如集群扩容需要所有节点都执行同一个 Job），如果一部分走新架构一部分走老架构，可能产生数据不一致。

### 监控指标

灰度过程中重点关注：

| 指标 | 正常范围 | 告警阈值 |
|------|---------|---------|
| Action 成功率 | >99.5% | <99% |
| Action 平均延迟 | <200ms | >500ms |
| DB 慢查询数 | <10/min | >50/min |
| Kafka 消费延迟 | <100ms | >1s |
| Agent 离线率 | <0.1% | >1% |

## Q93：Go 和 Java 混合架构是怎么回事？为什么不统一技术栈？

**[中等] [技术栈混合]**

### 回答

### 混合架构的现状

```
Go:
  ├── ctrl-gateway        (Gin)
  ├── ctrl-orchestrator   (Gin + Kafka Consumer)
  ├── ctrl-dispatcher     (Kafka Consumer)
  ├── ctrl-bridge         (gRPC Server)
  ├── ctrl-tracker        (Kafka Consumer + Gin)
  ├── ctrl-guardian       (Gin + 定时任务)
  └── ctrl-agent          (自研模块管理)

Java:
  └── TaskCenter          (Activiti 5.22 工作流引擎)
```

核心的 Server 和 Agent 全部是 **Go**，只有工作流引擎用的是 **Java**。

### 为什么不统一成 Go？

**根本原因：Activiti 是 Java 生态的，没有 Go 等价物。**

| 方面 | 分析 |
|------|------|
| **Activiti 本身** | 成熟的 BPMN 2.0 工作流引擎，支持复杂的流程定义（顺序执行、并行执行、条件分支）。系统中有 56 种 BPMN 流程模板，所有的 Stage 编排逻辑都依赖 Activiti |
| **Go 生态的 BPMN 引擎** | 几乎没有成熟的。存在一些简单的 workflow 库（如 go-workflow），但不支持完整的 BPMN 2.0 标准，特别是子流程、会签、多实例等复杂特性 |
| **替换成本** | 56 种流程模板 + 数十个 TaskProducer + 流程定义的解析和执行逻辑，替换工作量巨大 |
| **投入产出比** | Activiti 运行稳定，不是系统瓶颈。花大量时间重写工作流引擎的投入产出比很低 |

### Go 和 Java 之间怎么交互？

通过 **gRPC** 实现跨语言通信：

```
ctrl-orchestrator (Go) ──gRPC──→ TaskCenter (Java)
                                  │
                                  ├── 启动 Activiti 流程实例
                                  ├── 查询活跃的 Task
                                  └── 完成/回调 Task

一个 .proto 文件定义接口 → Go 和 Java 各自生成客户端/服务端代码
```

**已验证可行**：这不是纯设计，系统中 Go Server 和 Java TaskCenter 之间的 gRPC 通信已经在生产运行。Protobuf 的 tag-based 字段识别保证了跨语言兼容性。

### 为什么选 Go 做 Server/Agent？

| 维度 | Go | Java |
|------|------|------|
| **并发模型** | goroutine 极轻量（~8KB），天然适合高并发 | 线程较重（~1MB），需要线程池管理 |
| **内存占用** | Agent 常驻内存 ~30MB | JVM 启动就占 200MB+ |
| **启动速度** | <1s | 数秒（JVM 预热） |
| **编译部署** | 单一静态二进制，无运行时依赖 | 需要 JVM 环境 |
| **适合场景** | 高并发网络服务、轻量级 Agent | 复杂业务逻辑、丰富的生态库 |

对于 Agent（部署在每个节点上，需要轻量、低资源消耗）和 Server（需要处理 6 万+ QPS），Go 是更合适的选择。

### 长期演进方向

```
短期（现状）：
  Go Server/Agent + Java TaskCenter (Activiti)

中期（可选）：
  Go Server/Agent + Go 自研简化版工作流引擎
  （当 Activiti 的复杂特性用不到时，可以用 Go 自研替换）

长期（理想态）：
  全 Go 技术栈
  （前提是 Go 生态出现成熟的 BPMN 工作流引擎，或业务不再需要复杂的 BPMN 特性）
```

**务实的态度**：技术栈统一是理想目标，但不是所有场景都需要追求。Go 和 Java 混合架构通过 gRPC 可以良好协作，Activiti 稳定运行且不是瓶颈，此时花大量精力替换它不如把时间用在优化核心调度链路上。**技术决策的关键是"解决最重要的问题"，而非"统一所有东西"。**
