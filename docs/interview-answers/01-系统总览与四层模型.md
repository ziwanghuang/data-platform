# 面试答案 — 第一部分：系统总览与四层模型（1-12）

---

## Q1：请简要介绍一下你做的这个任务调度系统，它解决什么问题？

TBDS（Tencent Big Data Service）管控平台是腾讯云面向政企客户的大数据平台的核心调度系统。它的职责是将用户的集群操作（创建集群、扩缩容、配置下发、服务启停等）编排为具体的执行指令，下发到数千个节点的 Agent 执行，然后收集执行结果，驱动任务流转直至完成。

简单说就是三件事：
1. **编排**：把用户的一个操作请求（如"安装 YARN"）拆解成数千条具体的 Shell 命令
2. **下发**：把这些命令分发到对应的物理节点上的 Agent 去执行
3. **追踪**：收集每个节点的执行结果，驱动整个流程一步步推进，直到完成或失败

系统管控 5000+ 节点、30+ 大数据组件，支撑 57+ 种异步工作流。

## Q2：系统采用了 Job → Stage → Task → Action 四层模型，每一层分别代表什么？举个实际例子。

```
Job（一个操作，如"安装 YARN"）
 └── Stage（操作的一个阶段，Stage 之间顺序执行）
      └── Task（阶段内可并行的子任务）
           └── Action（每个节点具体执行的 Shell 命令）
```

以"安装 YARN 组件"为例：

| 层级 | 含义 | 示例 | 执行方式 |
|------|------|------|---------|
| **Job** | 一个完整的操作流程 | "安装 YARN" | 由 API 触发 |
| **Stage** | 操作的一个阶段 | "检查环境" → "下发配置" → "安装软件包" → "启动服务" → "健康检查" | Stage 之间**顺序执行** |
| **Task** | 阶段内的子任务 | "启动服务"阶段包含：启动 NameNode、启动 DataNode、启动 ResourceManager、启动 NodeManager | 同 Stage 内 Task **并行执行** |
| **Action** | 每个节点的具体命令 | `systemctl start hadoop-yarn-resourcemanager`，在 3 台 RM 节点上各执行一条 | 下发到各节点 Agent 执行 |

**典型数据规模**：安装 YARN → 1 Job → 16 Stage → 22 Task → 约 5,350 个 Action（100 节点集群）。如果是 6000 节点集群，Action 数量会达到约 32 万。

## Q3：为什么选择四层模型而不是更简单的两层或三层？多一层带来了什么好处？

四层模型的每一层都对应一个**不可替代的业务维度**：

| 层级 | 语义维度 | 为什么不能去掉 |
|------|---------|---------------|
| **Job** | 操作维度（What） | 用户操作的入口，一个 Job 对应一个用户意图 |
| **Stage** | 阶段维度（When） | 表达阶段间的**顺序依赖**。去掉 Stage，就丢失了"先下发配置，再启动服务"的阶段概念 |
| **Task** | 子任务维度（How） | 表达阶段内的**并行能力**。去掉 Task，同一阶段内的操作只能串行或只能全部并行，无法精细控制 |
| **Action** | 节点维度（Where） | 表达**物理节点**上的具体命令。去掉 Action，就无法追踪每个节点的执行状态 |

**与两层模型对比**（Job → Action）：
- 丢失阶段语义，只能用 `order_num` 模拟顺序，维护困难
- 无法部分重试（"启动服务"失败只能重试整个 Job）
- Action 表数据爆炸，所有信息都堆在一张表

**与三层模型对比**（Job → Step → Action）：
- 把"阶段"和"子任务"合并成一层，需要额外的 `group` 和 `parallel_flag` 字段来弥补语义损失
- 本质上是把四层模型的结构信息编码到了字段中，增加了理解和维护成本

四层模型的核心好处是**语义正交**：Stage 管顺序、Task 管并行、Action 管节点，各自独立，不需要额外的标记字段。

## Q4：有没有考虑过用 DAG（有向无图）来编排任务？四层模型和 DAG 编排各有什么优劣？

考虑过。DAG 模型（Airflow/Temporal 风格）的核心优势是灵活的依赖关系——任意 Task 之间可以定义依赖。但它有一个**致命问题**：缺少"节点维度"。

**DAG 不适合 TBDS 场景的原因**：

1. **数据量爆炸**：如果把每个节点的每条命令都建模为 DAG 节点，6000 节点 × 50 命令 = 30 万个 DAG 节点，DAG 调度器根本无法承受
2. **缺少批量操作语义**：在所有 DataNode 上执行 start 命令"在 DAG 中需要为每个节点创建一个独立 Task，无法表达"批量"语义
3. **进度追踪困难**：DAG 只知道某个 Task 是否完成，不知道"6000 个节点中有多少已经完成"

**核心区别**：
- DAG 是为**逻辑任务编排**设计的（Airflow 的 ETL Pipeline，每个 Task 在一台机器上执行，Task 总数几十个）
- 四层模型是为**大规模节点命令下发**设计的（每个 Task 需要在数千台机器上分别执行）

四层模型通过 Task → Action 的展开，天然地将"逻辑任务"映射到"物理节点"，这是 DAG 模型缺少的关键能力。

## Q5：Apache Ambari 也用了类似的四层模型，你了解这个设计吗？

了解。Ambari 是 Apache 基金会的大数据集群管理项目（被 Hortonworks/Cloudera 广泛使用），它采用了几乎相同的四层模型：

```
Ambari:   Request → Stage → Task → HostRoleCommand
TBDS:     Job     → Stage → Task → Action
```

两个独立开发的系统，面对相同的业务场景（大数据集群管理，数千节点），得出了相同的架构设计。这不是巧合，而是大数据集群管理场景的特殊性决定的：
- 多阶段顺序依赖 → 需要 Stage 层
- 阶段内并行 → 需要 Task 层
- 大规模节点执行 → 需要 Action/HostRoleCommand 层

这是四层模型合理性的最好的业界验证。Ambari 也经历了大规模生产验证（管理数千节点的 Hadoop 集群），证明了这个模型在实际场景中的可行性。

## Q6：如果将四层精简为三层（去掉 Stage 或 Task），系统会出现什么问题？

**去掉 Stage（Job → Task → Action）**：
- 丢失"阶段"概念，无法表达"先下发配置，再启动服务"的顺序依赖
- 所有 Task 要么全串行（效率低），要么全并行（无法保证执行顺序）
- 无法按阶段展示进度（用户看不到"下发配置完成 100%，启动服务完成 60%"）
- 无法按阶段重试（"启动服务"失败了只能重试整个 Job）

**去掉 Task（Job → Stage → Action）**：
- 丢失阶段内的并行能力，同一个 Stage 里"配置 HDFS"和"配置 YARN"无法并行
- Stage 直接关联 Action，Stage 数量暴增（原来 16 Stage × 1.4 Task/Stage = 22 Task，变成 22 个 Stage）
- Stage 间的顺序控制变得复杂（原来的并行任务变成了需要精确控制顺序的 Stage）

核心问题是：三层模型会把两个正交的维度（阶段顺序 + 子任务并行）压缩到一层，需要额外的 `group` 和 `parallel_flag` 字段来弥补，本质上是用"属性"替代了"层级结构"来表达语义，增加了理解和维护成本。

## Q7：如果扩展为五层（加入 Pipeline 层或 Command 层），适用于什么场景？为什么你们没有选择五层？

五层模型的思路是在 Action 下面加一层 Command：

```
Action（安装 YARN NodeManager）
 └── Command-1: 下载安装包（wget ...）
 └── Command-2: 解压安装包（tar ...）
 └── Command-3: 配置环境变量
 └── Command-4: 启动服务（systemctl start ...）
```

**适用场景**：需要极细粒度追踪的场景（如合规审计），或每个 Command 有不同的超时和重试策略。

**为什么没有选择**：
1. **数据量再次爆炸**：30 万 Action × 4 Command = 120 万 Command 记录，DB 压力翻 4 倍
2. **收益不大**：当前 Action 的 `commandJson` 已经可以包含复合 Shell 命令（用 `&&` 串联多步操作），Agent 顺序执行即可
3. **Agent 复杂度上升**：Agent 需要管理 Command 级别的状态和上报，增加了实现复杂度

如果未来确实需要 Command 级别的追踪，可以通过 Action 的 `stdout` 字段中的结构化日志来实现，而不需要增加一层数据模型。这是**"刚好够用"的设计**——不多也不少。

## Q8：四层模型中，Stage 之间是串行执行的，这个约束对调度器实现有什么影响？

Stage 串行执行意味着调度器需要实现**级联触发**机制：

1. **顺序保证**：Stage 表有 `order_num` 字段和 `next_stage_id` 字段，明确定义执行顺序
2. **触发逻辑**：当前 Stage 的所有 Task 完成后，调度器需要检测进度，然后触发下一个 Stage
3. **进度检测链路长**：Action 完成 → 检查 Task 进度 → 检查 Stage 进度 → 触发下一个 Stage

在原始实现中，这个检测是通过 `memStoreRefresher` 每 200ms 扫描 DB 实现的——这正是性能瓶颈之一。优化后改为事件驱动：Action 完成事件 → Task 完成事件 → Stage 完成事件 → 投递下一个 Stage 到 Kafka，延迟从秒级降至 <200ms。

**潜在改进**：如果未来需要 Stage 间的部分并行（如 HDFS 配置和 YARN 配置可以同时下发），可以在 Stage 表增加 `depends_on` 字段，将 Stage 间的严格顺序升级为 Stage DAG。但当前场景下严格顺序已经够用。

## Q9：请画出一个"HDFS 扩容"操作在四层模型中的展开过程。

```
Job: HDFS 扩容（添加 3 台新的 DataNode）
│
├── Stage-0: 准备节点（顺序执行第一步）
│   ├── Task: INIT_NODE → Action × 3（在新节点上执行初始化脚本）
│   └── Task: INSTALL_AGENT → Action × 3（在新节点上安装 Agent）
│
├── Stage-1: 下发配置（Stage-0 完成后执行）
│   └── Task: PUSH_HDFS_CONFIG → Action × 3（下发 hdfs-site.xml 等配置文件）
│
├── Stage-2: 安装服务（Stage-1 完成后执行）
│   └── Task: INSTALL_DATANODE → Action × 3（安装 Hadoop DataNode 组件）
│
├── Stage-3: 启动服务（Stage-2 完成后执行）
│   └── Task: START_DATANODE → Action × 3（systemctl start hadoop-hdfs-datanode）
│
└── Stage-4: 注册节点（Stage-3 完成后执行）
    └── Task: REGISTER_TO_CLUSTER → Action × 3（向 NameNode 注册新 DataNode）
```

**数据量**：1 Job → 5 Stage → 5 Task → 15 Action（3 台节点 × 5 个命令）

如果是扩容 200 台 DataNode，Action 数量 = 200 × 5 = 1000 个。

## Q10：系统的整体技术栈是什么？为什么选择 Go + gRPC + Redis + MySQL 这套组合？

| 类别 | 技术 | 选型理由 |
|------|------|---------|
| **语言** | Go | Server 和 Agent 均用 Go，高并发（goroutine）、低资源消耗（Agent 部署在每个节点上，必须轻量） |
| **数据库** | MySQL 5.7+ | 业务数据持久化，GORM 作为 ORM，成熟稳定 |
| **缓存** | Redis 5.0+ | Action 任务缓存（Sorted Set 保证下发顺序）、心跳信息存储（Hash + TTL）、分布式锁（SETNX） |
| **RPC** | gRPC + Protobuf | Server-Agent 通信，Protobuf 序列化高效（相比 JSON 体积减少 60%+），跨语言支持好（后来 TaskCenter 用 Java 实现） |
| **HTTP 框架** | Gin | RESTful API，轻量高性能 |
| **工作流引擎** | Activiti 5.22（Java） | 56 种 BPMN 流程定义，Activiti 是成熟的工作流引擎，但只有 Java 版本 |
| **消息队列（优化后）** | Kafka | 事件驱动架构的消息总线，平台本身管理 Kafka 集群，零额外运维成本 |

**为什么选 Go 而不是 Java**：
- Agent 部署在每个节点上（5000+），必须轻量。Go 编译为单二进制文件，内存占用低（几十 MB），启动快
- Server 需要高并发处理 6 万 QPS 的 Agent 轮询，goroutine 的并发模型非常适合
- 唯一例外是 TaskCenter 用了 Java，因为 Activiti 工作流引擎只有 Java 版本

## Q11：系统的 Server 端和 Agent 端分别部署在哪里？它们之间的关系是什么？

```
┌─────────────────────────────────────────────────┐
│          管控节点（3 台高可用部署）                  │
│                                                  │
│  woodpecker-server × 3（通过分布式锁选 Leader）     │
│  MySQL × 1（主从）                                │
│  Redis × 3（哨兵模式）                            │
│  Kafka × 3（集群）                               │
└─────────────────────┬───────────────────────────┘
                      │ gRPC / UDP
                      ▼
┌─────────────────────────────────────────────────┐
│          数据节点（5000+ 台）                      │
│                                                  │
│  每台节点部署一个 woodpecker-agent                  │
│  Agent 主动连接 Server（Agent → Server 方向）       │
└─────────────────────────────────────────────────┘
```

**关系**：
- Server 是**大脑**：负责任务编排、指令生成、进度追踪、状态管理
- Agent 是**执行者**：负责拉取任务、执行 Shell 命令、上报结果、心跳上报
- 通信模式是 **Agent 主动拉取**（Pull），而非 Server 推送。Agent 定时通过 gRPC 调用 Server 的 `CmdFetchChannel` 接口获取待执行的 Action
- 心跳机制：Agent 每 5 秒向 Server 上报一次心跳（节点状态、资源信息），Server 将心跳信息写入 Redis Hash（TTL=30s），超时未收到心跳则标记节点离线

## Q12：大概有多少台 Agent？并发量级是多少？

**规模**：
- Agent 数量：5000-6000 台（最大管控集群规模）
- 典型集群：100~2000 节点不等

**并发量级**：

| 指标 | 数值 | 说明 |
|------|------|------|
| Agent → Server QPS（优化前） | **6 万/s** | 6000 节点 × 每 100ms 拉取一次 = 10 次/s/节点 |
| Agent → Server QPS（优化后） | **1500/s** | 无任务时 2s 轮询 + UDP 事件通知 |
| 单次操作 Action 数量 | **数千~数十万** | 100 节点约 5000 Action，6000 节点约 32 万 Action |
| 突发 Action 上报 | **10 万次/批** | 2000 节点 × 50 Action 同时完成 |
| Action 表累积数据量 | **百万级**（1 个月 ~150 万条） | 这是整个系统数据量最大的实体 |

核心挑战在于：99% 时间没有任务（Agent 在空轮询），但偶尔会突发大量任务（如 6000 节点扩容，瞬间产生 32 万 Action）。系统需要同时应对"平时安静"和"偶尔爆发"两种模式。
