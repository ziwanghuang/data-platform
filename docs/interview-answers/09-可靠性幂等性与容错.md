## Q94：整个系统的端到端可靠性怎么保证？从 Job 创建到最终 Action 执行完成，中间有哪些可能丢失的环节？

**[进阶] [全链路可靠性]**

### 回答

先画出完整的消息流转链路，标注每个环节的可靠性风险和保障机制：

```
Job 创建 → Kafka → Stage 生成 → Kafka → Task 生成 → Kafka → Action 生成
                                                                  ↓
                                                          Kafka(action_batch_topic)
                                                                  ↓
                                                          DB 写入 + Redis 缓存
                                                                  ↓
                                                          UDP Push / Agent Pull
                                                                  ↓
                                                          Agent 执行
                                                                  ↓
                                                          gRPC 结果上报
                                                                  ↓
                                                          Kafka(action_result_topic)
                                                                  ↓
                                                          批量聚合 → DB UPDATE
                                                                  ↓
                                                          Stage 推进 → 下一轮
```

### 逐环节风险分析与保障

#### 环节 1：API → Kafka（Job 创建）

| 风险 | 概率 | 保障机制 |
|------|------|---------|
| API 写入 Kafka 失败 | 低 | 同步发送 + 重试 3 次，失败返回 HTTP 500，客户端可重试 |
| Kafka 消息丢失 | 极低 | `acks=all`，3 副本，ISR ≥ 2，`min.insync.replicas=2` |
| 重复写入（客户端重试） | 中 | Job 表 unique index on `(cluster_id, job_type, create_time)` 做去重 |

**保障级别**：at-least-once + DB 幂等 = effectively exactly-once

#### 环节 2：Kafka → 流程编排（Stage/Task/Action 生成）

| 风险 | 概率 | 保障机制 |
|------|------|---------|
| Consumer 处理失败 | 中 | 手动 offset commit，处理失败不提交，下次重新消费 |
| 重复消费（rebalance 导致） | 中 | `message_consume_log` 表唯一索引 `(topic, partition, offset)` 去重 |
| 生成 Stage/Task 到一半 crash | 低 | DB 事务保证原子性（一个 Stage 的所有 Task/Action 在一个事务内生成）|
| 中间 Stage 未被触发 | 低 | `ctrl-guardian` 补偿扫描：每 30s 检查是否有"上一个 Stage 已完成但下一个 Stage 未开始"的情况 |

#### 环节 3：Action 写入 DB → Redis → 通知 Agent

| 风险 | 概率 | 保障机制 |
|------|------|---------|
| DB 写入成功但 Redis 写入失败 | 低 | 补偿扫描：每 5s 检查 DB 中 state=0（待执行）但 Redis 中不存在的 Action，重新写入 Redis |
| Redis 写入成功但 UDP Push 失败 | 高（UDP 不可靠）| Agent Pull 兜底：即使 UDP 丢了，Agent 仍会通过 gRPC 定时拉取 |
| Redis 宕机 | 低 | Redis Sentinel 或 Cluster 自动故障转移；极端情况下 Agent Pull 直接查 DB |

**关键设计**：Redis 不是"必须成功"的环节，而是加速层。即使 Redis 完全不可用，Agent 的 Pull 机制仍能从 DB 获取 Action。

#### 环节 4：Agent 执行 Action

| 风险 | 概率 | 保障机制 |
|------|------|---------|
| Action 重复下发（Agent 拉取到已在执行的 Action）| 中 | Agent 本地 `executing_set` 去重，已在执行的 Action 不会再次进入 WorkPool |
| Action 重复下发（Agent 重启后拉取到已完成的 Action）| 中 | Agent 本地 `finished_set` 持久化到磁盘（BoltDB），重启后加载 |
| Action 执行超时 | 中 | WorkPool 对每个 Action 设置 `context.WithTimeout`，超时后 kill 子进程并上报 timeout 状态 |
| Action 执行失败 | 中 | 上报 exitCode ≠ 0，Server 根据 Task 配置决定是否重试（最多重试 3 次）|
| Agent 进程被 kill（OOM/运维）| 低 | Server 的 `ctrl-guardian` 超时检测：超过 5 分钟未收到结果的 Action 标记为 timeout，触发重试或告警 |

#### 环节 5：Agent 上报结果 → Server 处理

| 风险 | 概率 | 保障机制 |
|------|------|---------|
| gRPC 上报失败 | 中 | Agent 本地 `resultQueue` 缓存，3 次指数退避重试，最终写入本地 WAL 文件 |
| WAL 文件未被消费 | 低 | Agent 启动时扫描 WAL 目录，将遗留结果重新加入 resultQueue |
| Kafka 消费处理失败 | 中 | 手动 offset commit + `message_consume_log` 去重 |
| DB UPDATE 失败 | 低 | Kafka 消费不提交 offset，下次重新消费重试 |
| 重复上报导致状态回退 | 中 | 乐观锁 CAS：`UPDATE action SET state=3 WHERE id=? AND state=2`，已完成的 Action 不会被覆盖 |

### 兜底机制总结

系统设计了**三层兜底**：

```
┌────────────────────────────────────────────────────────────┐
│  第一层：正常路径                                            │
│  Kafka 事件驱动 → 毫秒级响应 → 覆盖 99%+ 的正常场景        │
├────────────────────────────────────────────────────────────┤
│  第二层：重试机制                                            │
│  Kafka 重新消费 + Agent 本地重试 → 覆盖临时故障              │
├────────────────────────────────────────────────────────────┤
│  第三层：补偿扫描（ctrl-guardian）                            │
│  定时扫描 DB 不一致状态 → 覆盖极端故障（双 crash、网络长时间分区）│
│  - 超时 Action 检测（每 30s）                                │
│  - 卡住的 Stage 检测（每 30s）                               │
│  - DB-Redis 一致性检查（每 5s）                              │
│  - 未触发的后续 Stage 检测（每 30s）                         │
└────────────────────────────────────────────────────────────┘
```

**设计哲学**：正常路径追求性能（事件驱动、批量聚合），异常路径追求正确性（补偿扫描、幂等重试）。两者分离，互不干扰。

## Q95：如果 Server 在处理到一半时 crash 了，任务的状态会不会不一致？恢复机制是什么？

**[进阶] [故障恢复]**

### 回答

分场景讨论 Server crash 的影响和恢复机制：

### 场景 1：Server 在生成 Stage/Task/Action 过程中 crash

```
时间线：
T=0s    Consumer 从 Kafka 取出 stage_topic 消息
T=0.1s  开始生成 Tasks 和 Actions
T=0.2s  已生成 Task 1-5，正在生成 Task 6 ← Server crash
```

**状态分析**：
- 手动 offset commit：消息处理未完成，offset 未提交
- DB 事务：如果使用了事务，Task 1-5 也会回滚

**恢复过程**：
1. Server 重启（或另一台 Server 接管该 partition）
2. 从上次 committed offset 重新消费该消息
3. 重新生成所有 Tasks 和 Actions
4. `message_consume_log` 的唯一索引防止消费记录重复插入
5. DB 唯一索引 + `INSERT IGNORE` 防止 Task/Action 重复生成

**关键点**：生成操作是幂等的。即使重复执行，唯一索引保证不会产生重复数据。

### 场景 2：Server 在 Action 状态更新过程中 crash

```
时间线：
T=0s    Consumer 从 action_result_topic 取出批量结果
T=0.1s  开始聚合 UPDATE
T=0.2s  已更新 Action 1-50，正在更新 51-100 ← Server crash
```

**状态分析**：
- Action 1-50：状态已正确更新为"完成"
- Action 51-100：状态仍为"执行中"

**恢复过程**：
1. offset 未提交，重新消费该批消息
2. 对 Action 1-50 执行 CAS UPDATE：`UPDATE SET state=3 WHERE id=? AND state=2`
   - 因为 state 已经是 3，WHERE 条件不匹配，UPDATE 影响 0 行 → 安全跳过
3. 对 Action 51-100 正常 UPDATE → 状态更新成功

**关键点**：CAS 乐观锁使得状态更新天然幂等。重复执行不会导致状态回退。

### 场景 3：Server 在 Stage 推进过程中 crash

```
时间线：
T=0s    stage_status_detect 消费者检测到 Stage-1 所有 Task 完成
T=0.1s  更新 Stage-1 状态为 completed
T=0.2s  准备触发 Stage-2 ← Server crash
```

**状态分析**：
- Stage-1：已标记完成 ✓
- Stage-2：未被触发 ✗

**恢复过程**：
- `ctrl-guardian` 的**补偿扫描**介入
- 每 30s 扫描一次：查找"前一个 Stage 已完成，但后续 Stage 状态仍为 pending"的 Job
- 发现 Stage-1 已完成但 Stage-2 未开始 → 主动触发 Stage-2

```sql
-- 补偿扫描 SQL
SELECT j.id as job_id, s.id as stage_id, s.order_num
FROM job j
JOIN stage s ON j.id = s.job_id
WHERE j.state = 1  -- Job 执行中
AND s.state = 0    -- Stage 未开始
AND EXISTS (
    SELECT 1 FROM stage prev
    WHERE prev.job_id = j.id
    AND prev.order_num = s.order_num - 1
    AND prev.state = 3  -- 前一个 Stage 已完成
)
```

**关键点**：补偿扫描是 Stage 推进的"安全网"。正常路径通过 Kafka 事件触发（毫秒级），异常路径通过补偿扫描兜底（30s 级）。

### 场景 4：Server crash 后 Leader 切换

在原始单 Leader 架构中：
```
T=0s    Leader Server-A crash
T=0-30s 锁 TTL 倒计时（最长 30s）
T=30s   锁过期
T=30.xs Server-B 抢锁成功，成为新 Leader
T=31s   Server-B 开始执行所有 Worker
```

在优化后的一致性哈希架构中：
```
T=0s    Server-A crash
T=5s    其他 Server 通过心跳感知 Server-A 离线
T=5.1s  一致性哈希环重新分配：Server-A 负责的 Cluster 迁移到 Server-B 和 Server-C
T=6s    Server-B、C 开始处理迁移过来的 Cluster
```

**对比**：
- 单 Leader：切换时间 = 锁 TTL（最长 30s），期间所有任务暂停
- 一致性哈希：切换时间 ≈ 心跳间隔（5s），只有 crash 节点负责的 Cluster 受影响

## Q96：Action 的幂等执行怎么保证？同一个 Action 会不会在两台 Agent 上同时执行？

**[进阶] [幂等性]**

### 回答

### 三层去重保障幂等性

```
┌─────────────────────────────────────────────────────┐
│              第一层：Server 端去重                     │
│                                                      │
│  Kafka 消费幂等：                                     │
│  message_consume_log 表，唯一索引 (topic,partition,offset) │
│  INSERT IGNORE，重复消息直接跳过                       │
│                                                      │
│  DB 层去重：                                          │
│  Action 分发时使用 CAS：                               │
│  UPDATE action SET state=2 WHERE id=? AND state=0    │
│  只有 state=0（待执行）才会被分发，已分发的不会重复下发  │
├─────────────────────────────────────────────────────┤
│              第二层：Redis 路由隔离                    │
│                                                      │
│  Action 通过 hostuuid 路由到指定节点：                 │
│  Redis Sorted Set: key=hostuuid, member=actionId     │
│                                                      │
│  每个 Action 只会出现在一个 hostuuid 的 Set 中         │
│  → 只有该节点的 Agent 能拉取到这个 Action              │
│  → 从路由层面杜绝"两台 Agent 同时执行同一 Action"     │
├─────────────────────────────────────────────────────┤
│              第三层：Agent 本地去重                    │
│                                                      │
│  executing_set（内存）：                               │
│  - 执行前检查：if actionId in executing_set → skip   │
│  - 执行开始：executing_set.Add(actionId)              │
│  - 执行完成：executing_set.Remove(actionId)           │
│                                                      │
│  finished_set（内存 + BoltDB 持久化）：                │
│  - 执行前检查：if actionId in finished_set → skip    │
│  - 执行完成：finished_set.Add(actionId)               │
│  - Agent 重启：从 BoltDB 加载恢复 finished_set         │
│  - 定期清理：7 天前的记录自动清除                      │
└─────────────────────────────────────────────────────┘
```

### 会不会在两台 Agent 上同时执行？

**正常情况下不会**。因为每个 Action 有明确的 `hostuuid` 字段，指定了目标执行节点。Redis 以 `hostuuid` 为 key 存储 Action 列表，Agent 只能拉取属于自己的 Action。

**但有一个边界场景需要处理**：Agent 迁移（同一个 hostuuid 的 Agent 在不同物理机上重启）

```
场景：节点 UUID=abc123 的 Agent 原本在 Host-A 上运行

T=0s     Host-A 上的 Agent 开始执行 Action-1001
T=1s     Host-A 宕机
T=60s    运维在 Host-B 上启动了 UUID=abc123 的新 Agent
T=60.1s  新 Agent 拉取任务 → 拿到 Action-1001（Server 认为该 Action 超时，重置为 state=0）
T=60.2s  新 Agent 检查 finished_set → 不在（BoltDB 在 Host-A 上）
T=60.3s  新 Agent 执行 Action-1001
```

**这是否是问题？**

严格来说，这不是"两台 Agent 同时执行"，而是**顺序执行**（Host-A 已宕机，Action 在它上面是中断状态）。新 Agent 重新执行是正确的容错行为。

但如果 Action 是非幂等操作（如 `format namenode`），重复执行会有问题。对此的保障是：
1. **操作层面**：危险操作（如格式化）由 TaskCenter 流程编排控制，不会轻易重试
2. **状态层面**：如果 Host-A 的 Agent 在宕机前已经上报了部分结果，CAS 乐观锁会保护状态不被回退
3. **业务层面**：大部分 Agent 操作是幂等的（启动服务、分发配置、执行安装脚本），天然支持重复执行

### 与 DB 层的配合

```sql
-- Agent 拉取 Action 时，Server 使用 CAS 标记为"已下发"
UPDATE action SET state = 2, update_time = NOW()
WHERE id = ? AND state = 0;

-- affected_rows = 1 → 成功下发，写入 Redis
-- affected_rows = 0 → 已被其他 Consumer 下发，跳过
```

这个 CAS 操作确保同一个 Action 只会被**一次**成功标记为"已下发"，即使多个 Kafka Consumer 并发处理同一批 Action。

## Q96-补充：MemStore 内存缓存与 DB 之间存在不一致窗口，怎么处理？

**[进阶] [缓存一致性 / 竞态分析]**

### 回答

这是一个好问题，涉及调度引擎中 processedStages/processedTasks map（内存去重）和 DB（持久化状态）之间的一致性边界。

#### 问题本质

MemStoreRefresher 每 200ms 扫描 DB，通过 `processedStages map` 做去重后入队到 channel，StageWorker 阻塞消费。map 和 DB 之间必然存在短暂的不一致窗口。

#### 两个典型竞态时序

**时序一：入队后、消费前，Stage 被其他流程完成**

```
T1: Refresher 扫 DB → Stage-X state=Running, taskCount=0 → 不在 map → 入队成功
T2: （同时）另一个已在执行的流程完成了 Stage-X → DB 更新 state=Success，Task 已创建
T3: StageWorker 取出 Stage-X → processStage() 执行
    → 基础版不回查 DB → 重复创建 Task
```

**时序二：缓存清理先于 DB 更新提交**

```
T1: Stage-X 处理完成 → 从 processedStages map 中 delete
T2: （DB UPDATE state=Success 尚未提交）
T3: Refresher 扫 DB → state 仍为 Running → 不在 map → 入队
T4: DB UPDATE 提交 → state=Success
T5: StageWorker 消费 → 又创建一轮 Task
```

#### 为什么这不影响最终正确性？

**核心结论：内存缓存是性能优化层，不是数据一致性保障层。**

`processedStages map` 的定位是 **best-effort 去重**——减少重复入队、降低无谓消费。它不承担也无法承担数据一致性保障。正确性由 DB 侧保证，具体分三层：

```
第一层：processedStages map     → 减少重复入队（性能优化，允许漏网）
第二层：消费侧 DB 回查           → 拦截大部分重复消费（taskCount > 0 → skip）
第三层：DB 唯一约束 INSERT IGNORE → 终极兜底，即使并发写入也只成功一次
```

**设计原则：把内存状态当 best-effort 优化，把 DB 当 source of truth。** 只要 DB 操作本身是事务性的且幂等的，map 与 DB 之间的短暂不一致就不影响最终正确性。最坏情况是多一次无效入队 + 消费侧 skip，浪费一点 CPU，不产生错误数据。

#### 面试表达

> "processedStages map 是性能优化，不是一致性保障。它减少了 99% 的重复入队，但我不依赖它来保证正确性。真正的防线在 DB 侧——消费时先查 taskCount 做幂等判断，Task 表有唯一索引做 INSERT IGNORE 兜底。这是分布式系统处理缓存一致性的标准套路：缓存做 best-effort，DB 做 source of truth，两层分离、互不依赖。"

#### 各版本的处理对比

| 版本 | 消费侧防护 | 效果 |
|------|-----------|------|
| Step 3 基础版 | `processStage()` 直接执行，不回查 DB | ⚠️ 有竞态窗口 |
| phase-1-design 版 | `COUNT(*) FROM task WHERE stage_id=?`，>0 则 skip | ✅ 消费侧回查 |
| Kafka 版 (opt-a1) | StageConsumer 内 `taskCount > 0 → skip` | ✅ 消费侧回查 |
| 幂等加固版 (opt-c1c2) | `INSERT IGNORE` + 唯一索引 | ✅ DB 层终极兜底 |

## Q97：网络分区场景下，Agent 和 Server 断连了，会发生什么？重连后怎么恢复？

**[进阶] [网络分区]**

### 回答

### 网络分区时的系统行为

```
┌──────────────────────┐          网络分区          ┌──────────────────────┐
│      Server 侧       │     ◄── ✕ 断连 ✕ ──►     │      Agent 侧        │
│                       │                            │                      │
│ 1. 心跳超时检测       │                            │ 1. gRPC 调用失败     │
│    → 5分钟无心跳      │                            │ 2. 进入重连模式      │
│    → 标记节点 offline │                            │ 3. 正在执行的 Action │
│                       │                            │    继续执行          │
│ 2. 该节点的 Action    │                            │ 4. 执行结果缓存到    │
│    → 超时检测触发     │                            │    本地 resultQueue  │
│    → 根据配置重试     │                            │ 5. 新 Action 拉取    │
│      或标记失败       │                            │    失败 → 无新任务   │
│                       │                            │ 6. 心跳上报失败      │
│ 3. 新 Action 不再     │                            │    → 仅打日志        │
│    下发到该节点       │                            │                      │
└──────────────────────┘                            └──────────────────────┘
```

### Agent 侧详细行为

#### 1. gRPC 连接断开检测

Agent 通过三个通道与 Server 通信，任何一个通道失败都会触发重连：

```go
// 伪代码：Agent 的重连状态机
type ConnState struct {
    fetchFails   int  // CmdFetch 连续失败次数
    reportFails  int  // CmdReport 连续失败次数
    heartFails   int  // HeartBeat 连续失败次数
    poolRecreated bool // 连接池是否已重建
}
```

#### 2. 指数退避重连

```
重连策略：
  初始间隔：1s
  退避因子：2x
  最大间隔：30s
  抖动：±20%

  第1次重试：1s ± 0.2s
  第2次重试：2s ± 0.4s
  第3次重试：4s ± 0.8s
  第4次重试：8s ± 1.6s
  第5次重试：16s ± 3.2s
  第6次及以后：30s ± 6s
```

**为什么要加抖动？**
防止网络恢复后大量 Agent 同时重连导致"惊群效应"（thundering herd）。5000 个 Agent 如果同一秒重连，Server 端瞬间收到 5000 个 TCP 握手请求。

#### 3. 正在执行的 Action 不受影响

```
网络断开不会中断正在执行的 Action：
  - Action 是本地 Shell 命令执行，不依赖网络
  - WorkPool 中的 goroutine 继续运行
  - 执行完成后结果放入 resultQueue
  - resultQueue 是内存 channel + 溢出到 WAL 文件
```

#### 4. 结果上报的本地缓存

```go
// 结果上报失败时的缓存策略
func (r *ReportModule) reportWithRetry(result *ActionResult) {
    for i := 0; i < 3; i++ {
        err := r.grpcClient.Report(result)
        if err == nil {
            return // 上报成功
        }
        time.Sleep(backoff(i))
    }
    // 3 次失败后写入 WAL 文件
    r.walWriter.Write(result)
    // WAL 文件在网络恢复后由 WalReplayTask 重新读取上报
}
```

WAL（Write-Ahead Log）文件设计：
```
/opt/agent/data/wal/
├── results_20260301_143022.wal  # 时间戳命名
├── results_20260301_143155.wal
└── results_20260301_143500.wal

每个文件最大 10MB，超过后创建新文件
每行一个 JSON 格式的 ActionResult
```

### Server 侧详细行为

#### 1. 心跳超时检测

```
心跳机制：
  Agent 每 30s 发送一次心跳
  Server 记录每个 Agent 的最后心跳时间
  
  超时判定：
  - 90s（3 个心跳周期）无心跳 → 标记为 warning
  - 300s（5 分钟）无心跳 → 标记为 offline
  - offline 后该节点不再接收新 Action
```

#### 2. Action 超时处理

```
ctrl-guardian 的超时检测逻辑：
  每 30s 扫描一次：
  SELECT * FROM action 
  WHERE state = 2  -- 执行中
  AND update_time < NOW() - INTERVAL 5 MINUTE  -- 超过 5 分钟

  对超时 Action：
  1. 检查 retry_count < max_retry（默认 3）
     → 是：重置 state=0（待重新分发），retry_count++
     → 否：标记 state=4（失败），触发告警
  2. 如果重试目标节点 offline，尝试分配到同 Cluster 的其他在线节点
```

### 网络恢复后的行为

```
┌──────────────────────────────────────────────────────────────┐
│                    网络恢复后时间线                             │
│                                                               │
│  T=0s    网络恢复                                             │
│  T=0.1s  Agent gRPC 重连成功（指数退避的下一次尝试成功）       │
│  T=0.2s  Agent 立即上报心跳 → Server 标记节点为 online        │
│  T=0.3s  Agent 开始消费 WAL 文件中的积压结果                  │
│  T=0.5s  WAL 结果逐批上报给 Server                            │
│  T=1s    Agent 恢复正常 Pull 拉取新 Action                    │
│  T=1.1s  Server 恢复向该节点分发 Action                       │
│                                                               │
│  状态恢复检查：                                               │
│  - Agent 的 executing_set 中仍有未完成的 Action               │
│    → 继续等待执行完成（不中断）                                │
│  - Server 已将某些 Action 标记为超时并重新分发                 │
│    → Agent 上报结果时 CAS 检查：                              │
│      UPDATE SET state=3 WHERE id=? AND state=2                │
│      如果 state 已被改为 0（待重分发），影响 0 行               │
│      → Agent 忽略此结果，日志记录 "stale result"               │
│                                                               │
│  冲突处理：                                                   │
│  - 同一个 Action 在 Agent-A（网络恢复后上报）和 Agent-B        │
│    （超时重分发后执行完成）都上报了结果                         │
│  - CAS 保证只有第一个上报被接受                                │
│  - 第二个上报 affected_rows=0 → 安全丢弃                      │
└──────────────────────────────────────────────────────────────┘
```

### 长时间网络分区（>30 分钟）

如果网络分区持续很长时间：
- Agent 侧 WAL 文件持续累积，磁盘空间消耗 → 设置 WAL 最大总量上限（默认 1GB），超过后丢弃最老的文件
- Server 侧该节点所有 Action 都会超时，被重分发到其他节点执行
- 网络恢复后 Agent 上报的历史结果会被 CAS 安全丢弃（因为状态已变化）
- Agent 的 `finished_set` 防止重复执行 Server 重新下发的"已经在本机完成过"的 Action

## Q98：系统有没有做过混沌测试（Chaos Engineering）？如果让你设计几个混沌实验，你会怎么做？

**[进阶] [韧性测试]**

### 回答

实际项目中没有做过正式的混沌测试（TBDS 是生产系统，直接做混沌测试风险较大）。但基于系统架构，如果要设计混沌实验，我会按照**故障类型**和**影响范围**分层设计：

### 实验 1：Agent 进程随机 kill

**目的**：验证 Agent crash 后的任务恢复能力

```yaml
实验设计：
  目标：随机选择 10% 的 Agent 进程 kill -9
  前置条件：系统正在执行一个 2000 节点的扩缩容任务
  预期行为：
    1. 被 kill 的 Agent 上正在执行的 Action 中断
    2. 5 分钟后 Server 超时检测触发
    3. 超时 Action 被重分发到同节点（Agent 重启后）或其他节点
    4. Job 最终完成（可能比正常慢几分钟）
  观测指标：
    - Job 是否最终完成 ✓/✗
    - 完成时间延长了多少
    - 是否有 Action 丢失（状态永远停在 executing）
    - Agent 重启后 WAL 结果是否正确上报
  
  验证点：
    - finished_set 持久化是否正常（重启后不重复执行）
    - WAL 文件是否正确回放
    - executing_set 是否正确重建
```

### 实验 2：Server 节点 crash

**目的**：验证 Leader 切换 / 一致性哈希迁移

```yaml
实验设计 A（单 Leader 架构）：
  目标：kill 当前 Leader Server
  预期行为：
    1. 锁 TTL 倒计时（最长 30s）
    2. 新 Leader 抢锁成功
    3. 所有 Worker 恢复运行
    4. Kafka Consumer 发生 rebalance
    5. 任务继续推进（最多延迟 30s）

实验设计 B（一致性哈希架构）：
  目标：kill 一台 Server 节点
  预期行为：
    1. 5s 内其他 Server 感知
    2. 该 Server 负责的 Cluster 重新分配
    3. 只有该 Server 处理的任务受影响
    4. 其他 Cluster 的任务不中断
    5. 恢复时间 < 10s
```

### 实验 3：Kafka Broker 宕机

**目的**：验证消息不丢失、消费者自动恢复

```yaml
实验设计：
  目标：kill 一个 Kafka Broker（3 Broker 集群中的 1 个）
  前置条件：
    - Topic 副本数 = 3
    - min.insync.replicas = 2
    - acks = all
  预期行为：
    1. ISR 缩减为 2
    2. Producer 写入仍然成功（满足 min.insync.replicas）
    3. Consumer 可能短暂中断（partition leader 切换）
    4. 1-5s 内自动恢复
    5. 无消息丢失
  
  升级实验：kill 2 个 Broker
  预期行为：
    1. ISR < min.insync.replicas
    2. Producer 写入失败 → 触发本地缓存或报错
    3. 系统降级但不崩溃
    4. Broker 恢复后自动恢复
```

### 实验 4：Redis 宕机

**目的**：验证 Redis 作为加速层而非关键路径

```yaml
实验设计：
  目标：shutdown Redis 实例（模拟 Sentinel 切换前的空窗期）
  持续时间：30s
  预期行为：
    1. Action 写入 Redis 失败 → 告警
    2. Agent Pull 请求到 Server → Server 查 Redis 失败 → 降级查 DB
    3. UDP Push 停止（无 Redis 数据）→ Agent 退化为纯 Pull 模式
    4. 性能下降（DB 直查），但功能不中断
    5. Redis 恢复后补偿扫描将 DB 中 state=0 的 Action 重新加载到 Redis
  
  观测指标：
    - Action 下发延迟增加了多少（预期从 100ms 增加到 1-2s）
    - DB QPS 增加了多少
    - 是否有 Action 丢失
```

### 实验 5：网络分区（Agent 子集与 Server 断连）

**目的**：验证 Q97 描述的分区处理逻辑

```yaml
实验设计：
  目标：使用 iptables/tc 隔离 20% 的 Agent 节点
  持续时间：10 分钟
  预期行为：
    1. 隔离的 Agent 进入指数退避重连
    2. 正在执行的 Action 继续执行，结果缓存到 WAL
    3. Server 5 分钟后标记节点 offline，超时 Action 重分发
    4. 网络恢复后 Agent 重连，WAL 回放，状态收敛
  
  关注的竞态条件：
    - 同一 Action 在原 Agent（WAL 回放）和新 Agent（超时重分发）同时上报
    - 预期 CAS 正确处理
```

### 实验 6：MySQL 主从切换

**目的**：验证数据库故障转移对上层业务的影响

```yaml
实验设计：
  目标：手动触发 MySQL 主从切换
  预期行为：
    1. 切换期间（通常 1-5s）DB 连接失败
    2. GORM 的连接池自动重连
    3. 飞行中的事务回滚 → Kafka Consumer 不提交 offset → 重新消费
    4. 5-10s 内恢复正常
    5. 无数据丢失（半同步复制保证）
```

### 混沌实验执行原则

```
┌────────────────────────────────────────────────────────┐
│  1. 先在预发布环境验证，确认无问题后再考虑生产环境      │
```

## Q100：如果让你重新设计这个系统，你会做哪些不同的技术决策？

**[进阶] [架构反思]**

### 回答

这是一个架构反思题，需要展示对系统的深度理解和技术判断力。我会从"确定会改的"和"需要权衡的"两个维度来回答。

### 确定会改的：设计缺陷

#### 1. 一开始就用事件驱动，而不是轮询

```
原始设计：6 个 Worker 高频轮询 DB
问题：空闲时 33 QPS，高峰时 10 万+ QPS，大量无效 I/O

如果重来：
  - 从 Day 1 就引入 Kafka 事件驱动
  - Job 创建/状态变更通过事件触发后续流程
  - 不需要 memStoreRefresher 这种全量加载机制
  - DB QPS 可以从 Day 1 就保持在合理水平

反思：轮询模式的"优点"是实现简单、容易理解，在早期原型阶段有其价值。
但当系统规模从 100 节点增长到 5000 节点时，轮询的 O(n) 复杂度就成了瓶颈。
教训：选择架构模式时要考虑系统的增长曲线，而不仅仅是当前规模。
```

#### 2. Agent 端从一开始就做幂等性设计

```
原始设计：
  - finished_set 仅在内存中
  - 无 WAL 持久化
  - 重启后丢失执行历史

如果重来：
  - finished_set 从 Day 1 就使用 BoltDB 持久化
  - 执行结果从 Day 1 就写入 WAL
  - 成本极低（几十行代码），但可靠性提升巨大

反思：可靠性设计不应该是"后期优化项"，而应该是"基础架构的一部分"。
在分布式系统中，幂等性不是可选的，是必须的。
```

#### 3. Action 表从一开始就做冷热分离

```
原始设计：
  - 所有 Action（包括历史已完成的）放在同一张表
  - AUTO_INCREMENT 已达 239 万+

如果重来：
  - 从 Day 1 就设计归档策略
  - 或直接使用分区表（按创建时间分区）
  - 热表始终保持 <50 万行

反思：数据治理是系统健康度的基础。"以后再清理"的想法往往意味着"永远不清理"。
```

### 确定会改的：技术选型

#### 4. 用 etcd 替代 Redis 做分布式锁

```
原始设计：Redis SETNX + TTL + 看门狗续期
问题：
  - Redis 主从切换时可能出现脑裂（两个 Leader）
  - TTL 机制依赖时钟同步
  - 看门狗续期增加了复杂度

如果重来：
  - 用 etcd 的 lease + watch 做 Leader 选举
  - etcd 基于 Raft，强一致性，无脑裂风险
  - TBDS 集群本身就依赖 ZooKeeper/etcd，不是额外引入新组件

反思：Redis 做缓存一流，做分布式协调只是"能用"，不是"最合适"。
选择工具时应该选择"专门做这件事"的组件，而不是"也能做这件事"的通用组件。
```

#### 5. 用 etcd watch 替代心跳探活

```
原始设计：
  - Agent 每 30s gRPC 心跳 → Server 记录最后心跳时间
  - 300s 无心跳判定离线

如果重来：
  - Agent 在 etcd 注册临时节点（lease 绑定）
  - Agent 存活 → 节点存在
  - Agent 挂了 → lease 过期 → 节点自动删除
  - Server watch etcd → 实时感知节点上下线
  - 检测速度从"5 分钟"缩短到"etcd lease TTL"（通常 10-30s）

反思：心跳机制是经典方案，但存在"检测延迟 = 超时时间"的固有问题。
基于 lease 的方案能提供更快的故障检测，代价是多了一个 etcd 依赖。
```

### 需要权衡的：架构演进方向

#### 6. 是否从 Day 1 就做微服务拆分？

```
我的判断：不会。

原因：
  1. 单体在早期开发效率更高（一个项目、一次部署、一套调试环境）
  2. 微服务的"正确边界"需要对业务足够理解后才能确定
  3. 过早拆分往往导致"拆错了"→ 后续调整成本更高
  4. 团队规模（3-5 人）不需要微服务的组织解耦

正确的做法（也是我们实际的做法）：
  1. 代码层面做好模块化（清晰的 package 边界、接口抽象）
  2. 数据层面做好隔离（不同模块访问不同的数据库表集合）
  3. 在系统规模和团队规模增长后，沿着已有的模块边界拆分

反思："模块化单体"是从单体到微服务最佳的过渡形态。
拆分时机 = 单体已经成为协作瓶颈（多团队修改同一个代码库频繁冲突）。
```

#### 7. 是否统一用 Go 而不是 Go + Java 混合？

```
我的判断：在当前约束下不会改。

原因：
  TaskCenter 基于 Activiti 工作流引擎（Java 生态），没有成熟的 Go 替代品。
  重写 Activiti 的成本远大于维护一个 Java 服务。
  
  但如果完全重新设计，可能会考虑：
  1. 使用 Temporal（Go SDK）替代 Activiti → 统一 Go 技术栈
  2. 或者自研轻量级状态机引擎（我们的工作流本质上是 DAG 执行）
  
反思：
  "统一技术栈"不是目的，"降低总体复杂度"才是。
  如果引入一个组件能大幅降低复杂度（Activiti 提供了成熟的工作流功能），
  那么多一种语言的代价是可以接受的。
```

#### 8. 通信协议是否选择其他方案？

```
当前选择：gRPC（Agent ↔ Server）+ Kafka（Server 内部）+ UDP（Push 通知）

如果重来可能考虑：
  MQTT（Agent ↔ Server）：
  - 专为 IoT/设备管理场景设计，天然支持 subscribe/publish
  - Agent 订阅自己的 topic（hostuuid），Server 发布 Action 到对应 topic
  - 省去了 Pull 轮询 + UDP Push 的组合，一个协议解决两个问题
  - 支持 QoS（0/1/2），天然保障可靠交付

  但不选 MQTT 的理由：
  - TBDS 平台没有 MQTT Broker，需要额外引入
  - gRPC 在 RPC 调用场景（心跳、上报结果）比 MQTT 更自然
  - 团队对 gRPC 更熟悉

反思：
  技术选型不仅看"技术上是否最优"，还要看"团队是否熟悉"和"基础设施是否现成"。
  在 TBDS 环境下，Kafka + gRPC + UDP 的组合是最务实的选择。
```

### 不会改的：经过验证的好决策

```
✓ Protobuf + gRPC 做 Agent 通信 → 跨语言、高性能、强类型
✓ Redis 做 Action 缓存层 → 降低 DB 压力、加速下发
✓ 分布式锁做 Leader 选举 → 在当时的规模下简单有效
✓ Go 做 Server 和 Agent → 并发模型好、部署简单、性能优秀
✓ MySQL 做持久化 → 成熟稳定、团队熟悉、TBDS 平台已有
✓ 批量聚合 + CAS 乐观锁 → 性能和正确性的最佳平衡
```

> **面试技巧**：回答这类"重新设计"的问题时，避免两个极端——既不要说"一切都不改"（显得没有反思能力），也不要说"全部推翻重来"（显得不理解工程现实）。最好的回答是：**识别出明确的改进点，解释为什么当初那样做是合理的，以及现在知道了什么使你能做出更好的决策**。这展示的是工程成熟度和持续学习能力。
