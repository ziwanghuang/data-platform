## 一、消息队列：Kafka 核心场景

### 1.1 异步削峰填谷

**问题**：秒杀场景下 QPS 瞬间从 1000 飙到 50 万，DB 扛不住，怎么办？

**核心方案**：

```
用户请求 → API 网关 → Kafka（缓冲层） → Consumer（限速消费） → DB
                          ↑                    ↑
                     瞬时写入 50w/s         匀速消费 5000/s
```

**关键设计点**：

| 环节 | 设计 | 原因 |
|------|------|------|
| Producer | 异步发送 + batch.size=64KB + linger.ms=5 | 提升吞吐，不阻塞用户请求 |
| Topic | 分区数 = Consumer 实例数 × 2（留扩展空间） | 保证并行消费能力 |
| Consumer | 限速消费（令牌桶/信号量），batch poll + 批量写 DB | 平滑流量，保护下游 |
| 响应 | 同步返回"已受理"，异步处理后推送结果 | 用户不等待，体验好 |

**追问：消息堆积怎么办？**

1. **监控**：Consumer Lag 报警（Burrow / Kafka Exporter + Prometheus）
2. **扩容**：增加 Consumer 实例（前提是分区数够）
3. **降级**：非核心消息暂停消费，优先保证核心链路
4. **跳过**：如果是过期消息（如超过 5 分钟的秒杀请求），直接丢弃

**追问：怎么保证消息不丢？**

三段式保障：

```
Producer 端：acks=all + retries=3 + enable.idempotence=true
Broker 端 ：replication.factor=3 + min.insync.replicas=2 + unclean.leader.election.enable=false
Consumer 端：手动提交 offset（处理完才 commit），而非自动提交
```

🔗 **项目关联**：你的项目从 6-Worker 轮询改为 Kafka 事件驱动，DB QPS 从峰值 10 万降到 3000 以下，就是典型的削峰。

### 1.2 消息顺序性保证

**问题**：订单状态变更（创建→支付→发货→签收）必须按序消费，怎么保证？

**方案**：

```
// 同一订单的消息发到同一分区
producer.send(new ProducerRecord<>(
    "order-events",
    orderId,           // 用 orderId 作为 key → hash 到同一分区
    eventPayload
));
```

**原理**：Kafka 保证 **单分区内** 消息有序，通过 key hash 把同一实体的消息路由到同一分区。

**追问：单分区消费太慢怎么办？**

分区内有序 + 分区间并行：

```
方案 1：增加分区数，让不同订单天然并行
方案 2：分区内按 orderId 分组，组内串行、组间并行（本地线程池）
方案 3：如果业务允许，放宽到"最终一致"——用版本号（乐观锁）做乱序修正
```

**追问：Consumer 消费失败了，重试会不会打乱顺序？**

会。解决方案：

| 策略 | 做法 | 代价 |
|------|------|------|
| 阻塞重试 | 同一分区的消息停止消费，原地重试 N 次 | 阻塞后续消息 |
| 死信队列 | 失败的消息投入 DLQ，继续消费后续消息 | 该订单后续状态可能乱序 |
| 本地缓存 + 延迟合并 | 缓存失败消息，后续消息到达时合并处理 | 实现复杂 |

🔗 **项目关联**：你的 Stage 链表必须顺序执行，Stage₁ → Stage₂ → Stage₃，用的就是 Kafka 分区内有序 + 三态幂等消费保证不乱序。

### 1.3 消息幂等性 / Exactly-Once

**问题**：Consumer 处理完消息但 commit offset 前宕机了，重启后会重复消费，怎么办？

**Kafka 原生支持（Producer 端）**：

```properties
# Producer 幂等：同一 Producer 实例的消息不会重复写入 Broker
enable.idempotence=true

# 事务性 Producer：跨分区/跨 Topic 的原子写入
transactional.id=my-tx-producer
```

**Consumer 端幂等（核心！）**：

Kafka 不保证 Consumer 端 Exactly-Once，需要业务侧自己做：

| 方案 | 实现 | 适用场景 |
|------|------|---------|
| **唯一键去重** | DB 的 UNIQUE INDEX + INSERT IGNORE / ON DUPLICATE KEY | 写入操作 |
| **幂等 token** | 请求带 idempotency_key，Redis SETNX 判重 | API 层 |
| **状态机** | CAS `UPDATE SET state=new WHERE state=old`，affected_rows=0 则跳过 | 状态变更 |
| **版本号** | `UPDATE SET data=x WHERE version=old_version`，防止旧消息覆盖新数据 | 更新操作 |

🔗 **项目关联**：你的 5 个 Consumer 各自用不同的幂等策略（DB 唯一键 / CAS 状态机 / 三态判断），就是 Consumer 端幂等的完整实践。

### 1.4 Kafka 消息积压处理

**问题**：线上发现 Consumer Lag 持续增大，已经积压了 500 万条消息，怎么处理？

**应急三步**：

```
第 1 步：定位原因
  - Consumer 挂了？→ 重启
  - Consumer 处理太慢？→ 查慢查询/外部依赖超时
  - 上游突发流量？→ 确认是否需要限流

第 2 步：快速消费
  - 横向扩容 Consumer（前提：分区数 >= Consumer 数）
  - 如果分区数不够：新建临时 Topic（分区数 × 10），
    写一个中转 Consumer 把原 Topic 消息转发到临时 Topic，
    再用 10 倍 Consumer 消费临时 Topic

第 3 步：善后
  - 监控 Lag 下降到安全阈值后缩容
  - 复盘根因，调整 Consumer 处理能力或增加分区数
```

**追问：消息过期了（如 Kafka retention 到期被删了），怎么办？**

这是**数据丢失**，不是积压。需要：
1. 增大 retention.ms（如从 7 天改到 30 天）
2. 从源头重放（如果源头有持久化）
3. 补偿扫描（定时任务扫 DB 找缺失的数据）

🔗 **项目关联**：你的 CleanerWorker 就是补偿扫描的角色——即使 Kafka 消息丢了，CleanerWorker 每 5s 扫描 DB 发现超时/遗漏的 Task 并重试。

### 1.5 延迟消息 / 定时消息

**问题**：订单 30 分钟未支付需要自动关闭，怎么实现？

| 方案 | 实现 | 优缺点 |
|------|------|--------|
| **定时扫表** | `SELECT * FROM orders WHERE state='unpaid' AND create_time < NOW() - 30min` | 简单但有延迟（取决于扫描频率），大表性能差 |
| **Redis 过期回调** | `SETEX order:123 1800 ""; // keyspace notification` | 不可靠，Redis 不保证精确触发 |
| **Redis ZSet** | `ZADD delay_queue score=到期时间戳 orderId`，轮询 ZRANGEBYSCORE | 可靠、精确，推荐 |
| **RocketMQ 延迟消息** | 原生支持 18 个延迟级别 | 天然支持，但依赖 RocketMQ |
| **Kafka + 时间轮** | 自建延迟服务，到期后转发到目标 Topic | 灵活但开发成本高 |

**推荐方案**：Redis ZSet（中小规模）或 RocketMQ 延迟消息（大规模）。

```go
// Redis ZSet 实现延迟队列
// 入队
rdb.ZAdd(ctx, "delay:order_close", redis.Z{
    Score:  float64(time.Now().Add(30 * time.Minute).Unix()),
    Member: orderId,
})

// 消费（轮询）
func consumeDelayQueue() {
    for {
        results := rdb.ZRangeByScore(ctx, "delay:order_close", &redis.ZRangeBy{
            Min: "0",
            Max: strconv.FormatInt(time.Now().Unix(), 10),
            Count: 100,
        }).Val()
        for _, orderId := range results {
            if rdb.ZRem(ctx, "delay:order_close", orderId).Val() > 0 {
                // ZRem 返回 1 说明是我删的 → 我来处理（分布式去重）
                closeOrder(orderId)
            }
        }
        time.Sleep(1 * time.Second)
    }
}
```

## 二、缓存体系：Redis 核心场景

### 2.1 缓存穿透

**问题**：查询一个不存在的 key（如 userId=-1），缓存没有，每次都打到 DB，怎么办？

| 方案 | 做法 | 适用场景 |
|------|------|---------|
| **缓存空值** | DB 查不到时，缓存 `key → ""` + 短 TTL（如 60s） | 简单，key 空间有限时 |
| **布隆过滤器** | 请求前先过布隆过滤器，不存在则直接返回 | key 空间巨大（如用户 ID）|

```go
// 布隆过滤器 + 缓存空值
func GetUser(userId string) (*User, error) {
    // 第 1 层：布隆过滤器
    if !bloomFilter.Test([]byte(userId)) {
        return nil, ErrNotFound  // 一定不存在
    }
    
    // 第 2 层：缓存
    cached, err := rdb.Get(ctx, "user:"+userId).Result()
    if err == nil {
        if cached == "" { return nil, ErrNotFound }  // 缓存空值
        return deserialize(cached), nil
    }
    
    // 第 3 层：DB
    user, err := db.QueryUser(userId)
    if err != nil || user == nil {
        rdb.Set(ctx, "user:"+userId, "", 60*time.Second)  // 缓存空值
        return nil, ErrNotFound
    }
    
    rdb.Set(ctx, "user:"+userId, serialize(user), 30*time.Minute)
    return user, nil
}
```

**追问：布隆过滤器有误判怎么办？**

布隆过滤器说"不存在"一定不存在，说"存在"可能不存在（假阳性率通常 1%~5%）。少量误判穿透到 DB 没关系，因为 DB 查不到还可以缓存空值兜底。两层防御互补。

### 2.2 缓存击穿

**问题**：一个热点 key 过期的瞬间，大量请求同时打到 DB，怎么办？

| 方案 | 做法 | 优缺点 |
|------|------|--------|
| **互斥锁（singleflight）** | 同一 key 只允许一个请求回源 DB，其他等待 | 简单有效，但等待时间长 |
| **逻辑过期** | 不设 Redis TTL，由业务判断是否过期并异步更新 | 高可用但短暂读到旧数据 |
| **热点 key 永不过期** | 后台定时刷新 | 最简单，适合不常变的数据 |

```go
// Go singleflight 方案（推荐）
import "golang.org/x/sync/singleflight"

var sf singleflight.Group

func GetProduct(productId string) (*Product, error) {
    cacheKey := "product:" + productId
    
    // 先查缓存
    if cached, err := rdb.Get(ctx, cacheKey).Result(); err == nil {
        return deserialize(cached), nil
    }
    
    // singleflight：同一 productId 只有一个请求回源
    result, err, _ := sf.Do(cacheKey, func() (interface{}, error) {
        product, err := db.QueryProduct(productId)
        if err != nil { return nil, err }
        rdb.Set(ctx, cacheKey, serialize(product), 30*time.Minute)
        return product, nil
    })
    
    if err != nil { return nil, err }
    return result.(*Product), nil
}
```

**追问：singleflight 在分布式环境下有效吗？**

`singleflight` 是**进程级别**的，只能防止单个实例内的重复请求。分布式环境下需要 **Redis 分布式锁**：

```go
// 分布式 singleflight
func GetProductDistributed(productId string) (*Product, error) {
    cacheKey := "product:" + productId
    lockKey := "lock:product:" + productId
    
    if cached, err := rdb.Get(ctx, cacheKey).Result(); err == nil {
        return deserialize(cached), nil
    }
    
    // 尝试获取分布式锁
    acquired, _ := rdb.SetNX(ctx, lockKey, "1", 5*time.Second).Result()
    if acquired {
        defer rdb.Del(ctx, lockKey)
        // 双重检查
        if cached, err := rdb.Get(ctx, cacheKey).Result(); err == nil {
            return deserialize(cached), nil
        }
        product, err := db.QueryProduct(productId)
        if err != nil { return nil, err }
        rdb.Set(ctx, cacheKey, serialize(product), 30*time.Minute)
        return product, nil
    }
    
    // 没拿到锁，等待并重试
    time.Sleep(100 * time.Millisecond)
    return GetProductDistributed(productId)  // 递归重试（加退出条件！）
}
```

### 2.3 缓存雪崩

**问题**：大批缓存同时过期，请求全部打到 DB，怎么办？

**核心方案**：

```
1. 过期时间加随机值：TTL = baseTime + rand(0, 300s)
2. 热点数据永不过期 + 后台异步刷新
3. 多级缓存：本地缓存 → Redis → DB（即使 Redis 雪崩，本地缓存还在）
4. 限流降级：DB 前加限流器，超过阈值直接返回降级数据
```

**追问：Redis 集群整体宕机了呢？**

那就不是缓存雪崩了，是**缓存层完全不可用**。需要：
1. 本地缓存兜底（Go 的 `go-cache`、`ristretto`，Java 的 Caffeine）
2. DB 前加熔断（如 gobreaker），防止 DB 被拖垮
3. 降级策略：返回默认值/缓存的上一次结果

🔗 **项目关联**：你设计的 Redis 降级方案就是这个思路——Redis 挂了，所有操作 fallback 到 DB（虽然慢但能用）。

### 2.4 缓存与 DB 一致性（超高频！）

**问题**：更新数据时，先更新缓存还是先更新 DB？怎么保证一致性？

**四种策略对比**：

| 策略 | 流程 | 一致性 | 问题 |
|------|------|--------|------|
| ❌ 先更新缓存，再更新 DB | SET cache → UPDATE DB | 差 | DB 失败了，缓存是脏数据 |
| ❌ 先更新 DB，再更新缓存 | UPDATE DB → SET cache | 中 | 并发写时，后写的 DB 可能先写缓存（ABA 问题） |
| ✅ 先更新 DB，再删缓存（Cache Aside） | UPDATE DB → DEL cache | 好 | 极小概率不一致（见下文） |
| ✅ 先删缓存，再更新 DB + 延迟双删 | DEL cache → UPDATE DB → sleep → DEL cache | 好 | 延迟期间可能读到旧数据 |

**推荐：Cache Aside Pattern（旁路缓存）**

```
读：先读缓存 → 缓存没有 → 读 DB → 写缓存
写：先更新 DB → 再删缓存
```

**追问："先更新 DB 再删缓存"就不会不一致了吗？**

极端情况下会：

```
T1: 线程 A 读缓存 miss → 读 DB（旧值 V1）
T2: 线程 B 更新 DB（新值 V2）
T3: 线程 B 删缓存
T4: 线程 A 把 V1 写入缓存  ← 脏数据！
```

但这个窗口**极小**，因为要求"DB 读"比"DB 写"慢，这在实际中几乎不会发生（读通常比写快得多）。

**如果你非要保证强一致**：

| 方案 | 做法 | 代价 |
|------|------|------|
| **延迟双删** | 先删缓存 → 更新 DB → sleep(500ms) → 再删缓存 | 有 500ms 脏数据窗口 |
| **Canal 订阅 binlog** | MySQL binlog → Canal → 消费者删缓存 | 引入 Canal 组件，准实时（秒级延迟） |
| **分布式锁** | 写操作加锁，同一 key 串行执行 | 影响写性能 |

**终极方案：Canal + 延迟双删组合**：

```
写请求：
  1. DEL cache              ← 第一次删
  2. UPDATE DB              ← 更新数据
  3. binlog → Canal → DEL cache  ← 异步第二次删（兜底）
```

### 2.5 多级缓存架构

**问题**：如何设计一个多级缓存体系，减少 Redis/DB 压力？

```
                    ┌─────────┐
                    │  客户端  │
                    └────┬────┘
                         │
                    ┌────▼────┐
                    │ CDN 缓存 │  ← 静态资源（图片/JS/CSS）
                    └────┬────┘
                         │
                    ┌────▼────┐
                    │ Nginx   │  ← proxy_cache / Lua（对页面级缓存）
                    └────┬────┘
                         │
                    ┌────▼─────────┐
                    │  L1: 本地缓存 │  ← 进程内，如 Caffeine / go-cache / ristretto
                    │  TTL: 10~30s  │     命中率高的热点数据
                    └────┬─────────┘
                         │ miss
                    ┌────▼─────────┐
                    │  L2: Redis    │  ← 分布式缓存，TTL: 5~30min
                    │  集群模式     │     承担绝大部分读流量
                    └────┬─────────┘
                         │ miss
                    ┌────▼─────────┐
                    │  L3: MySQL    │  ← 最终数据源
                    └──────────────┘
```

**关键设计点**：

| 层级 | TTL 策略 | 一致性 | 注意事项 |
|------|---------|--------|---------|
| L1（本地） | 短 TTL（10~30s） | 弱一致 | 多实例间本地缓存不一致，适合容忍短暂不一致的场景 |
| L2（Redis） | 中 TTL（5~30min） | 最终一致 | Canal/延迟双删保证与 DB 一致 |
| L3（DB） | 无缓存 | 强一致 | 被 L1+L2 挡掉 95%+ 的流量 |

**追问：L1 本地缓存多实例之间数据不一致怎么办？**

| 方案 | 做法 |
|------|------|
| 短 TTL | L1 只缓存 10~30s，不一致窗口很短 |
| Redis Pub/Sub | 数据变更时广播通知所有实例清除 L1 |
| Kafka 广播 | 同上，但用 Kafka（每个实例一个消费者组） |

🔗 **项目关联**：你项目的 L2 层多级缓存设计（MemStore → Redis → DB）就是这个思路的简化版。

### 2.6 Redis 大 Key 问题

**问题**：Redis 中某个 key 的 value 特别大（如一个 Hash 有 100 万个 field），会有什么问题？怎么解决？

**影响**：
- 读写耗时长 → 阻塞 Redis 单线程 → 其他请求超时
- DEL 大 key 会阻塞（Redis 6.0 前）
- 内存碎片
- 集群模式下导致数据倾斜（hot shard）

**解决方案**：

```
1. 拆分：将大 Hash 拆成多个小 Hash
   原来：user:1 → {field1, field2, ..., field1000000}
   拆后：user:1:0 → {field1~field1000}
         user:1:1 → {field1001~field2000}
         ...
   路由：bucket = hash(field) % N

2. 压缩：对 value 做 gzip/snappy 压缩
   适用于 value 是大 JSON/文本的场景

3. 异步删除：UNLINK（Redis 4.0+）代替 DEL
   UNLINK 在后台线程异步释放内存，不阻塞主线程

4. 过期策略：对大 key 设置合理 TTL + 懒加载
```

### 2.7 热点 Key 问题

**问题**：某个 key 被高频访问（如秒杀商品详情），单个 Redis 节点扛不住，怎么办？

| 方案 | 做法 | 适用场景 |
|------|------|---------|
| **L1 本地缓存** | 热点 key 在每个应用实例本地缓存一份 | 最推荐 |
| **key 分片** | `product:123` → `product:123:0` ~ `product:123:9`，随机读一个 | 读多写少 |
| **Redis 集群 + 读写分离** | 主写从读，多个从节点分担热点 key 的读压力 | 通用方案 |

## 三、数据库：MySQL 核心场景

### 3.1 分库分表

**问题**：单表数据量到了 5000 万，查询变慢，怎么做分库分表？

**分表策略**：

| 策略 | 做法 | 优缺点 |
|------|------|--------|
| **Range 分表** | 按时间/ID 范围分：order_202401, order_202402 | 简单，但热点集中在最新表 |
| **Hash 分表** | `table = order_{hash(orderId) % 16}` | 数据均匀，但扩容需迁移 |
| **一致性 Hash** | 虚拟节点 + 环形 Hash | 扩容只迁移少量数据 |

**分库策略**：

```
按业务垂直拆分：订单库、用户库、商品库
按数据水平拆分：order_db_0 ~ order_db_3，每个库 16 张表
路由：db = hash(userId) % 4, table = hash(orderId) % 16
```

**追问：分库分表后，跨库 JOIN 怎么办？**

| 方案 | 做法 |
|------|------|
| 冗余字段 | 在订单表里冗余用户名，避免 JOIN 用户表 |
| 应用层聚合 | 先查订单库，再查用户库，应用层合并 |
| 宽表/ES | 异步同步到 ES 或宽表，在宽表上做复杂查询 |
| CQRS | 读写分离——写走分库分表，读走 ES/ClickHouse |

**追问：分库分表后，全局唯一 ID 怎么生成？**

| 方案 | QPS | 特点 |
|------|-----|------|
| UUID | 无限 | 无序、长、索引性能差 |
| Snowflake | 40w/s | 有序、64bit、依赖时钟 |
| 号段模式（Leaf/Tinyid） | 极高 | DB 批量发号，本地缓存 |
| Redis INCR | 10w+/s | 简单，依赖 Redis |

🔗 **项目关联**：你项目用雪花算法生成 Job/Stage/Task/Action ID，就是分布式唯一 ID 的实践。

### 3.2 MySQL 事务隔离级别 & MVCC

**问题**：MySQL 的四个隔离级别分别解决什么问题？

```
读未提交（Read Uncommitted）→ 啥都不防，会脏读
读已提交（Read Committed）  → 防脏读，不防不可重复读
可重复读（Repeatable Read） → 防脏读+不可重复读，InnoDB 用 MVCC+Gap Lock 也防幻读
串行化  （Serializable）    → 全防，但性能最差
```

**MVCC 原理（精简版）**：

```
每行数据有两个隐藏列：
  - trx_id：最近修改该行的事务 ID
  - roll_pointer：指向 undo log（旧版本数据）

Read View（读视图）：
  - 事务开始时创建（RC 每次 SELECT 创建，RR 只在第一次 SELECT 创建）
  - 包含：当前活跃事务 ID 列表、最小活跃事务 ID、下一个待分配的事务 ID

读取规则：
  - trx_id < min_active_trx_id → 事务已提交，可见
  - trx_id ∈ 活跃事务列表 → 未提交，不可见，走 undo log 找旧版本
  - trx_id >= next_trx_id → 在 Read View 之后开始，不可见
```

**追问：RR 级别下真的完全防幻读吗？**

InnoDB 的 RR 通过 **Next-Key Lock（间隙锁+行锁）** 防止幻读，但有例外：

```sql
-- 事务 A
BEGIN;
SELECT * FROM users WHERE age > 20;  -- 快照读，用 MVCC，不加锁
-- 此时事务 B INSERT 一条 age=25 的记录并提交

UPDATE users SET name='test' WHERE age > 20;  -- 当前读！能看到事务 B 的新行
SELECT * FROM users WHERE age > 20;  -- 现在能看到 age=25 的行了！幻读！
```

原因：`UPDATE` 是**当前读**（读最新已提交数据），而 `SELECT` 是**快照读**。混合使用会出现不一致。

### 3.3 死锁分析与处理

**问题**：MySQL 什么情况下会死锁？怎么排查和解决？

**经典死锁场景**：

```sql
-- 事务 A                        -- 事务 B
BEGIN;                           BEGIN;
UPDATE orders SET state=1        UPDATE inventory SET stock=stock-1
  WHERE id=100;  -- 锁 orders:100   WHERE product_id=200;  -- 锁 inventory:200

UPDATE inventory SET stock=stock-1  UPDATE orders SET state=2
  WHERE product_id=200;  -- 等 B     WHERE id=100;  -- 等 A
-- DEADLOCK!
```

**排查**：

```sql
-- 查看最近的死锁信息
SHOW ENGINE INNODB STATUS\G
-- 关注 LATEST DETECTED DEADLOCK 部分

-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
SELECT * FROM performance_schema.data_lock_waits;  -- MySQL 8.0+
```

**解决方案**：

```
1. 统一加锁顺序：所有事务都按 orders → inventory 的顺序操作
2. 减少事务粒度：把大事务拆成小事务
3. 加锁超时：SET innodb_lock_wait_timeout = 5;  -- 等 5s 就放弃
4. 乐观锁代替悲观锁：用 CAS + version 代替 SELECT FOR UPDATE
5. 减少锁范围：用精确的 WHERE 条件避免 Gap Lock
```

🔗 **项目关联**：你项目全程使用 CAS 乐观锁（`UPDATE ... WHERE state = old_state`）而非 `SELECT FOR UPDATE`，从根本上避免了死锁。

### 3.4 慢 SQL 优化

**问题**：线上发现一条 SQL 执行了 3 秒，怎么排查和优化？

**排查流程**：

```sql
-- 1. 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过 1s 记录

-- 2. EXPLAIN 分析
EXPLAIN SELECT * FROM orders WHERE user_id = 123 AND state = 1 ORDER BY create_time DESC LIMIT 20;

-- 关注这几列：
-- type：ALL（全表扫描）→ 需要加索引
-- rows：扫描行数，越少越好
-- Extra：Using filesort（需要排序优化）、Using temporary（需要临时表优化）
```

**常见优化手段**：

| 问题 | 优化 |
|------|------|
| 全表扫描 | 加索引（覆盖索引最佳） |
| 回表查询 | 改为覆盖索引 `INDEX(user_id, state, create_time, id)` |
| 大偏移量分页 `LIMIT 100000, 20` | 改为游标分页 `WHERE id > last_id LIMIT 20` |
| 索引失效（函数/类型转换） | 避免 `WHERE YEAR(create_time) = 2024`，改为范围查询 |
| JOIN 驱动表选择不当 | 小表驱动大表，`STRAIGHT_JOIN` 强制指定 |

🔗 **项目关联**：你的 Action 表覆盖索引优化 `idx_action_stage_state_cover(stage_id, state, id, hostuuid)` 就是一个实际的慢 SQL 优化案例——250 倍扫描行数缩减。

### 3.5 MySQL 主从延迟

**问题**：写入主库后立即从从库读，读不到最新数据，怎么办？

| 方案 | 做法 | 代价 |
|------|------|------|
| **强制走主库** | 关键读操作加 `/*master*/` hint 或指定主库数据源 | 主库压力增大 |
| **等待同步** | `SELECT MASTER_POS_WAIT(file, pos, timeout)` | 阻塞等待，增加延迟 |
| **半同步复制** | `rpl_semi_sync_master_wait_point = AFTER_SYNC` | 主库写入变慢（等一个从库 ACK） |
| **业务层缓存** | 写入后把数据写入 Redis，短期内从 Redis 读 | 增加缓存复杂度 |
| **GTID + 读自己写** | 记录写操作的 GTID，读时检查从库是否已执行该 GTID | 实现精准 |

**推荐组合**：半同步复制 + 关键业务读走主库 + Redis 缓存兜底。

## 四、分布式基础设施

### 4.1 分布式锁

**问题**：分布式环境下如何实现互斥？

**三种方案**：

| 方案 | 加锁 | 优缺点 |
|------|------|--------|
| **Redis SETNX** | `SET lock_key value NX PX 30000` | 简单、高性能、但有主从切换丢锁风险 |
| **Redlock** | 半数以上 Redis 节点加锁成功才算成功 | 更可靠，但争议大（Martin Kleppmann vs Antirez 之争） |
| **ZooKeeper** | 创建临时顺序节点 | 可靠性最高，但性能差（每次加锁 2 次网络 IO） |
| **etcd** | 基于 Lease + Revision | 可靠、线性一致，Go 生态首选 |

**Redis 分布式锁的 5 个坑**：

```
坑 1：忘记设 TTL → 进程挂了，锁永远不释放
方案：SET lock_key value NX PX 30000（原子设值+TTL）

坑 2：TTL 内没处理完，锁被其他进程拿走
方案：Watch Dog 续期（Redisson 的自动续期机制）

坑 3：释放了别人的锁（A 的锁过期，B 拿到锁，A 又来 DEL）
方案：value 存自己的 UUID，释放时用 Lua 脚本判断

坑 4：Redis 主从切换，锁丢失
方案：Redlock（有争议）或 etcd/ZK

坑 5：可重入性
方案：value 存 requestId + 计数器，同一线程可重复加锁
```

```lua
-- 释放锁的 Lua 脚本（原子操作）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

🔗 **项目关联**：你的 Leader 选举就是 Redis SETNX 实现的分布式锁，TTL=30s，续期间隔=10s。

### 4.2 分布式事务

**问题**：微服务架构下，一个业务操作需要跨多个服务修改数据，怎么保证一致性？

**常见方案**：

| 方案 | 原理 | 适用场景 | 一致性 |
|------|------|---------|--------|
| **2PC / XA** | 协调者两阶段提交 | 传统数据库 | 强一致 |
| **TCC** | Try-Confirm-Cancel 三阶段 | 资金类 | 强一致 |
| **Saga** | 编排一组本地事务，失败时执行补偿 | 长事务 | 最终一致 |
| **本地消息表** | 写业务数据 + 消息表在同一事务，定时投递 | 异步场景 | 最终一致 |
| **事务消息（RocketMQ）** | Half Message + 二次确认 | MQ 场景 | 最终一致 |

**最常见：本地消息表方案**

```
服务 A（订单服务）：
  BEGIN;
    INSERT INTO orders (id, ...) VALUES (...);
    INSERT INTO outbox (id, topic, payload, state) VALUES (..., 'order-created', '{}', 'pending');
  COMMIT;
  
  -- 定时任务扫描 outbox 表，发送到 Kafka
  -- 发送成功后标记 state = 'sent'

服务 B（库存服务）：
  -- 消费 Kafka 消息
  -- 扣减库存
  -- 幂等处理（通过 outbox.id 去重）
```

**追问：本地消息表的消息一直发不出去（Kafka 挂了）怎么办？**

1. 重试 + 指数退避
2. 报警通知
3. 补偿：服务 B 提供一个"对账"接口，定时比对订单和库存

🔗 **项目关联**：你项目的"先 Kafka 后 DB"方案本质上是一种变体——用 Kafka 投递作为第一步，DB 状态变更作为第二步，失败时依赖 CleanerWorker 补偿扫描。

### 4.3 幂等性设计（通用版）

**问题**：接口被重复调用（网络重试/用户重复点击），怎么保证只执行一次？

**分层防御**：

```
前端层：按钮置灰 + 防抖节流
网关层：请求指纹去重（requestId + Redis SETNX）
业务层：幂等 Token / 状态机 / 乐观锁
数据层：唯一约束 / INSERT IGNORE / ON DUPLICATE KEY UPDATE
```

**按场景选方案**：

| 场景 | 方案 | 示例 |
|------|------|------|
| 创建类 | 唯一键 + INSERT IGNORE | 创建订单用 orderId 唯一键 |
| 更新类 | CAS 乐观锁 | `UPDATE SET state=2 WHERE id=? AND state=1` |
| 支付类 | 幂等 Token（请求前先获取 token，提交时校验并删除） | 前端先 GET /token → POST /pay + token |
| 消息消费 | 消息 ID 去重表 | `INSERT INTO consumed_msgs(msg_id) VALUES(?)` |

**幂等 Token 实现**：

```go
// 获取 Token
func GetIdempotentToken(c *gin.Context) {
    token := uuid.New().String()
    rdb.Set(ctx, "idem:"+token, "1", 10*time.Minute)
    c.JSON(200, gin.H{"token": token})
}

// 使用 Token（原子校验+删除）
func ProcessPayment(c *gin.Context) {
    token := c.GetHeader("X-Idempotent-Token")
    
    // Lua 脚本原子操作：存在则删除并返回 1，不存在返回 0
    script := `
        if redis.call("get", KEYS[1]) then
            redis.call("del", KEYS[1])
            return 1
        end
        return 0
    `
    result, _ := rdb.Eval(ctx, script, []string{"idem:" + token}).Int()
    if result == 0 {
        c.JSON(409, gin.H{"error": "duplicate request"})
        return
    }
    
    // 正常处理...
}
```

🔗 **项目关联**：你的全链路幂等设计——CAS 状态机 + DB 唯一索引 + Kafka Consumer 三态判断，是业界幂等方案的组合应用。

### 4.4 限流算法

**问题**：如何保护下游服务不被流量打垮？

**四种算法**：

| 算法 | 原理 | 特点 |
|------|------|------|
| **固定窗口** | 每秒 N 次，超过拒绝 | 简单，但窗口边界有 2N 突刺 |
| **滑动窗口** | 按时间滑动，统计最近 1s 内请求数 | 平滑，Redis ZSET 实现 |
| **漏桶** | 请求进桶，匀速流出 | 平滑流量，但无法应对合法突发 |
| **令牌桶** | 匀速产生令牌，请求消耗令牌 | 允许合法突发（桶里有存量令牌） |

```go
// Go 标准库令牌桶
import "golang.org/x/time/rate"

limiter := rate.NewLimiter(rate.Limit(1000), 100)  // 每秒 1000 个，桶容量 100

func HandleRequest(c *gin.Context) {
    if !limiter.Allow() {
        c.JSON(429, gin.H{"error": "rate limited"})
        return
    }
    // 正常处理
}
```

**分布式限流（Redis + Lua）**：

```lua
-- 滑动窗口限流 Lua 脚本
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- 删除窗口外的请求
redis.call("ZREMRANGEBYSCORE", key, 0, now - window)
-- 当前窗口内请求数
local count = redis.call("ZCARD", key)

if count < limit then
    redis.call("ZADD", key, now, now .. math.random())
    redis.call("EXPIRE", key, window / 1000)
    return 1  -- 允许
end
return 0  -- 拒绝
```

🔗 **项目关联**：你的 API 限流用的是令牌桶（`golang.org/x/time/rate`）+ 多维度限流（全局/用户/API 三级）。

### 4.5 熔断降级

**问题**：下游服务响应变慢（从 50ms 变成 5s），怎么防止连锁雪崩？

**熔断器三态**：

```
     ┌───────────────────────────────────────────┐
     │                                           │
     │   Closed ──→ Open ──→ Half-Open ──→ Closed │
     │     │          ↑         │    │            │
     │     │          │         │    └──→ Open    │
     │     │    失败率>阈值     试探请求            │
     │     └──────────┘    成功→Closed/失败→Open  │
     │                                           │
     └───────────────────────────────────────────┘
```

```go
// gobreaker 实现
import "github.com/sony/gobreaker/v2"

cb := gobreaker.NewCircuitBreaker[[]byte](gobreaker.Settings{
    Name:        "mysql",
    MaxRequests: 3,                    // Half-Open 时试探请求数
    Interval:    30 * time.Second,     // Closed 状态统计窗口
    Timeout:     15 * time.Second,     // Open 多久后进入 Half-Open
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        return counts.ConsecutiveFailures >= 5  // 连续 5 次失败 → 熔断
    },
})

// 使用
result, err := cb.Execute(func() ([]byte, error) {
    return callMySQL()
})
if err == gobreaker.ErrOpenState {
    return fallbackData()  // 降级返回
}
```

🔗 **项目关联**：你的基础设施韧性设计中，MySQL/Redis/Kafka 各有独立熔断器，参数不同（MySQL 30s/5次，Redis 15s/3次，Kafka 60s）。

**关键追问**：

**Q：Redis 扣库存和 DB 扣库存不一致怎么办？（超卖/少卖）**

```lua
-- Redis 预扣库存（Lua 原子操作）
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock <= 0 then
    return -1  -- 售罄
end
redis.call("DECR", KEYS[1])
return stock - 1
```

Redis 预扣成功但 DB 下单失败 → 需要**回补**：

```go
func ProcessOrder(msg *OrderMsg) error {
    tx := db.Begin()
    
    // DB 扣库存（CAS 防超卖）
    result := tx.Exec("UPDATE products SET stock = stock - 1 WHERE id = ? AND stock > 0", msg.ProductID)
    if result.RowsAffected == 0 {
        tx.Rollback()
        rdb.Incr(ctx, "stock:"+msg.ProductID)  // 回补 Redis
        return ErrStockInsufficient
    }
    
    // 创建订单
    if err := tx.Create(&order).Error; err != nil {
        tx.Rollback()
        rdb.Incr(ctx, "stock:"+msg.ProductID)  // 回补 Redis
        return err
    }
    
    return tx.Commit().Error
}
```

**Q：怎么防止超卖？**

三层防御：
1. Redis Lua 原子扣减（一般情况够用）
2. DB 乐观锁 `WHERE stock > 0`（兜底）
3. 最终对账（定时比对 Redis 库存和 DB 库存）

**Q：秒杀结束后，库存没抢完的 Redis 怎么和 DB 对齐？**

```
1. 秒杀结束后，以 DB 库存为准
2. 未支付订单超时关闭 → 回补 DB 库存 → 同步到 Redis
3. 定时对账任务兜底
```

### 5.3 扣减库存的三种方式

**问题**：电商场景下，什么时候扣库存？

| 时机 | 做法 | 优缺点 |
|------|------|--------|
| **下单减库存** | 创建订单时扣 | 会被恶意占库存（不支付） |
| **支付减库存** | 支付成功时扣 | 可能超卖（下单时看到有库存但支付时被抢光） |
| **预扣库存** | 下单时预扣（冻结），支付时确认，超时回补 | 最合理，但实现复杂 |

**推荐：预扣 + 超时回补**

```sql
-- 预扣（下单时）
UPDATE products SET stock = stock - 1, frozen_stock = frozen_stock + 1
  WHERE id = ? AND stock > 0;

-- 确认（支付成功时）
UPDATE products SET frozen_stock = frozen_stock - 1
  WHERE id = ?;

-- 回补（超时未支付）
UPDATE products SET stock = stock + 1, frozen_stock = frozen_stock - 1
  WHERE id = ?;
```

### 5.4 Feed 流设计（Timeline）

**问题**：设计微博/朋友圈的 Feed 流，用户看到关注人的最新内容。

**两种模型**：

| 模型 | 做法 | 适用 |
|------|------|------|
| **推模式（写扩散）** | 发布时推送到所有粉丝的收件箱 | 粉丝数少（如朋友圈） |
| **拉模式（读扩散）** | 读取时从所有关注人拉取合并排序 | 粉丝数多、关注数少 |
| **推拉结合** | 活跃用户推，不活跃用户拉；大 V 拉，普通用户推 | 通用方案 |

**推拉结合实现**：

```
发布内容：
  1. 写入 content 表
  2. 查粉丝列表
     - 粉丝 < 1000 → 推模式：LPUSH inbox:{fanId} contentId
     - 粉丝 >= 1000（大 V）→ 不推，等粉丝来拉

读取 Feed：
  1. 从 inbox 拉取推过来的内容 ID（LRANGE inbox:{myId} 0 99）
  2. 查关注的大 V 列表，拉取他们最新内容
  3. 合并排序，去重，分页返回
```

### 5.5 红包系统设计

**问题**：微信群发红包，10 个人抢一个 100 元红包，怎么设计？

**二倍均值法拆红包**：

```go
func splitRedPacket(totalAmount int64, count int) []int64 {
    packets := make([]int64, 0, count)
    remaining := totalAmount
    
    for i := 0; i < count-1; i++ {
        // 在 [1, remaining/剩余人数*2) 范围内随机
        avg := remaining / int64(count-i)
        max := avg * 2
        if max <= 1 { max = 2 }
        amount := rand.Int63n(max-1) + 1  // 至少 1 分
        packets = append(packets, amount)
        remaining -= amount
    }
    packets = append(packets, remaining)  // 最后一人拿剩余
    
    return packets
}
```

**抢红包并发方案**：

```
方案 1：Redis List（推荐）
  - 发红包时：拆好 N 份 → LPUSH red_packet:{id} amount1 amount2 ...
  - 抢红包时：LPOP red_packet:{id}（原子操作，天然防超领）
  - 记录领取：异步写 DB

方案 2：Redis + Lua
  - 判断是否已领 + 扣减计数 + 记录领取，一个 Lua 脚本搞定
```

### 5.6 分布式 ID 生成

**问题**：分布式系统下怎么生成全局唯一且递增的 ID？

**Snowflake（雪花算法）详解**：

```
┌─────────────────────────────────────────────────────────┐
│  0 │ 41 bit 时间戳（毫秒级，69 年） │ 10 bit 机器 ID │ 12 bit 序列号 │
│  1 bit                                                   │
│  符号位                                                   │
└─────────────────────────────────────────────────────────┘

- 时间戳：41 bit → 2^41 ms ≈ 69 年
- 机器 ID：10 bit → 最多 1024 个节点（可拆成 5bit 数据中心 + 5bit 机器）
- 序列号：12 bit → 每毫秒每台机器 4096 个 ID
- 总 QPS：4096 × 1000 × 1024 ≈ 40 亿/s（理论值）
```

**时钟回拨问题**：

```go
func (s *Snowflake) NextID() (int64, error) {
    now := time.Now().UnixMilli()
    
    if now < s.lastTimestamp {
        // 时钟回拨了！
        diff := s.lastTimestamp - now
        if diff < 5 { // 5ms 以内：等待
            time.Sleep(time.Duration(diff) * time.Millisecond)
            now = time.Now().UnixMilli()
        } else { // 超过 5ms：报错
            return 0, fmt.Errorf("clock moved backwards: %dms", diff)
        }
    }
    // ... 生成 ID
}
```

### 6.1 设计一个短链接系统

**核心流程**：

```
长链接 → Hash/自增ID → Base62 编码 → 短码
https://example.com/very/long/url → https://t.cn/AbCdEf
```

**关键设计**：

| 环节 | 方案 |
|------|------|
| 短码生成 | 自增 ID + Base62（1~8 位，62^8 ≈ 2180 亿） |
| 存储 | MySQL：`short_code → long_url`，Redis 缓存热点短码 |
| 跳转 | 301（永久重定向，浏览器缓存）或 302（临时重定向，可统计点击） |
| 防碰撞 | 如果用 Hash 截取，用布隆过滤器判重 |
| 高并发 | 号段模式批量发号 + Redis 缓存 |

**追问：同一个长链接，每次生成的短码一样吗？**

看需求：
- 一对一映射：先查 `long_url → short_code` 表，有则复用 → 节省存储
- 每次新建：每次生成新 short_code → 可独立统计各渠道点击数

rdb.ZIncrBy(ctx, "leaderboard:daily", 10, "user:123")

### 6.3 设计一个评论系统（树形结构）

| 方案 | 表结构 | 查询 | 写入 |
