## Q59. 数据库一共有几张表？分别存什么数据？

**[基础] [模型设计]**

### 回答

系统共有 **6 张核心表**，严格映射四层任务模型 + 基础设施：

```
┌──────────┐   1:N   ┌──────────┐   1:N   ┌──────────┐   1:N   ┌──────────┐
│   Job    │────────→│  Stage   │────────→│   Task   │────────→│  Action  │
│  操作任务 │         │  操作阶段 │         │   子任务  │         │  节点命令 │
└──────────┘         └──────────┘         └──────────┘         └──────────┘

┌──────────┐         ┌──────────┐
│   Host   │         │ Cluster  │
│  节点信息 │         │  集群信息 │
└──────────┘         └──────────┘
```

| 表名 | 用途 | 预估数据量 | 读写频率 |
|------|------|-----------|---------|
| `job` | 一个完整的操作流程（如"安装YARN"） | 千级 | 低频读写 |
| `stage` | 操作的一个阶段（Stage 间顺序执行） | 万级 | 中频读写 |
| `task` | 阶段内的子任务（Task 间并行执行） | 万级 | 中频读写 |
| `action` | 每个节点具体执行的 Shell 命令 | **百万级** | **高频读写** |
| `host` | 被管控的节点信息 | 千级 | 低频读写 |
| `cluster` | 集群信息 | 百级 | 低频读写 |

**数据量差异极大**：Job/Stage/Task 三张表数据量可控（千~万级），但 Action 表是整个系统的**数据热点**，一个 Job 就可能产生几十万个 Action（6000 节点 × ~53.5 Action/节点 = ~32 万）。

**关键关联关系**：

- Job → Stage：通过 `job_id` 关联，一个 Job 平均 16 个 Stage
- Stage → Task：通过 `stage_id` 关联，一个 Stage 平均 1~5 个 Task
- Task → Action：通过 `task_id` 关联，一个 Task 可能有 1~6000 个 Action（每个节点一个）
- Action → Host：通过 `hostuuid` 关联，每个 Action 分配到一个具体节点执行

## Q60. Action 表为什么是"热表"？数据量有多大？

**[中等] [热点分析]**

### 回答

#### 为什么是热表？

Action 表集中了系统中**最频繁的读写操作**：

| 操作 | 频率 | 来源 |
|------|------|------|
| **批量 INSERT** | 每个 Task 生成时 | TaskWorker 批量写入（每批 200 条） |
| **SELECT 全表扫描** | 每 100ms | RedisActionLoader 扫描 state=init 的 Action |
| **UPDATE 状态变更** | 每 100ms | RedisActionLoader 更新 init→cached |
| **SELECT by ID** | 每个 Agent 拉取时 | Agent 通过 gRPC 拉取 Action 详情 |
| **UPDATE 结果写入** | Agent 上报时 | 每个 Action 完成后逐条 UPDATE |
| **SELECT 进度统计** | 每 200ms | MemStoreRefresher 统计 Task/Stage 进度 |

几乎所有核心操作都涉及 Action 表，它既是**写入热点**（批量 INSERT + 大量 UPDATE），也是**查询热点**（全表扫描 + ID 查询 + 统计查询）。

#### 数据量估算

**典型场景：安装 YARN 组件（6000 节点集群）**

| 表 | 新增数据量 |
|----|-----------|
| job | 1 条 |
| stage | 16 条 |
| task | 22 条 |
| **action** | **~321,000 条** |

**累积数据量（假设每天 10 个 Job，每个 Job 平均 5000 个 Action）**：

| 时间范围 | Action 数据量 |
|---------|--------------|
| 1 天 | 5 万 |
| 1 周 | 35 万 |
| 1 月 | **150 万** |
| 6 月 | **900 万** |
| 1 年 | **1800 万** |

实际系统的 `AUTO_INCREMENT` 已经达到 **239 万+**，说明历史数据确实大量累积。

#### 热表带来的问题

1. **索引膨胀**：B+ 树层级增加，查询时需要更多的磁盘 I/O
2. **全表扫描变慢**：即使 LIMIT 2000，在百万级表上扫描 state=0 的行也需要 200ms+
3. **行锁竞争**：INSERT（生成 Action）和 UPDATE（状态更新）同时高频执行，产生锁竞争
4. **DDL 操作困难**：对百万级表加索引需要长时间锁表

## Q61. Action 表上最频繁的查询是什么？原始查询的性能如何？

**[中等] [SQL 优化]**

### 回答

#### 最频繁的查询：RedisActionLoader 扫描

```sql
-- 每 100ms 执行一次，扫描待加载到 Redis 的 Action
SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000;
```

**EXPLAIN 分析**：

```
type:  ref
key:   idx_action_state
rows:  ~50000（估算扫描行数）
Extra: Using index condition  ← 需要回表！
```

**性能问题**：

1. **索引 `idx_action_state(state)` 只包含 state 字段**，查询 `hostuuid` 需要回表——即先通过索引找到主键 ID，再回到聚簇索引读取完整行数据。
2. **state=0 的区分度低**：整张表中可能只有几个状态值（0/1/2/3/-1/-2），索引的选择性很差。
3. **百万级表 + 回表 + 低选择性索引**：耗时约 **200ms**。

#### 第二频繁：Agent 拉取时的详情查询

```sql
-- Agent 拉取 Action 时，先从 Redis 获取 ID 列表，再查 DB 获取详情
SELECT * FROM action WHERE id IN (1001, 1002, 1003) 
   OR dependent_action_id IN (1001, 1002, 1003)
ORDER BY id ASC;
```

这个查询涉及 `OR` 条件和 `dependent_action_id` 关联查询，性能一般。

#### 第三频繁：逐条 UPDATE

```sql
-- Agent 上报结果后，逐条更新 Action 状态
UPDATE action SET state = 3, exit_code = 0, result_state = 1, 
       stdout = '...', stderr = '', endtime = NOW()
WHERE id = 1001;
```

单条 UPDATE 性能尚可（主键定位 ~1ms），但在突发场景下 **2000 节点 × 50 Action = 10 万次 UPDATE**，累计耗时约 100 秒。

## Q62. 你做了 IP Hash 分片 + 覆盖索引优化，查询从 200ms 降到了 20ms，具体怎么做的？

**[进阶] [索引优化]**

### 回答

#### 优化方案分两步

**Step 1：Stage 维度查询（优先，改动最小）**

```sql
-- 优化前：全表扫描所有 pending Action
SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000;

-- 优化后：按 stage_id 维度查询（利用已有索引）
SELECT id, hostuuid FROM action WHERE stage_id = ? AND state = 0;
```

**原理**：一个 Stage 通常只有几百个 Action（一个 Stage 对应一个阶段，如"下发配置"），按 stage_id 缩小扫描范围后，查询量从百万级降到百/千级。

**Step 2：哈希分片 + 覆盖索引（进阶优化）**

```sql
-- 1. 新增 IP 哈希分片字段（MySQL Generated Column 自动维护）
ALTER TABLE action ADD COLUMN ip_shard TINYINT AS (CRC32(ipv4) % 16) STORED;

-- 2. 创建覆盖索引（包含 hostuuid，避免回表）
CREATE INDEX idx_action_shard_state ON action(ip_shard, state, id, hostuuid);

-- 3. 分片增量扫描（16 个分片并行）
SELECT id, hostuuid FROM action WHERE ip_shard = ? AND state = 0 LIMIT 100;
```

#### 覆盖索引为什么有效？

**优化前**的查询路径：

```
① 扫描索引 idx_action_state(state) 
② 找到 state=0 的记录的主键 ID
③ 回表：通过主键 ID 到聚簇索引读取 hostuuid（回表操作）
④ 重复 ③ 2000 次
```

**优化后**的查询路径（覆盖索引 `(ip_shard, state, id, hostuuid)`）：

```
① 扫描索引 idx_action_shard_state
② 索引中已经包含了 id 和 hostuuid
③ 直接从索引返回结果（不需要回表！）
```

**关键**：优化后的索引 `(ip_shard, state, id, hostuuid)` 包含了 SELECT 语句需要的所有字段（id 和 hostuuid），MySQL 可以**仅扫描索引就返回结果**，完全避免了回表操作。

#### 为什么要加 ip_shard？

直接在 `(state, id, hostuuid)` 上建索引也能实现覆盖索引，但加入 ip_shard 有额外好处：

1. **并行扫描**：16 个分片可以由多个 Server 并行消费（每个 Server 负责若干分片）
2. **缩小扫描范围**：每个分片只扫描 1/16 的数据
3. **均匀分布**：CRC32 哈希保证数据在各分片间均匀分布

#### 量化效果

| 指标 | 优化前 | 优化后（Stage 维度） | 优化后（分片 + 覆盖索引） |
|------|--------|-------------------|----------------------|
| 扫描行数 | ~50,000 | ~500 | ~100/分片 |
| 回表次数 | 2,000 | 500 | **0**（覆盖索引） |
| 查询耗时 | ~200ms | ~30ms | ~**20ms** |
| EXPLAIN Extra | Using index condition | Using where | **Using index** |

## Q63. 覆盖索引是什么？为什么能提升这个查询的性能？

**[中等] [索引原理]**

### 回答

#### 覆盖索引定义

**覆盖索引（Covering Index）**：当一个查询所需的所有字段（SELECT 列 + WHERE 条件列）都包含在某个索引中时，MySQL 可以仅通过扫描索引就返回结果，不需要回表到聚簇索引读取完整行数据。

#### InnoDB 存储结构回顾

```
聚簇索引（主键索引）：
  叶子节点存储 完整行数据（id, state, hostuuid, command_json, stdout, stderr, ...）
  数据量大（每行可能几 KB，因为 command_json/stdout 是 TEXT 类型）

二级索引（如 idx_action_state）：
  叶子节点只存储 (state, 主键id)
  数据量小
```

#### 回表的代价

```sql
SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000;
```

使用 `idx_action_state(state)` 索引时：

```
Step 1: 扫描二级索引，找到 state=0 的 (state, id) → 得到 2000 个主键 ID
Step 2: 对每个主键 ID，到聚簇索引中找到完整行 → 读取 hostuuid
        这就是"回表"，需要 2000 次随机 I/O
```

**回表在 Action 表上特别慢的原因**：

- Action 表有 `command_json`（TEXT）、`stdout`（TEXT）、`stderr`（TEXT）三个大字段
- 每行数据可能几 KB 甚至几十 KB
- 回表读取完整行时，即使只需要 `hostuuid`（几十字节），也会读取整行数据
- 2000 次回表 × 每次读取几 KB = 大量随机 I/O

#### 覆盖索引消除回表

创建 `idx_action_shard_state(ip_shard, state, id, hostuuid)` 后：

```
Step 1: 扫描这个二级索引，索引叶子节点已经包含了 (ip_shard, state, id, hostuuid)
Step 2: 查询需要的 id 和 hostuuid 都在索引里 → 直接返回，无需回表
```

在 EXPLAIN 中表现为 `Extra: Using index`（纯索引扫描），而非 `Using index condition`（需要回表）。

#### 关键限制

覆盖索引不是万能的：
1. **索引越大，写入越慢**：每个索引都需要在 INSERT/UPDATE 时维护
2. **不适合 SELECT ***：只有当查询的字段全部被索引覆盖时才生效
3. **字段类型限制**：TEXT/BLOB 类型不能直接加入索引（但 hostuuid 是 VARCHAR，可以）

在我们的场景中，RedisActionLoader 的查询只需要 `id` 和 `hostuuid` 两个字段，非常适合用覆盖索引优化。

## Q64. Action 表的状态字段区分度低（只有几个值），建索引有效果吗？你是怎么处理的？

**[进阶] [索引设计]**

### 回答

#### 问题分析

Action 的 `state` 字段只有 6 个值：

```
0 = init（初始化）
1 = cached（已缓存到 Redis）
2 = executing（执行中）
3 = success（成功）
-1 = failed（失败）
-2 = timeout（超时）
```

在百万级表上，每个状态值对应几万到几十万条记录。索引 `idx_action_state(state)` 的**选择性（Cardinality）极低**——MySQL 优化器可能认为这个索引不够高效，甚至选择全表扫描。

#### 为什么低区分度索引在这里仍然有用？

关键在于**数据分布不均匀**：

```
在任务执行过程中：
  state=0（init）：通常只有 0~几千条（新生成的，很快被加载到 Redis）
  state=1（cached）：通常只有 0~几千条（已在 Redis，等待 Agent 拉取）
  state=2（executing）：通常几百~几千条（正在执行）
  state=3（success）：几十万~百万条（历史已完成）
  state=-1（failed）：几千~几万条（失败的）
  state=-2（timeout）：几百~几千条（超时的）
```

我们查询的是 `state=0`（init），这个状态通常只有**极少量**的记录（因为 RedisActionLoader 每 100ms 就会把它们加载走）。所以虽然索引整体区分度低，但对 `state=0` 的查询来说，索引仍然很有效——它能快速定位到这少量记录。

#### 我的处理方案

**方案 1：Stage 维度查询替代状态扫描（优先采用）**

```sql
-- 不再按 state 全表扫描
-- 而是按当前正在执行的 Stage 查询
SELECT id, hostuuid FROM action WHERE stage_id = ? AND state = 0;
```

Stage 维度天然就是小范围查询（一个 Stage 通常几百个 Action），绕过了低区分度索引的问题。

**方案 2：复合索引提升选择性**

```sql
-- 原索引：idx_action_state(state) → 区分度低
-- 新索引：idx_action_stage_state(stage_id, state) → 区分度高
CREATE INDEX idx_action_stage_state ON action(stage_id, state, id, hostuuid);
```

`(stage_id, state)` 的组合区分度远高于单独的 `state`。

**方案 3：哈希分片进一步提升选择性**

```sql
CREATE INDEX idx_action_shard_state ON action(ip_shard, state, id, hostuuid);
```

`(ip_shard, state)` 的组合将数据分成 16 × 6 = 96 个桶，每个桶的数据量只有原来的 1/96。

#### 关于"冗余索引"

当前表上有两个可能重叠的索引：

```sql
KEY `action_index` (`hostuuid`, `state`, `action_type`)  -- Agent 拉取用
KEY `action_state` (`state`)                              -- 状态扫描用
```

`action_state(state)` 索引可能是**冗余**的，因为 `action_index` 的前缀 `hostuuid` 无法用于 `WHERE state=?` 查询（不满足最左前缀原则），但引入 Stage 维度查询后，`action_state` 索引的使用场景会减少，可以考虑评估删除以减少写入开销。

## Q65. 热冷数据分离是怎么做的？归档策略是什么？热表保持在多少数据量以下？

**[中等] [数据治理]**

### 回答

#### 核心设计

```
热表（action）：仅保留最近 7 天 + 所有未完成的 Action
冷表（action_archive）：存储历史已完成的 Action，用于审计和查询
```

**目标**：热表数据量控制在 **<50 万条**。

#### 归档策略

```sql
-- 定时任务：每天凌晨 2:00 执行（低峰期）
-- Step 1: 将已完成的 Action 迁移到归档表
INSERT INTO action_archive 
SELECT * FROM action 
WHERE state IN (3, -1, -2)   -- success/failed/timeout
  AND endtime < DATE_SUB(NOW(), INTERVAL 7 DAY);

-- Step 2: 从热表中删除已归档的数据
DELETE FROM action 
WHERE state IN (3, -1, -2)
  AND endtime < DATE_SUB(NOW(), INTERVAL 7 DAY)
LIMIT 10000;  -- 分批删除，避免长事务锁表
```

#### 关键设计要点

**1. 先 INSERT 后 DELETE（而非 DELETE 后补数据）**

```
✅ 正确顺序：INSERT INTO archive → 确认成功 → DELETE FROM action
❌ 错误顺序：DELETE FROM action → INSERT INTO archive（如果 INSERT 失败，数据丢失）
```

**2. 分批操作**

不能一次性迁移全部数据（可能几十万条），否则会产生长事务锁表：

```sql
-- 每批迁移 10000 条，循环执行直到完成
REPEAT
    INSERT INTO action_archive 
    SELECT * FROM action 
    WHERE state IN (3, -1, -2) AND endtime < ? 
    LIMIT 10000;
    
    DELETE FROM action 
    WHERE state IN (3, -1, -2) AND endtime < ? 
    LIMIT 10000;
    
    -- 每批之间 sleep 100ms，让出锁资源
UNTIL ROW_COUNT() = 0
```

**3. 查询路由**

```
业务查询（调度相关）→ 默认查热表 action
历史查询（审计相关）→ 查冷表 action_archive
跨表查询（如果需要）→ UNION ALL
```

**4. 归档表可以用不同的存储引擎**

归档表只做读查询，可以考虑使用压缩存储：

```sql
-- 归档表使用 COMPRESSED 行格式，节省 50%+ 存储空间
ALTER TABLE action_archive ROW_FORMAT=COMPRESSED;
```

#### 量化效果

| 指标 | 无归档 | 有归档 |
|------|--------|--------|
| 热表数据量 | 百万级（持续增长） | **<50 万**（稳定） |
| 查询性能 | 持续劣化 | **稳定** |
| 索引大小 | 持续膨胀 | **可控** |
| 备份恢复时间 | 越来越长 | **稳定在分钟级** |

## Q66. 归档用的是什么方式？直接 DELETE 还是先 INSERT INTO archive 再 DELETE？为什么？

**[中等] [归档实现]**

### 回答

采用 **先 INSERT INTO archive 再 DELETE** 的方式，原因：

#### 1. 数据安全

```
如果直接 DELETE：
  数据永久删除，无法恢复
  如果后续发现需要查询历史 Action（比如排查问题），数据已经不存在

先 INSERT 再 DELETE：
  数据迁移到 archive 表，仍然可以查询
  即使归档过程中出错，原始数据仍在 action 表中
```

#### 2. 事务安全

```go
// 伪代码：归档流程
func archiveActions(batchSize int) error {
    for {
        // 1. 先迁移数据到归档表
        result := db.Exec(`
            INSERT INTO action_archive 
            SELECT * FROM action 
            WHERE state IN (3, -1, -2) AND endtime < ? 
            LIMIT ?`, cutoffTime, batchSize)
        
        if result.RowsAffected == 0 {
            break // 没有更多数据需要归档
        }
        
        // 2. 确认迁移成功后，才从热表删除
        db.Exec(`
            DELETE FROM action 
            WHERE state IN (3, -1, -2) AND endtime < ? 
            LIMIT ?`, cutoffTime, batchSize)
        
        // 3. 每批之间暂停，避免锁表
        time.Sleep(100 * time.Millisecond)
    }
    return nil
}
```

**注意**：INSERT 和 DELETE 不在同一个事务中。如果 INSERT 成功但 DELETE 失败，下次重跑归档任务时：
- INSERT 会尝试再次插入（但归档表有唯一索引 `uk_action_id`，INSERT IGNORE 会跳过已存在的数据）
- DELETE 重新执行
- 最终一致

#### 3. 为什么不用 RENAME TABLE 交换？

有些方案用 RENAME TABLE 实现瞬间"交换"热冷表。但这不适合我们的场景：

- 我们不是全量迁移，而是按条件（state + endtime）迁移部分数据
- RENAME TABLE 是 DDL 操作，会造成短暂的表不可用
- 正在执行中的 Action（state=0/1/2）不能被迁移

## Q67. Action 表上有大量 UPDATE 操作（状态变更），MySQL 的行锁竞争是否成为瓶颈？

**[进阶] [锁竞争]**

### 回答

#### 是的，行锁竞争是实际瓶颈之一

在 200 节点扩容场景下，系统曾出现过 **死锁导致任务卡死** 的问题。具体场景：

```
并发冲突场景：

线程 A（RedisActionLoader）：
  SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000;
  → 获取了一批 Action 的共享锁（S Lock）
  UPDATE action SET state = 1 WHERE id IN (...);
  → 需要升级为排他锁（X Lock）

线程 B（Agent 上报结果）：
  UPDATE action SET state = 3, exit_code = 0 WHERE id = 1001;
  → 需要排他锁（X Lock）

如果线程 A 和线程 B 操作的 Action 在同一个数据页上，就会产生锁竞争。
```

更严重的死锁场景：

```
线程 A（节点级状态更新）：
  UPDATE taskflow_status SET status=xx WHERE node_id=? AND task_id=?
  → 持有 row-1 的 X Lock，等待 row-2

线程 B（集群级进度更新）：
  UPDATE taskflow_status SET progress=xx WHERE cluster_id=? AND task_id=?
  → 持有 row-2 的 X Lock，等待 row-1

  → 死锁！MySQL 检测到后回滚其中一个事务
  → 在 200 节点并发上报时，死锁频率急剧上升
  → 事务超时，任务卡死在 push 配置阶段
```

#### 优化方案

**1. 批量聚合 UPDATE（核心优化）**

```sql
-- 优化前：逐条 UPDATE（10 万次）
UPDATE action SET state = 3, exit_code = 0 WHERE id = 1001;
UPDATE action SET state = 3, exit_code = 0 WHERE id = 1002;
...

-- 优化后：批量 UPDATE（200ms 窗口聚合后，数百次）
UPDATE action SET state = 3, exit_code = 0 WHERE id IN (1001, 1002, 1003, ...);
```

**批量 UPDATE 减少锁竞争的原理**：
- 单次批量 UPDATE 内部，MySQL 会一次性获取所有需要的行锁
- 减少了锁的获取/释放次数（从 10 万次降到数百次）
- 单次事务持锁时间更短（批量操作比逐条操作更快完成）

**2. 乐观锁替代行锁**

```sql
-- 使用乐观锁：只有状态匹配时才更新
UPDATE action SET state = 3, exit_code = 0 
WHERE id = 1001 AND state = 2;  -- 只有 state=executing 时才更新

-- 如果返回 affected_rows = 0，说明状态已被其他线程更新，直接跳过
```

**3. 拆分更新维度**

将节点级状态更新与集群级进度更新拆分到不同的表/操作中，消除交叉锁竞争。

**4. 异步化状态更新**

在微服务架构中：
- Agent 上报结果 → ctrl-bridge 投递到 Kafka（不直接写 DB）
- ctrl-tracker 消费 Kafka → 内存聚合（200ms 窗口）→ 批量 UPDATE DB

这样 Agent 的上报请求不会直接触发 DB 写入，彻底消除了锁竞争。

## Q68. 批量聚合方案中，200ms 的时间窗口是怎么确定的？太大或太小有什么影响？

**[进阶] [批量优化]**

### 回答

#### 200ms 的设计推导

```
目标：将 10 万次逐条 UPDATE 聚合为数百次批量 UPDATE

参数估算：
  - 突发场景：2000 节点 × 50 Action = 10 万个结果需要上报
  - Agent 批量上报间隔：50ms（或 6 条聚合）
  - Agent 上报 QPS：~3000/s（Agent 端聚合后）
  - 每秒到达 Server 的结果数：~3000 批 × 6 条/批 = ~18000 条/s

  200ms 窗口内到达的结果数：18000 × 0.2 = ~3600 条
  批量 UPDATE 一次处理 3600 条 → 远好于逐条 UPDATE

  突发场景持续时间：10 万 / 18000 ≈ 5.5 秒
  总 UPDATE 次数：5.5s / 0.2s = ~28 次批量 UPDATE
  实际：10 万次 → ~28 次（降幅 99.97%）
```

#### 太大（如 1s）的影响

| 问题 | 说明 |
|------|------|
| **延迟增加** | Action 完成到状态更新到 DB 需要等 1s |
| **进度检测滞后** | Task/Stage 的进度计算延迟 1s |
| **内存压力** | 1s 窗口需要缓存更多结果（可能几万条） |
| **大事务风险** | 一次 UPDATE 几万条可能产生大事务 |

#### 太小（如 10ms）的影响

| 问题 | 说明 |
|------|------|
| **聚合效果差** | 10ms 窗口只能聚合 ~180 条，仍然很多次 UPDATE |
| **DB 压力仍大** | 5.5s / 0.01s = 550 次 UPDATE（还是太多） |
| **调度开销** | 高频定时器本身有 CPU 开销 |

#### 200ms 是平衡点

| 指标 | 10ms | 50ms | **200ms** | 500ms | 1s |
|------|------|------|-----------|-------|-----|
| 窗口内聚合条数 | 180 | 900 | **3600** | 9000 | 18000 |
| 总 UPDATE 次数 | 550 | 110 | **28** | 11 | 6 |
| 最大延迟 | 10ms | 50ms | **200ms** | 500ms | 1s |
| 内存缓存大小 | 小 | 小 | **适中** | 大 | 很大 |
| 进度检测感知 | 实时 | 近实时 | **可接受** | 略滞后 | 滞后 |

200ms 在**聚合效果（99.97% 降幅）**和**延迟容忍度（<200ms 可接受）**之间取得了平衡。

#### 配合双触发条件

实际实现中不仅看时间窗口，还有**数量阈值触发**：

```
触发批量 UPDATE 的条件（满足任一即触发）：
1. 时间窗口到期：200ms
2. 缓冲区满：累积 >= 100 条
```

这样在突发场景下（短时间大量结果到达），不需要等满 200ms 就会触发批量写入。

## Q69. GORM 的 N+1 查询问题你们遇到过吗？怎么处理的？

**[中等] [ORM 陷阱]**

### 回答

#### N+1 问题的典型场景

```go
// 查询某个 Job 的所有 Stage 及其 Task
job, _ := db.First(&Job{}, jobId)

// 查询该 Job 的所有 Stage
var stages []Stage
db.Where("job_id = ?", job.Id).Find(&stages)

// ⚠️ N+1 问题：对每个 Stage 查询其 Task
for _, stage := range stages {
    var tasks []Task
    db.Where("stage_id = ?", stage.StageId).Find(&tasks)
    // 16 个 Stage = 16 次 DB 查询
    
    // 更严重：对每个 Task 查询其 Action
    for _, task := range tasks {
        var actions []Action
        db.Where("task_id = ?", task.Id).Find(&actions)
        // 22 个 Task = 22 次 DB 查询
    }
}

// 总查询次数：1（Job）+ 1（Stages）+ 16（Tasks）+ 22（Actions）= 40 次
```

#### 实际影响

在我们的系统中，N+1 问题主要出现在 **MemStoreRefresher** 中——它需要每 200ms 将 Job/Stage/Task 的状态全量加载到内存 MemStore 中。如果用朴素的嵌套查询，每次刷新需要几十次 DB 查询。

#### 处理方式

**1. 批量查询替代循环查询**

```go
// ❌ N+1 模式
for _, stage := range stages {
    var tasks []Task
    db.Where("stage_id = ?", stage.StageId).Find(&tasks)
}

// ✅ 批量查询模式
stageIds := extractStageIds(stages)
var allTasks []Task
db.Where("stage_id IN ?", stageIds).Find(&allTasks)

// 在内存中按 stage_id 分组
tasksByStage := groupByStageId(allTasks)
```

查询次数从 16 次降到 **1 次**。

**2. 进度统计用聚合查询**

```go
// ❌ 逐个 Task 查询 Action 完成数
for _, task := range tasks {
    var count int64
    db.Model(&Action{}).Where("task_id = ? AND state = 3", task.Id).Count(&count)
}

// ✅ 一次聚合查询
type TaskProgress struct {
    TaskId  int64
    Total   int
    Success int
}
var progress []TaskProgress
db.Model(&Action{}).
    Select("task_id, COUNT(*) as total, SUM(CASE WHEN state = 3 THEN 1 ELSE 0 END) as success").
    Where("task_id IN ?", taskIds).
    Group("task_id").
    Find(&progress)
```

**3. MemStore 增量更新**

优化前 MemStoreRefresher 每次全量加载，优化后改为增量更新：

```go
// 只加载状态发生变化的记录（通过 updatetime 过滤）
var changedActions []Action
db.Where("updatetime > ?", lastRefreshTime).Find(&changedActions)
```

## Q70. 如果让你重新设计数据库，会考虑分库分表吗？怎么分？

**[进阶] [扩展方案]**

### 回答

#### 先说结论：当前规模不需要分库分表

| 评估维度 | 当前状况 | 分库分表阈值 | 结论 |
|---------|---------|------------|------|
| 单表数据量 | 百万级（归档后 <50 万） | 千万级 | ❌ 不需要 |
| QPS | 万级 | 十万级 | ❌ 不需要 |
| 写入 TPS | 千级（批量聚合后） | 万级 | ❌ 不需要 |
| 数据增长速度 | 5 万/天 | 持续高增长 | ⚠️ 长期可能 |

**冷热分离 + 批量聚合** 已经足以解决当前的性能问题。分库分表会带来巨大的复杂度（跨分片查询、分布式事务、分片键选择），投入产出比不高。

#### 如果确实需要分，怎么分？

**Action 表的分表方案**：

```
分表维度：cluster_id（集群 ID）
原因：
1. 集群是最自然的业务边界（每个集群的 Action 完全独立）
2. 所有查询都有 cluster_id 条件（或可以推导出 cluster_id）
3. 不会出现跨分片查询
```

```sql
-- 按 cluster_id 哈希分 16 张表
action_0000, action_0001, ..., action_0015

-- 路由规则
table_suffix = CRC32(cluster_id) % 16

-- 查询时自动路由
SELECT * FROM action_{suffix} WHERE cluster_id = ? AND state = 0;
```

**Job/Stage/Task 的分表方案**：

这三张表数据量小，不需要分表。但如果要分，应该使用相同的分片键（cluster_id），保证**同一个 Job 的所有关联数据在同一个分片**，避免跨分片查询。

#### 分库方案（更大规模）

```
分库维度：cluster_id
每个数据库实例负责一组集群的全部数据

DB-1: cluster_id % 3 = 0 → job, stage, task, action
DB-2: cluster_id % 3 = 1 → job, stage, task, action
DB-3: cluster_id % 3 = 2 → job, stage, task, action
```

这与**微服务拆分中的一致性哈希**完美匹配——每个 Server 实例负责一组 cluster，对应一个数据库分片。

#### 替代方案

| 方案 | 适用场景 | 我们的选择 |
|------|---------|----------|
| **冷热分离（✅）** | 数据量增长但查询集中在近期 | 当前采用 |
| 分表 | 单表超过千万且无法归档 | 未来可能 |
| 分库 | 单库 QPS 超过上限 | 暂不需要 |
| TiDB/CockroachDB | 需要水平扩展的分布式数据库 | 引入成本高 |

## Q71. 数据库的读写比例大概是多少？有没有考虑读写分离？

**[中等] [架构优化]**

### 回答

#### 读写比例分析

| 表 | 读操作 | 写操作 | 读写比 |
|----|--------|--------|--------|
| job | 查询状态、进度 | 创建、更新状态 | ~3:1 |
| stage | 查询列表、进度 | 创建、更新状态 | ~3:1 |
| task | 查询状态 | 创建、更新状态 | ~2:1 |
| **action** | **扫描 state=0、进度统计、拉取详情** | **批量 INSERT、状态 UPDATE** | **~1:2**（写多读少） |

**Action 表是"写多读少"的**：

```
写操作：
  - 批量 INSERT（TaskWorker 生成 Action）
  - 状态 UPDATE init→cached（RedisActionLoader）
  - 状态 UPDATE executing→success/failed（Agent 上报结果）
  合计写 QPS：~数千/s

读操作：
  - RedisActionLoader 扫描 state=0：10 次/s
  - Agent 拉取 Action 详情：通过 Redis 缓存，只查少量
  - 进度统计：定时查询
  合计读 QPS：~数百/s
```

#### 是否需要读写分离？

**当前不需要**，原因：

1. **Redis 已经承担了读缓存的角色**：
   - Agent 拉取 Action 先查 Redis（ZRANGE），再查 DB（SELECT by ID）
   - Redis 层已经拦截了大量读请求

2. **写操作是主要瓶颈**：
   - 批量 INSERT + 大量 UPDATE 是性能瓶颈
   - 读写分离只能分担读压力，对写压力无帮助

3. **主从复制延迟问题**：
   - MySQL 主从复制有延迟（通常 100ms~1s）
   - Action 状态更新后需要立即被读取（如进度检测），主从延迟会导致状态不一致

#### 如果要做读写分离，怎么做？

```
主库（写）：批量 INSERT、状态 UPDATE
从库（读）：进度统计查询、历史 Action 查询、Job/Stage 列表查询

注意事项：
- 强一致性读（如进度检测）必须走主库
- 弱一致性读（如历史查询、列表展示）可以走从库
```

在微服务架构中，更好的方案是**CQRS（命令查询职责分离）**：
- ctrl-dispatcher（写服务）：负责 Action 的写入
- ctrl-tracker（读服务）：负责状态查询和进度统计，可以用自己的读优化存储

## Q72. 乐观锁在 Action 状态更新中是怎么用的？版本号机制还是 CAS？

**[中等] [并发控制]**

### 回答

#### 采用 CAS 模式（基于状态字段）

我们没有引入额外的 `version` 版本号字段，而是利用 Action 本身的 **`state` 字段做 CAS（Compare-And-Swap）**：

```sql
-- CAS 更新：只有状态符合预期时才更新
UPDATE action SET state = 3, exit_code = 0, endtime = NOW()
WHERE id = 1001 AND state = 2;  -- 只有 state=executing 时才能更新为 success

-- 检查是否更新成功
-- affected_rows = 1 → 更新成功
-- affected_rows = 0 → 状态已被其他线程更新（或已超时），跳过
```

#### 为什么用状态 CAS 而非版本号？

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| **状态 CAS（✅ 采用）** | `WHERE state = old_state` | 无需新增字段、语义清晰 | 只能检测状态变化 |
| 版本号 | `WHERE version = old_version` + `SET version = version + 1` | 更通用、任意字段变更都能检测 | 需要新增字段、每次更新都要+1 |

选择状态 CAS 的原因：

1. **Action 的状态流转是严格单向的**：`init → cached → executing → success/failed/timeout`，不存在"状态没变但其他字段变了"的情况。
2. **不需要额外字段**：state 字段本身就是 WHERE 条件，不增加存储和索引开销。
3. **语义明确**：`WHERE state = 2`（executing）就表示"只在执行中的状态下更新"，比 `WHERE version = 5` 更容易理解。

#### 实际应用场景

**场景 1：防止重复完成**

```sql
-- Agent-A 和 Agent-B 都尝试将 Action-1001 标记为成功
-- 只有第一个成功的请求会生效
UPDATE action SET state = 3 WHERE id = 1001 AND state = 2;
-- Agent-A: affected_rows = 1 → 成功
-- Agent-B: affected_rows = 0 → 跳过（状态已不是 executing）
```

**场景 2：防止超时覆盖正常结果**

```sql
-- 时间线：
-- T=0s     Action-1001 开始执行（state=executing）
-- T=119s   Agent 执行完成，上报成功 → UPDATE SET state=3 WHERE state=2 ✅
-- T=120s   CleanerWorker 超时检测 → UPDATE SET state=-2 WHERE state=2 ❌（affected_rows=0）
--          因为状态已经是 3（success），不满足 WHERE state=2

-- 反过来：
-- T=120s   CleanerWorker 先超时 → UPDATE SET state=-2 WHERE state=2 ✅
-- T=121s   Agent 上报成功 → UPDATE SET state=3 WHERE state=2 ❌（affected_rows=0）
--          因为状态已经是 -2（timeout），不满足 WHERE state=2
```

**场景 3：RedisActionLoader 状态更新**

```sql
-- 只有 init 状态的 Action 才能被加载到 Redis
UPDATE action SET state = 1 WHERE id IN (...) AND state = 0;
-- 如果某个 Action 已经被其他 RedisActionLoader 实例处理了（多 Server 场景），
-- state 已经不是 0，UPDATE 自动跳过
```

#### CAS 的局限性

CAS 不能解决所有并发问题：

```
ABA 问题（在我们的场景中不存在）：
  状态流转是单向的（0→1→2→3），不会出现 A→B→A 的情况

丢失更新（需要额外处理）：
  如果两个线程同时读取同一个 Action 的 state=2，
  然后都计算出新状态=3，最终只有一个成功。
  但在我们的场景中这是正确的行为——同一个 Action 只应该被一个 Agent 执行。
```
