## 一、技术选型追问 — "为什么用 X 不用 Y？"

> **面试官心理**：技术选型是工程判断力的直接体现。不想听"因为公司用了所以我用了"，想听你对多种方案的理解深度和权衡逻辑。

### 1.1 "Kafka 做事件驱动，为什么不用 RocketMQ / RabbitMQ / Pulsar？"

**参考回答**：

| 维度 | Kafka | RocketMQ | RabbitMQ | Pulsar |
|------|-------|----------|----------|--------|
| 吞吐 | 百万级/s | 十万级/s | 万级/s | 百万级/s |
| 延迟 | ms 级 | ms 级 | μs 级 | ms 级 |
| 消息回溯 | ✅ offset 回溯 | ✅ 时间戳回溯 | ❌ 消费即删 | ✅ cursor 回溯 |
| 顺序语义 | Partition 级有序 | Queue 级有序 | 不保证 | Key 有序 |
| 延迟消息 | ❌ 原生不支持 | ✅ 18 个级别 | ✅ TTL + DLX | ✅ 原生支持 |
| 生态成熟度 | 最高（大数据生态） | 国内成熟 | 中等 | 新兴 |

选 Kafka 的核心理由：
1. **平台已有 Kafka 集群**（大数据平台天然依赖 Kafka），运维成本为零
2. **需要消息回溯能力**——"先 Kafka 后 DB" 策略依赖 Consumer offset 回溯来做故障恢复
3. **Partition 级有序**正好匹配 Job 维度的有序消费需求（同一 Job 的事件路由到同一 Partition）
4. 不选 RabbitMQ：吞吐不够，且消费即删无法回溯
5. 不选 RocketMQ：功能上可以，但公司技术栈没有，引入额外运维负担
6. 不选 Pulsar：存算分离架构更适合超大规模，当前场景 overkill

**追问陷阱**：

> "你说 Kafka 不支持延迟消息，那你的任务超时怎么做的？"

回答：超时由 CleanerWorker 的定时扫描补偿，5 秒一轮扫描 DB 中超过 SLA 的 Task，而不是依赖消息延迟投递。这是 **Pull 模型补偿** vs **Push 模型触发** 的权衡——Pull 模型实现简单、可控性强，代价是最多 5 秒的超时发现延迟。

### 1.2 "MySQL 做状态存储，为什么不用 etcd / Redis / TiDB？"

**参考回答**：

| 需求 | MySQL | etcd | Redis | TiDB |
|------|-------|------|-------|------|
| 事务 ACID | ✅ | ❌ | ❌ | ✅ |
| 复杂查询 | ✅ | ❌ | ❌ | ✅ |
| 存储容量 | TB 级 | GB 级 | 受内存限制 | PB 级 |
| 运维成本 | 低（团队熟悉） | 中 | 低 | 高（需要 PD/TiKV） |

核心理由：
1. 任务调度系统的**状态机转换**需要强事务保证——CAS 乐观锁 (`UPDATE ... WHERE state = old_state`) 依赖数据库事务隔离
2. **复杂聚合查询**需求（如 TaskCenterWorker 需要聚合 Stage 下所有 Task 状态）不适合 KV 存储
3. 不选 etcd：容量太小（官方建议 < 8GB），且不支持复杂查询
4. 不选 Redis：非持久化设计做主存储不靠谱（虽然有 AOF/RDB，但 Redis 的定位是缓存）
5. 不选 TiDB：运维复杂度远高于 MySQL，当前数据量（万级 Job/日）完全不需要分布式数据库

**面试官真正想考什么**：不是要你背选型对比表，而是看你能不能从**业务需求**反推出存储选型的决策路径。

### 1.3 "Go 写后端，为什么不用 Java？"

**参考回答**：

这不是语言优劣问题，是**团队和场景匹配**：
1. **协程模型天然适合任务调度**——6 个 Worker 跑 6 个 goroutine，比 Java 线程池模型更轻量
2. **编译型单二进制部署**——Agent 部署到数千台机器上，不需要 JVM 环境，部署体积从 200MB+ 降到 20MB
3. **大数据平台组的技术栈就是 Go**——TBDS 核心组件（Hive Gateway, HDFS Proxy 等）全是 Go 写的
4. 如果用 Java：Spring Boot + 线程池也完全能做，但 goroutine + channel 的 producer-consumer 模式写起来更自然

**追问陷阱**：

> "Go 的 GC 对你的系统有影响吗？有没有做过 GC 调优？"

回答：诚实说影响很小。任务调度系统不是低延迟系统（不像交易引擎要求 μs 级延迟），Go 1.19+ 的 GC 暂停时间已经在 1ms 以内。我们做过的优化是**减少堆上小对象分配**——MemStore 的 Stage/Task 队列用 sync.Pool 复用 slice，降低 GC 扫描开销。

### 1.4 "gRPC 做 Server-Agent 通信，为什么不用 HTTP/REST？WebSocket？"

**参考回答**：

| 维度 | gRPC | HTTP/REST | WebSocket |
|------|------|-----------|-----------|
| 通信模式 | 四种（Unary/Stream） | 请求-响应 | 全双工 |
| 序列化 | Protobuf（二进制） | JSON（文本） | 自定义 |
| 代码生成 | ✅ 强类型 | ❌ 手写解析 | ❌ |
| 双向通信 | 双向流 | 需要轮询/SSE | 原生支持 |
| 网络开销 | 低（HTTP/2 多路复用） | 高（每请求新连接或 Keep-Alive） | 低 |

选 gRPC 的理由：
1. **Heartbeat 场景**：Agent 每 5s 心跳一次，gRPC 的 HTTP/2 长连接 + Protobuf 序列化比 HTTP/JSON 节省 80%+ 网络开销
2. **双向通信需求**：心跳捎带通知（Piggyback）需要 Server 在心跳响应中推送命令，Unary RPC 天然支持
3. **强类型 IDL**：Protobuf 定义接口，Server/Agent 代码生成保证一致性，避免 JSON 字段拼错的运行时 bug
4. 不选 WebSocket：没有标准化的 IDL 和代码生成，长连接管理更复杂
5. 不选 HTTP/REST：心跳轮询的网络开销太大，且无法利用 HTTP/2 特性

**追问陷阱**：

> "如果 Agent 在内网不同网段，gRPC 能穿透防火墙吗？"

回答：gRPC 基于 HTTP/2，走 443/80 端口，和 HTTPS 一样穿透防火墙。真正可能遇到的问题是**中间代理不支持 HTTP/2**（比如某些老旧 Nginx 版本）。解决方案：1. 升级代理，2. 使用 gRPC-Web 做降级，3. Envoy 做 sidecar 代理。

### 1.5 "Redis 做缓存/分布式锁，为什么不用 Memcached / Zookeeper？"

**参考回答**：

缓存选 Redis 不选 Memcached：
1. 需要**数据结构**——心跳状态用 Hash，通知队列用 Sorted Set，纯 KV 的 Memcached 做不了
2. 需要**持久化兜底**——Redis 的 AOF 至少能在重启后恢复缓存热数据
3. 需要**Lua 脚本原子操作**——分布式锁的获取+设置过期时间需要原子性

分布式锁选 Redis 不选 ZooKeeper：
1. ZK 锁基于临时节点 + Watch，**可靠性更高**（不存在锁自动过期的问题）
2. 但 ZK 锁的**性能开销大**（每次加锁都是一次 ZK 写入 + 同步复制），在高频场景下（MemStoreRefresher 200ms 一轮）不适合
3. 我们的锁是**保护性锁**（防止双主并发写 MemStore），不是**互斥资源锁**——即使偶尔锁续期失败导致双重处理，CAS 乐观锁会兜底，所以对锁的可靠性要求不如金融场景高

## 二、规模与极限追问 — "扩大 10 倍怎么办？"

> **面试官心理**：考察你有没有**架构演进意识**。不是要你现在就做到，而是看你能不能预见瓶颈、知道优化路径。

### 2.1 "你说现在万级 Job/日，如果变成百万级呢？"

**参考回答**：

**当前架构瓶颈分析**（按触及顺序）：

```
                  当前        10x         100x        瓶颈类型
Job 数/日         1万         10万        100万        
──────────────────────────────────────────────────────────
MySQL 写入       ~100 TPS    ~1000 TPS   ~10000 TPS   ⚠️ 单机 MySQL 极限 ~5000 TPS
Kafka 消息量     ~500/s      ~5000/s     ~50000/s     ✅ 单集群轻松扛住
Redis 操作       ~2000/s     ~20000/s    ~200000/s    ✅ 单实例 10W+ QPS
MemStore 队列    ~1000       ~10000      ~100000      ⚠️ 内存 + 刷新延迟
Agent 连接数     ~100        ~1000       ~10000       ⚠️ 单 Server gRPC 连接数
```

**10 倍（10 万 Job/日）优化路径**：

1. **MySQL 读写分离**：MemStoreRefresher 的 SELECT 走从库，减轻主库压力
2. **MemStore 改增量刷新**：当前是全量 SCAN，改为基于 Kafka 事件的增量更新——收到 StageActivatedEvent 就直接入队，不再依赖 200ms 定时全量扫描
3. **Server 水平扩展**：从单主改为**分片主**——不同 processCode 的 Job 路由到不同 Server 实例，每个实例只处理一部分 Job

**100 倍（100 万 Job/日）优化路径**：

4. **MySQL 分库分表**：按 job_id 做 hash 分片，分 16 库 × 64 表
5. **独立 Task 执行引擎**：当前 StageWorker/TaskWorker 在同一进程，拆成独立的 Task Executor 集群，通过 Kafka 消费任务
6. **Agent 连接层独立**：引入连接网关（类似 MQTT Broker），Server 不再直接持有 Agent 连接

### 2.2 "单个 Job 包含 1 万个 Task，你的 TaskCenterWorker 扛得住吗？"

**参考回答**：

扛不住，这是已知瓶颈。

**问题分析**：TaskCenterWorker 的 `checkStage()` 方法：
```
1. 查询 Stage 下所有 Task（SELECT * FROM task WHERE stage_id = ?）
2. 对每个 Task，查询其所有 Action（SELECT * FROM action WHERE task_id = ?）
3. 聚合判断 Stage 是否完成
```

1 万 Task 意味着 1 次 SELECT + 1 万次 SELECT = **1 万零 1 次 DB 查询**，而且没有事务保护，中间状态可能变化。

**优化方案**：

1. **批量查询**：`SELECT * FROM action WHERE task_id IN (?, ?, ..., ?)` 一次查出所有 Action，减少到 2 次 DB 查询
2. **计数器方案**：Stage 表增加 `total_tasks`、`completed_tasks`、`failed_tasks` 计数字段，每次 Task 状态变更时原子递增（`UPDATE stage SET completed_tasks = completed_tasks + 1 WHERE id = ? AND completed_tasks < total_tasks`），Stage 完成判断变成 `completed_tasks == total_tasks`，无需遍历
3. **Event-Driven 替代 Polling**：Task 完成后发 TaskCompletedEvent 到 Kafka，Stage 聚合器消费事件做增量计算，不再轮询 DB

方案 2 是性价比最高的，一个 `UPDATE` 搞定，不引入额外复杂度。

### 2.3 "Agent 从 100 台扩到 10000 台，心跳怎么扛？"

**参考回答**：

**当前心跳设计**：

- 每个 Agent 每 5s 发一次心跳（gRPC Unary）
- 100 Agent → 20 QPS → 完全没压力
- 10000 Agent → 2000 QPS → 单实例还行，但有优化空间

**瓶颈在哪**：
- 不在网络——2000 QPS 对 gRPC 是小意思
- **在 Redis**——每次心跳要更新 Redis Hash（Agent 状态）+ ZCARD 检查通知队列 = 4000 次 Redis 操作/s
- **在 DB**——心跳捎带的 Action 进度上报需要写 DB

**优化路径**：

1. **心跳聚合**：引入 Agent Gateway 层，Gateway 批量聚合心跳后批量写 Redis（Pipeline），2000 次单写 → 20 次批量写（每批 100）
2. **动态心跳间隔分级**：
   - 空闲 Agent：30s 一次（当前 5s 太频繁）
   - 有任务 Agent：5s 一次
   - 活跃执行中 Agent：1s 一次
   
   10000 台中 80% 空闲 → 实际心跳 QPS：8000/30 + 1600/5 + 400/1 = 267 + 320 + 400 ≈ **987 QPS**，压力降低一半
3. **进度上报解耦**：Action 进度不走心跳，走独立的 gRPC Stream 上报 + 本地 WAL 缓冲批量提交

### 2.4 "如果 Kafka 的某个 Partition 消息严重积压了怎么办？"

**参考回答**：

**识别积压**：
- 监控 Consumer Lag（Kafka Exporter + Prometheus + Grafana）
- 设告警阈值：Lag > 10000 或 Lag 增长速率 > 100/s 触发 P1 告警

**短期应急**（分钟级恢复）：
1. **增加 Consumer 实例**：Partition 数允许的范围内增加 Consumer 并行度
2. **临时跳过非关键消息**：如果积压的是通知类消息，配置 Consumer 跳过 10 分钟前的旧消息
3. **降级处理**：Consumer 端临时关闭非核心逻辑（如指标上报、审计日志）

**中期根治**：
4. **排查积压原因**：
   - Consumer 处理慢？→ 优化处理逻辑、增加并行度
   - Producer 突发洪峰？→ 上游限流
   - 某条消息导致 Consumer 报错重试？→ 配死信队列，毒丸消息隔离
5. **增加 Partition 数**：从 6 扩到 24，提高并行消费能力（注意：Partition 数只能增不能减）

**本项目关联**：
我们的 TaskEventTopic 用 job_id 做 Partition Key，如果某个热点 Job 产生大量事件，会导致单 Partition 热点。解决方案：热点 Job 的事件 Key 改为 `job_id + stage_id`，在保证 Stage 级有序的前提下分散到多个 Partition。

## 三、故障全链路追问 — "X 挂了走一遍"

> **面试官心理**：这是区分"写过代码"和"做过系统"的分水岭。能把故障链路说清楚的人，才是真正理解过系统的人。

### 3.1 "Job 提交后，Server 在 StageWorker 处理 Stage 的过程中突然宕机，全链路走一遍"

**参考回答**（按时间线拆解）：

**T0：Job 提交**
- Client 调 gRPC CreateJob
- Server 写 MySQL（Job=Init, 第一个 Stage=Init）
- 发 Kafka JobCreatedEvent

**T1：JobWorker 处理**
- 消费 JobCreatedEvent（或定时扫描到 Init Job）
- UPDATE Job SET state=Running, UPDATE Stage SET state=Ready
- 发 Kafka StageActivatedEvent

**T2：StageWorker 处理（宕机时刻）**
- 从 MemStore 取到 Ready Stage
- 调用 Producer 创建 Task → **写了一半 Task 到 DB，还没全部写完**
- 💥 Server 宕机

**T3：恢复路径**
- 场景 A（Leader 切换）：新 Leader 启动，MemStoreRefresher 200ms 后扫描 DB，发现 Stage 状态=Ready → 重新入 MemStore 队列 → StageWorker 再次处理
- **关键问题**：之前写了一半的 Task 怎么办？
  - Producer 的 `Execute()` 方法使用 `INSERT IGNORE`，重复创建的 Task 会被唯一索引忽略
  - 已经创建的 Task 状态还是 Init，会被正常调度
  - **结论：无数据丢失，无脏数据，自动恢复**

**面试官可能追问**：
> "如果 Producer 是先批量 INSERT Task，再 UPDATE Stage 为 Processing，宕机发生在两者之间呢？"

回答：Stage 还是 Ready，所以 MemStoreRefresher 会再次捞起来。Producer 再次执行时，INSERT IGNORE 跳过已存在的 Task，UPDATE Stage 为 Processing——**幂等性保证了重复执行的安全性**。

### 3.2 "Kafka 集群整个挂了（不是单 Broker 挂），系统会怎样？"

**参考回答**：

**受影响的链路**：
1. ❌ 事件驱动链路断裂：StageActivatedEvent、TaskCompletedEvent 等无法发送/消费
2. ❌ 异步通知推送失败：Agent 通知队列（如果走 Kafka）失效
3. ✅ 心跳不受影响：走 gRPC 直连，不依赖 Kafka
4. ✅ DB 读写不受影响

**降级行为**：

```
正常链路:  事件驱动（Kafka）→ 实时处理
降级链路:  定时补偿扫描（CleanerWorker）→ 5s 延迟处理
```

- CleanerWorker 每 5s 扫描一次 DB，找出"应该推进但没推进"的 Job/Stage/Task
- `retryFailedTasks()` 和 `processStuckStages()` 就是 Kafka 的**地面备份**
- **代价**：任务推进延迟从**亚秒级**退化到 **5-10 秒级**

**恢复后**：
- Kafka 恢复后，Consumer 从上次的 committed offset 继续消费
- 可能出现**重复消费**（Kafka 恢复期间 CleanerWorker 已经推进了一些 Task，Kafka 恢复后事件又来一遍）
- 防御：3-state 幂等消费——Consumer 收到事件后先检查 DB 状态，如果已经是目标状态就 skip

### 3.3 "两台 Server 同时认为自己是 Leader（脑裂），会发生什么？"

**参考回答**：

这是分布式锁最经典的故障场景。

**发生条件**：Redis 锁 TTL 过期（网络分区或 GC 暂停导致锁续期失败），两台 Server 同时持有 Leader 身份。

**影响链**：
1. 两个 MemStoreRefresher 同时扫描 DB → 两个 MemStore 各自入队
2. 两个 StageWorker 同时取到同一个 Stage → 同时调用 Producer
3. Producer 批量 INSERT Task →**INSERT IGNORE 唯一索引** 保证只有一份 Task
4. 两个 TaskWorker 同时取到同一个 Task → 同时下发给 Agent →**Agent 收到重复指令**

**防御层级**：

```
Layer 1: Redis 分布式锁（乐观防线，正常情况下阻止双主）
Layer 2: DB CAS 乐观锁（UPDATE ... WHERE state = old_state，只有一个能成功）
Layer 3: INSERT IGNORE 唯一索引（防止重复创建）
Layer 4: Agent 端幂等执行（相同 task_id 的命令只执行一次）
```

**总结**：脑裂时系统不会**数据错乱**，最坏情况是**任务重复执行一次**。对于我们的场景（大数据任务调度，比如启动 HDFS 服务），重复执行的危害远小于丢失执行。

### 3.4 "Redis 主从切换期间，分布式锁失效了，正在执行的任务怎么办？"

**参考回答**：

**场景**：Redis Master 宕机 → Sentinel 提升 Slave 为新 Master → 但 Slave 可能没同步到最新的锁数据 → 锁丢失。

**影响**：
- 正在执行的任务**不受影响**——任务已经下发给 Agent，Agent 本地执行不依赖 Redis
- Leader 选举可能短暂异常——新 Master 上没有锁信息，多台 Server 可能同时获得锁

**我们的处理**：
1. **锁的 Value 带 ownerID**：即使多台 Server 获得锁，释放锁时检查 ownerID，不会误释放别人的锁
2. **Sentinel 模式已经足够**：我们不需要 Redlock（多 Master），因为 Layer 2-4 的防御已经覆盖了锁失效场景
3. **看门狗机制**：锁持有者每隔 TTL/3 续期一次，Redis 切换完成后（通常 < 30s），新的锁竞争自然恢复正常

## 四、设计权衡追问 — "牺牲了什么？"

> **面试官心理**：世界上没有完美的设计，能说出自己方案的代价和局限，比只说优点更有说服力。

### 4.1 "先 Kafka 后 DB 确实避免了链路断裂，但牺牲了什么？"

**参考回答**：

牺牲了**数据最终一致的时间窗口**和**系统复杂度**。

**代价 1：Consumer 端需要处理"消息到了但 DB 没写"的时间窗**
- Kafka 消息已经发出，但 DB 写入还没完成（或失败了）
- Consumer 消费到事件后查 DB，发现数据不存在
- 解决方案：3-state 幂等消费（Pending → Processing → Done），遇到数据不存在就 retry with backoff
- **额外复杂度**：每个 Consumer 都要实现 3-state 状态机，代码量增加 30%+

**代价 2：极端情况下可能出现"幽灵事件"**
- Kafka 消息发送成功，但后续 DB 写入失败（比如 MySQL 宕机）
- 此时 Kafka 里有事件，但 DB 里没有对应数据
- Consumer 会一直 retry 直到超过最大重试次数，最终进入死信队列
- **需要人工介入**处理死信队列里的幽灵事件

**代价 3：调试复杂度上升**
- 出问题时要同时看 Kafka 消息轨迹 + DB 状态 + Consumer 日志
- 比纯 DB 事务方案的排查链路长得多

**为什么还是选它**：因为"先 DB 后 Kafka"的代价更大——Stage 链路断裂意味着整个 Job 卡死，需要人工介入恢复，而"先 Kafka 后 DB"的代价都可以自动化处理。

### 4.2 "MemStore 全量扫描 200ms 一轮，为什么不用 Kafka 事件驱动 MemStore？"

**参考回答**：

好问题，这确实是一个权衡。

**当前方案（Pull 模型 — 定时全量扫描）**：
- 优点：实现简单，MemStore 的数据永远和 DB 保持最终一致
- 缺点：200ms 延迟 + 全量扫描开销（万级 Job 时 SQL 压力大）

**替代方案（Push 模型 — Kafka 事件驱动）**：
- 优点：实时性好，MemStore 收到事件立即更新，无扫描开销
- 缺点：
  1. **MemStore 和 DB 可能不一致**——Kafka 消息丢了/延迟了，MemStore 里就缺数据
  2. **需要全量兜底**——即使用事件驱动，还是需要定时全量扫描做对账，否则数据会越来越不一致
  3. **Leader 切换时需要全量重建**——新 Leader 的 MemStore 是空的，必须从 DB 全量加载

**结论**：纯 Push 不够可靠，最终还是要 Pull 做兜底。当前规模下直接用 Pull 就够了，等规模上来后改为 **Push + Pull 混合**（事件驱动为主，每 5 分钟全量对账一次）。

### 4.3 "CAS 乐观锁 vs 悲观锁，你怎么选的？为什么？"

**参考回答**：

**选乐观锁的理由**：

1. **冲突概率低**：正常情况下只有一个 Leader 在操作，CAS 冲突只在脑裂等异常场景出现
2. **不需要持有锁**：悲观锁（`SELECT ... FOR UPDATE`）需要在事务期间持有行锁，如果事务时间长（比如 StageWorker 调用外部 Producer），会严重影响并发
3. **天然幂等**：`UPDATE task SET state='Running' WHERE id=? AND state='Init'` 返回 affected_rows=0 就知道被抢了，不需要额外的幂等逻辑
4. **无死锁风险**：悲观锁多表操作顺序不一致会死锁，CAS 完全不涉及锁等待

**什么场景该用悲观锁**：
- 资源扣减（库存、余额）——冲突概率高，CAS 会大量 retry 浪费 CPU
- 需要读取最新值再决策——CAS 的"读-改-写"窗口可能读到过期数据

### 4.4 "四层模型 Job → Stage → Task → Action，为什么要这么多层？三层不行吗？"

**参考回答**：

**为什么不是三层（去掉 Stage）**：

Stage 的存在解决了一个关键问题——**有序依赖编排**。

```
Job: "部署 Kafka 集群"
  Stage 1: "安装 JDK"         → 必须先完成
  Stage 2: "部署 ZooKeeper"   → 依赖 Stage 1
  Stage 3: "部署 Kafka Broker" → 依赖 Stage 2
  Stage 4: "配置 Topic"       → 依赖 Stage 3
```

如果没有 Stage，要么：
- 所有 Task 拍平 → 失去有序性，必须自己维护依赖图（DAG），复杂度爆炸
- 用 Task 的前后依赖 → 每个 Task 要知道自己的前置 Task 是否完成，聚合逻辑散落在各处

Stage 的 **链表推进模型**（每个 Stage 有 NextStageId）让编排逻辑集中在 StageWorker + ResultAggregator 两处，其他组件无需关心执行顺序。

**为什么不是五层（再加一层）**：

曾考虑过在 Action 下再加 Step（一个 Action 分多步），但：
- 实际场景中 Action 粒度已经够细（一条 shell 命令）
- 再加一层带来的**状态机复杂度翻倍**（每层都有 Init/Running/Success/Failed 四态）
- 收益 < 成本，所以停在四层

> **面试官心理**：这是验证你有没有**真正上过战场**的问题。背八股文的人在这里露馅。

### 5.1 "上线后遇到的最诡异的 Bug 是什么？"

**参考回答**：

**Bug：Ghost Stage — Leader 切换后 Stage 消失**

**现象**：某个 Job 的 Stage 2 状态显示 Ready，但 StageWorker 就是不处理它。日志里完全没有 Stage 2 的处理记录。

**排查过程**：
1. 检查 MemStore — Stage 2 不在队列里
2. 检查 DB — Stage 2 状态确实是 Ready
3. MemStoreRefresher 日志 — 确实在扫描，但查询条件 `WHERE state = 'Ready' AND id NOT IN (processedStages)` 排除了它
4. **processedStages 里居然有 Stage 2 的 ID！**

**根因**：
- T0：旧 Leader 的 StageWorker 取出 Stage 2，加入 processedStages，开始处理
- T1：旧 Leader 失去锁，变成 Follower，StageWorker 停止但 **processedStages 没有清空**
- T2：旧 Leader 重新获得锁，变成新 Leader，processedStages 还残留着 Stage 2 的 ID
- T3：MemStoreRefresher 扫描时排除了 processedStages 里的 ID → Stage 2 永远不会被捞起

**修复**：Leader 角色变更时（无论失去还是重新获得），**清空 processedStages 和 processedTasks**。

**面试官评价点**：这个 Bug 涉及到分布式系统中**状态残留**的经典问题，说出来能体现你确实 debug 过真实的分布式系统。

### 5.2 "性能调优做过什么？"

**参考回答**：

**案例 1：Action 表 longtext 导致主从延迟**

- 问题：Action 表的 stdout/stderr 字段是 longtext，单行可能几 MB
- 影响：Binlog 传输变慢 → 主从延迟从 100ms 飙到 10s+
- 排查：`SHOW SLAVE STATUS` 看到 `Seconds_Behind_Master` 持续增长
- 修复：
  1. stdout/stderr 改为存 OSS/MinIO，DB 只存 URL
  2. 过渡期：限制 longtext 最大长度为 64KB，超出截断 + 存 OSS

**案例 2：MemStore 扫描慢查询**

- 问题：MemStoreRefresher 的 SQL `SELECT * FROM stage WHERE state IN ('Ready', 'Running')` 没有索引
- 影响：Stage 表数据量增长到 50 万后，扫描耗时从 5ms 涨到 500ms
- 修复：`ALTER TABLE stage ADD INDEX idx_state (state)` + 定期归档已完成的 Stage 到历史表

**案例 3：gRPC 心跳连接泄漏**

- 问题：Agent 异常退出后，Server 端的 gRPC 连接没有及时释放
- 影响：长时间运行后 Server 的 fd 数持续增长，逼近 ulimit
- 修复：
  1. Server 端设置 `keepalive.ServerParameters{MaxConnectionAge: 5 * time.Minute}`
  2. 心跳超时检测：30s 无心跳 → 主动断开连接

### 5.3 "最难排查的线上问题是什么？用了什么工具？"

**参考回答**：

**问题：CleanerWorker 重试风暴导致 Job 反复执行**

**现象**：某个 Job 在凌晨 3 点被反复重试，一小时内执行了 47 次。

**排查链路**：
1. **Grafana 指标看板**：Job 执行次数曲线在 3:00 AM 突然飙升
2. **Kibana 日志**：搜索 job_id 发现大量 `retryFailedTasks: resetting task to Init` 日志
3. **MySQL 慢查询日志**：发现 CleanerWorker 的 `UPDATE` 语句执行频繁
4. **代码走查**：发现 `retryFailedTasks()` 的逻辑——
   ```
   重置 Task 为 Init → 重置 Stage 为 Running → 重置 Job 为 Running
   ```
   但**没有检查同 Stage 下其他 Task 的状态**！

**根因**：
- Stage 有 3 个 Task：Task A 成功、Task B 失败、Task C 正在执行
- CleanerWorker 重试 Task B → 重置 Stage 为 Running → 但 Task C 还在跑
- Task C 完成后，TaskCenterWorker 判断"所有 Task 完成" → 推进到下一 Stage
- 下一 Stage 执行完 → 整个 Job 标记为 Success
- 但此时 Task B 的重试也完成了 → TaskCenterWorker 再次判断 → 又推进一次
- **循环往复**

**修复**：
- CleanerWorker 重试 Task 前，检查同 Stage 下是否有其他 Running Task
- 如果有，只重置单个 Task，不动 Stage/Job 状态
- 增加重试次数上限（max_retries = 3），超过后标记为 PermanentFailed

**用到的工具**：
- Grafana + Prometheus：指标监控
- Kibana + ELK：日志搜索
- MySQL slow_query_log：慢查询定位
- `pprof`：排除 Server 内存泄漏的可能性

## 六、知识深度追问 — "你说的这个展开讲讲"

> **面试官心理**：你提到了一个概念，面试官要确认你不是只知道名词，而是理解底层原理。

### 6.1 "你说用了 CAS 乐观锁，MySQL 底层怎么实现的？"

**参考回答**：

CAS 本身不是 MySQL 的原生概念，我们用 SQL 语义模拟 CAS：

```sql
UPDATE task SET state = 'Running', version = version + 1 
WHERE id = 123 AND state = 'Init';
-- affected_rows == 1 → 成功
-- affected_rows == 0 → 被其他线程抢先修改了
```

**MySQL 底层执行路径**：
1. InnoDB 找到 id=123 的行 → **加行锁**（即使是乐观锁实现，UPDATE 执行时也需要行级锁）
2. 检查 `state = 'Init'` 条件 → 满足则修改，不满足则返回 affected_rows=0
3. 释放行锁

**关键认知**：
- "乐观锁"不是"不加锁"——UPDATE 执行瞬间还是要加 X 锁的
- "乐观"体现在**不提前加锁、不长时间持有锁**——读数据时不加锁，只在写入时通过条件检查做冲突检测
- 对比悲观锁（`SELECT ... FOR UPDATE`）：在读取时就加锁，直到事务结束才释放

### 6.2 "你提到 Kafka Partition 有序，能讲讲 Kafka 怎么保证 Partition 内有序的？"

**参考回答**：

**Producer 端**：
- 同一 Partition 的消息通过 **TCP 连接串行发送**（每个 Partition 一个 produce request queue）
- `max.in.flight.requests.per.connection = 1` 时严格有序（否则可能因重试乱序）
- `enable.idempotence = true` 开启幂等 Producer，通过序列号防止重试导致的乱序

**Broker 端**：
- 每个 Partition 是一个 **append-only 的 commit log**
- 消息按到达顺序追加，分配单调递增的 offset
- 不存在"插入"或"修改"操作 → 天然有序

**Consumer 端**：
- 每个 Partition 只能被**同一 Consumer Group 的一个 Consumer** 消费
- Consumer 按 offset 顺序 poll → 消费顺序 = 写入顺序

**本项目的用法**：
- job_id 做 Partition Key → `hash(job_id) % partition_count` → 同一 Job 的事件进同一 Partition
- 保证同一 Job 的 StageActivatedEvent 按顺序消费 → Stage 链表按顺序推进

### 6.3 "你提到了 gRPC 基于 HTTP/2，HTTP/2 的多路复用是怎么回事？"

**参考回答**：

HTTP/1.1 的问题：**队头阻塞**
```
连接1: [Request A] ----等待----> [Response A] [Request B] → 串行
```
想并行？开多个连接（浏览器通常限制 6 个/域名）。

HTTP/2 的解法：**Stream 多路复用**
```
单个 TCP 连接:
  Stream 1: [HEADERS A] [DATA A-1] [DATA A-2] ...
  Stream 2: [HEADERS B] [DATA B-1] ...
  Stream 3: [HEADERS C] [DATA C-1] [DATA C-2] [DATA C-3] ...
```

- 一个 TCP 连接上可以同时跑多个 **Stream**
- 每个 Stream 有唯一 ID，Frame 按 Stream ID 区分
- Frame 级别交错传输 → 不会因为某个大响应阻塞其他请求

**对 gRPC 的意义**：
- 100 个 Agent 的心跳可以走**同一个 TCP 连接**的不同 Stream
- Server 端一个端口就能服务所有 Agent，不需要连接池
- 但实际部署中我们还是**每个 Agent 一个连接**，因为 Agent 分布在不同机器上

### 6.4 "分布式锁用 Redis 实现，说说 Redlock 算法？你为什么没用？"

**参考回答**：

**Redlock 核心思想**：
1. 准备 N 个（通常 5 个）独立的 Redis Master 实例（不是集群，是独立实例）
2. 客户端**依次**向 N 个实例请求加锁，每次设置相同的 Key 和 TTL
3. 如果在超时时间内，**大多数（N/2 + 1）**实例加锁成功 → 加锁成功
4. 如果失败 → 向所有实例发送 DEL 释放锁

**为什么没用**：

1. **Martin Kleppmann 的批判**（How to do distributed locking 那篇文章）：
   - Redlock 假设进程不会暂停太久（GC、网络延迟），但这个假设不成立
   - 即使用了 Redlock，如果持有锁的进程 GC 暂停后再操作共享资源，锁已经过期，还是会出问题
   - 正确做法：给锁带 **fencing token**（单调递增的 token），资源端检查 token 是否过期

2. **我们的场景不需要**：
   - 我们的分布式锁用于 Leader 选举，不是互斥资源保护
   - 即使锁失效导致双主，CAS + 幂等的 Layer 2-4 防御足够
   - 引入 5 个 Redis 实例的运维成本远大于收益

3. **如果真的需要强一致锁**：用 etcd/ZooKeeper 基于 Raft/ZAB 的 CP 锁，而不是 Redis 这种 AP 系统

### 6.5 "你提到了 WAL（Write-Ahead Log），能讲讲 WAL 的原理吗？它在你的 Agent 里怎么用的？"

**参考回答**：

**WAL 通用原理**：
- 核心思想：**先写日志，再改数据**
- 为什么：日志是顺序写（append-only），速度极快；数据修改是随机写，速度慢
- 如果崩溃：通过重放日志恢复未完成的数据修改
- 经典应用：MySQL 的 redo log、PostgreSQL 的 WAL、LevelDB 的 write-ahead log

**Agent 里的 WAL 设计**：

```
Agent 收到 Task → 写本地 WAL 文件 → 执行 Task → 成功后标记 WAL 条目为 committed
                                     ↓
                                  如果崩溃
                                     ↓
                              Agent 重启 → 读 WAL → 发现未 committed 的条目 → 重新执行/上报
```

具体实现：
1. WAL 文件格式：每行一个 JSON，包含 `{task_id, action_id, command, status, timestamp}`
2. 写入时 `fsync` 确保刷盘（不 fsync 的话 OS 缓存里的数据断电丢失）
3. Agent 重启时按顺序扫描 WAL：
   - status=pending → 检查进程是否还在跑，在跑就等，不在就重新执行
   - status=executing → 检查结果，上报 Server
   - status=committed → 跳过
4. WAL 定期压缩（truncate committed 条目）

## 七、业界对比追问 — "跟 K8s/Airflow 比呢？"

> **面试官心理**：看你是否关注业界方案，有没有**技术视野**。不是要你都用过，而是要知道别人怎么做的。

### 7.1 "跟 Kubernetes 的 Job/CronJob 比，你的系统有什么不同？"

**参考回答**：

| 维度 | 本系统 | K8s Job |
|------|--------|---------|
| 调度粒度 | 四层模型（Job→Stage→Task→Action） | 两层（Job→Pod） |
| 编排能力 | Stage 链表有序编排 | 无原生编排（需要 Argo Workflows 等） |
| 执行位置 | Agent 在**已有物理机**上执行 | Pod 运行在 K8s 集群内 |
| 调度目标 | 面向**大数据组件运维**（安装、升级、扩缩容） | 面向**容器化应用** |
| 失败处理 | CAS 状态机 + CleanerWorker 补偿 | backoffLimit + restartPolicy |
| 横向扩展 | 单主 + 分片（未来） | 原生分布式（kube-scheduler） |

**核心区别**：K8s 的 Job 是**面向容器的一次性任务**，我们是**面向物理机的多阶段运维编排**。K8s 做不了的：
1. 在已有物理机上执行 shell 命令（K8s Job 必须跑在容器里）
2. Stage 级的有序依赖编排（K8s 原生不支持，需要 Argo）
3. Action 级的细粒度进度追踪（K8s 只到 Pod 级别）

### 7.2 "跟 Apache Airflow 比呢？"

**参考回答**：

| 维度 | 本系统 | Airflow |
|------|--------|---------|
| 编排模型 | Stage 链表（线性有序） | DAG（有向无环图，可并行） |
| 任务类型 | 大数据组件运维命令 | 数据管道（ETL/ELT） |
| 执行方式 | gRPC 下发到 Agent | Executor（Local/Celery/K8s） |
| 调度器 | 自研 6-Worker 引擎 | Airflow Scheduler（单点） |
| 依赖管理 | NextStageId 链表 | 任务级依赖 + 触发规则 |
| UI | 无（API 为主） | 丰富的 Web UI |

**为什么不直接用 Airflow**：
1. Airflow 是 **数据工程**工具，设计目标是跑 ETL Pipeline，不是**基础设施运维**
2. Airflow 的 Executor 模型（Celery/K8s）不适合"登录到物理机执行 shell 命令"的场景
3. Airflow 的 DAG 定义用 Python，对非开发人员不友好——我们的用户是运维团队，需要通过 API/UI 配置任务
4. Airflow Scheduler 是**单点**，我们的 Leader 选举 + 热备至少有 HA

**如果用 Airflow 来做类似的事**：需要自定义 Operator（SSHOperator）+ 自定义 Hook + 自定义 Sensor，开发量不比自研少多少，但架构上多了 Airflow 这个大依赖。

### 7.3 "跟 Temporal / Cadence 比呢？"

**参考回答**：

Temporal 是目前**工作流编排**领域最强的开源方案。

| 维度 | 本系统 | Temporal |
|------|--------|----------|
| 编排模型 | 状态机 + 链表 | 工作流即代码（Workflow as Code） |
| 持久化 | MySQL + Kafka | 内置持久化（MySQL/PostgreSQL/Cassandra） |
| 重试/补偿 | CleanerWorker 扫描 | 原生 Activity Retry + Saga 补偿 |
| 信号/查询 | 心跳捎带 | Signal/Query API |
| 版本管理 | 无 | Workflow Versioning |
| 学习曲线 | 低（简单状态机） | 高（需要理解 Event Sourcing） |

**如果重新选型**，Temporal 是最佳选择：
1. 天然支持 Activity 级重试、超时、补偿
2. 不需要自己维护状态机——Temporal 的 Event Sourcing 自动持久化每一步
3. 分布式执行天然支持——Worker 可以水平扩展

**为什么当时没用**：
1. 2022 年 Temporal 在国内生态还不成熟，文档和社区都偏英文
2. 团队对 Go + MySQL + Kafka 更熟悉，自研可控性更高
3. Temporal 的学习曲线陡——Event Sourcing + Replay 的概念需要团队培训

> 诚实说"如果重来我会选 Temporal"是加分项，不是减分项——面试官看的是你的**技术视野和反思能力**。

### 7.4 "跟 Ansible / SaltStack 这类运维工具比呢？"

**参考回答**：

Ansible/SaltStack 是**配置管理和临时执行**工具，不是任务调度系统。

| 维度 | 本系统 | Ansible | SaltStack |
|------|--------|---------|-----------|
| 定位 | 长期运行的任务调度平台 | 临时执行 + 配置管理 | 远程执行 + 配置管理 |
| 持久化 | MySQL 记录全部状态 | 无持久化 | 无持久化（JID 记录有限） |
| 编排 | 四层模型 + 状态机 | Playbook（线性） | State/Orchestrate |
| 连接模式 | Agent 常驻 + 心跳 | SSH 临时连接 | Agent (minion) 常驻 |
| 进度追踪 | Action 级实时进度 | 只有最终结果 | 事件总线 |
| 断点续传 | 支持（WAL + 状态恢复） | 不支持 | 不支持 |

**核心区别**：
- Ansible 是"一次性做完"模型——执行完 Playbook 就结束了，没有状态持久化
- 我们是"持续管理"模型——Job 可以运行数小时，中间需要状态追踪、故障重试、断点续传
- 如果大数据组件升级执行到一半失败了，Ansible 只能整体重跑，我们可以从断点继续

## 八、反思改进追问 — "如果重来你会改什么？"

> **面试官心理**：这是考察你的**自我反思和成长能力**。没有完美的系统，能说出不足的人比只说优点的人更有深度。

### 8.1 "如果重新设计这个系统，架构上你会改什么？"

**参考回答**：

**改进 1：用 Event Sourcing 替代状态表**

当前：
```
stage 表: id | state | updated_at
```

改为：
```
stage_event 表: id | stage_id | event_type | payload | created_at
事件序列: StageCreated → StageActivated → TasksCreated → TaskCompleted → ... → StageCompleted
```

好处：
1. 天然的审计日志——每一步状态变更都有记录
2. 可以任意时刻回溯到历史状态
3. 不需要 CAS 乐观锁——事件是 append-only 的，不存在并发写冲突
4. 调试神器——出问题直接看事件序列就能定位

代价：
- 查"当前状态"需要聚合事件序列 → 引入物化视图
- 事件表数据增长快 → 需要定期归档
- 学习曲线陡

**改进 2：Stage 链表改为 DAG**

当前的 Stage 链表只支持线性编排（Stage 1 → 2 → 3 → 4），但实际场景中有些 Stage 可以并行：

```
          → Stage 2a (安装组件A) ─┐
Stage 1 →                          → Stage 3 (配置)
          → Stage 2b (安装组件B) ─┘
```

改为 DAG 后：
- Stage 表增加 `depends_on` 字段（JSON 数组存依赖的 Stage ID 列表）
- 所有依赖完成才激活当前 Stage
- 并行度提升 → Job 执行时间缩短

**改进 3：MemStore 改为 Redis 队列**

当前 MemStore 是进程内内存，Leader 切换时需要全量重建。改为 Redis 队列（List/Stream）：
- Leader 切换无缝——新 Leader 直接从 Redis 读队列
- Worker 变成无状态——可以水平扩展到多实例
- 代价：增加了 Redis 依赖（但 Redis 已经在用了）

### 8.2 "如果重新来，技术栈上你会改什么？"

**参考回答**：

1. **Temporal 替代自研状态机**（前面 7.3 已经详述）
2. **ClickHouse 替代 MySQL 做历史数据存储**——Action 表的 stdout/stderr 是分析型数据，写多读少，ClickHouse 的列存 + 压缩更适合
3. **OpenTelemetry 从第一天就接入**——而不是后来补加 tracing/metrics/logging，统一的可观测性标准从一开始就该有
4. **引入 API Gateway**——当前 Server 同时承担业务逻辑 + API 网关 + 连接管理，职责太重，应该用 Kong/APISIX 做前置网关

### 8.3 "代码层面，你觉得最大的技术债务是什么？"

**参考回答**：

**债务 1：TaskCenterWorker 的非原子状态检查**

```go
// 当前代码
tasks := db.Query("SELECT * FROM task WHERE stage_id = ?", stageID)
for _, task := range tasks {
    actions := db.Query("SELECT * FROM action WHERE task_id = ?", task.ID)
    // ... 聚合判断
}
// 问题：两次查询之间，task/action 状态可能已经变了
```

应该改成：
- 方案 A：包在事务里（但事务时间可能很长）
- 方案 B（推荐）：改为计数器方案，Stage 表维护 completed_count/total_count

**债务 2：Producer Registry 的 init() 注册**

```go
func init() {
    registry.Register("install_kafka", &KafkaInstallProducer{})
    registry.Register("config_topic", &TopicConfigProducer{})
    // ...
}
```

init() 函数的问题：
- 执行顺序不确定（Go 只保证同一包内的 init 顺序）
- 如果 init 内发生 panic，错误信息很难定位
- 应该改为**显式注册** + **依赖注入**（wire/fx）

**债务 3：错误处理太粗糙**

```go
if err != nil {
    log.Errorf("failed to process stage: %v", err)
    return
}
```

问题：
- 没有错误分类（临时错误 vs 永久错误）
- 没有结构化错误信息（缺少 context、stack trace）
- 应该引入自定义 error 类型 + `errors.Is()/As()` 做精确错误处理

### 8.4 "你觉得这个项目最值得骄傲的设计是什么？"

**参考回答**：

**"先 Kafka 后 DB" 的一致性策略 + 多层防御体系。**

这不是一个复杂的架构，但它是一个**充分考虑了失败场景的架构**：

```
Layer 1: Redis 分布式锁    → 正常情况下防止并发
Layer 2: Kafka 事件驱动    → 即使 DB 写入延迟也不丢链路
Layer 3: DB CAS 乐观锁    → 即使双主也不会数据混乱
Layer 4: CleanerWorker    → 即使 Kafka 挂了也能靠扫描推进
Layer 5: Agent WAL        → 即使 Agent 崩溃也不丢执行进度
```

每一层都在问"如果上一层失败了怎么办"，这是**防御性设计**的核心理念。面试时说"我最骄傲的不是任何单个组件，而是整个防御体系的完整性"，是一个很强的结束语。

## 九、压力面场景题 — 开放式设计挑战

> **面试官心理**：这类题没有标准答案，考的是你的**系统设计思维过程**——怎么分析需求、怎么做取舍、怎么分步骤实现。

### 9.1 "给你的系统加一个任务优先级功能，怎么设计？"

**思考框架**：

**需求拆解**：
- 高优先级 Job 的 Task 应该比低优先级的先执行
- 高优先级 Job 不应该被低优先级 Job 的大量 Task 饿死

**方案设计**：

```
Job 表增加 priority 字段 (1=LOW, 2=NORMAL, 3=HIGH, 4=URGENT)

MemStore 改造:
  当前: 单队列 []Stage
  改后: 多优先级队列 map[Priority][]Stage

DequeueStage 逻辑:
  1. 先看 URGENT 队列
  2. 再看 HIGH 队列
  3. 按权重比从 NORMAL/LOW 轮询（防饿死）
  
权重分配: URGENT:HIGH:NORMAL:LOW = 8:4:2:1
实现: 每 15 次出队，URGENT 出 8 个，HIGH 出 4 个...
```

**追问准备**：
- "优先级反转怎么办？"（高优先级依赖低优先级的结果）→ 依赖提升：被高优先级依赖的低优先级任务自动提升优先级
- "动态调整优先级？"→ API 支持运行时修改，MemStore 刷新时重新排序

### 9.2 "如果要支持任务的定时调度（CronJob），怎么加？"

**思考框架**：

**方案 1：数据库轮询**
```
cron_job 表: id | cron_expr | job_template | next_fire_time | status
CronWorker: 每秒扫描 next_fire_time <= now() 的记录，创建 Job，更新 next_fire_time
```
- 优点：简单可靠
- 缺点：扫描频率限制精度，大量 CronJob 时 DB 压力大

**方案 2：时间轮（Timing Wheel）**
```
内存时间轮: 60 个 slot（每秒一个），每个 slot 挂载到期的 CronJob
到期触发: 从 slot 取出 CronJob → 创建 Job → 计算下次触发时间 → 重新挂载
```
- 优点：O(1) 触发，无 DB 扫描开销
- 缺点：重启丢失（需要持久化 + 启动恢复）

**推荐**：方案 1 + 优化（索引 next_fire_time + 只扫描最近 1 分钟的记录）。规模大了再切方案 2。

### 9.3 "如果让你给这个系统加灰度发布能力，怎么设计？"

**思考框架**：

**灰度维度**：
- 按集群灰度：先升级测试集群 → 再升级生产集群
- 按比例灰度：先升级 10% 的节点 → 观察 → 再扩大到 50% → 100%
- 按标签灰度：只升级打了 `canary=true` 标签的节点

**实现方案**：

```
Job 增加灰度策略:
{
  "grayscale": {
    "strategy": "percentage",  // percentage | label | cluster
    "percentage": 10,          // 第一批 10%
    "pause_between_batches": true,  // 每批之间暂停等人工确认
    "rollback_on_failure": true,    // 任何一批失败自动回滚
    "health_check": {
      "type": "http",
      "url": "/health",
      "timeout": "30s",
      "success_threshold": 3
    }
  }
}
```

**Stage 拆解**：
```
Stage 1: 选择灰度目标节点（10% 随机选取）
Stage 2: 执行灰度批次（和正常部署一样，只是 Task 数量少）
Stage 3: 健康检查（调 Agent 执行 health check）
Stage 4: 人工确认 Gate（暂停，等 API 调用确认继续）
Stage 5: 扩大灰度比例（重复 Stage 2-4）
Stage 6: 全量发布
```

**关键点**：
- 回滚：灰度过程中任何阶段失败 → 触发回滚 Stage → 恢复到升级前版本
- 监控：灰度期间对比灰度组和对照组的指标（错误率、延迟、资源使用）

### 9.4 "你的 Agent 如果需要支持多租户隔离，怎么改？"

**思考框架**：

**隔离级别**：
- 资源隔离：不同租户的 Task 不抢资源
- 数据隔离：不同租户看不到彼此的 Job/Task
- 安全隔离：不同租户的命令不能访问彼此的文件系统

**Server 端改造**：
```
所有表增加 tenant_id 字段
所有查询增加 WHERE tenant_id = ?
Kafka Topic 按租户分：{tenant_id}_task_events
Redis Key 增加租户前缀：{tenant_id}:stage:queue
```

**Agent 端改造**：
```
方案 A: 每个租户独立 Agent 进程（最强隔离，运维成本高）
方案 B: 单 Agent + namespace 隔离（cgroup 限制 CPU/Memory）
方案 C: 单 Agent + 容器沙箱（每个 Task 跑在独立容器内）
```

推荐方案 C——Agent 负责拉起容器，Task 在容器内执行：
- 文件系统隔离（容器 rootfs）
- 资源隔离（cgroup limits）
- 网络隔离（network namespace）
- Agent 本身是轻量的"容器管理器"

### 9.5 "如果要做跨机房容灾（异地多活），你的系统怎么改？"

**思考框架**：

**CAP 决策**：任务调度系统选 **CP**（一致性 + 分区容忍），不选 AP。原因：任务不能重复执行（不是所有操作都幂等），一致性比可用性重要。

**架构方案**：

```
机房 A (主)                    机房 B (备)
┌───────────┐                ┌───────────┐
│  Server   │  ──MySQL 半同步──>│  Server   │
│  (Leader) │  ──Kafka MirrorMaker──>│  (Standby)│
│  Redis    │  ──Redis Sentinel──>│  Redis    │
└───────────┘                └───────────┘
     │                            │
   Agent A1                    Agent B1
   Agent A2                    Agent B2
```

**关键设计**：
1. **MySQL**：半同步复制（semi-sync），主挂了切到从，最多丢 1 个事务
2. **Kafka**：MirrorMaker 2.0 做跨机房复制，RPO ≈ 秒级
3. **Redis**：Sentinel 跨机房部署，自动 failover
4. **Server**：机房 B 的 Server 是 warm standby，平时不处理请求，通过心跳检测机房 A 是否存活
5. **Agent**：双注册——Agent 同时连接机房 A 和机房 B 的 Server，正常情况下只从主机房接收指令
6. **切换**：DNS 切换 + Server 角色切换，RTO 目标 < 1 分钟
