## 二、项目详述版（面试口述 / 项目详述页）

### 2.1 项目背景

大数据平台会管控监控很多大数据集群，需要对所有集群中的节点和组件（如 Hadoop/Spark/Kafka/Flink）的状态及服务上运行的任务进行采集上报监控，以便及时识别集群中服务或任务的异常情况。

**原始架构问题**：

| 问题 | 现象 | 根因 |
|------|------|------|
| **单点故障** | 采集服务挂掉后所有监控数据中断 | 简单 CS 架构，无高可用设计 |
| **扩展性差** | 新增组件类型需要大量改动 | 采集和上报功能耦合，无插件化机制 |
| **存储瓶颈** | 指标存储服务无法横向扩展 | Prometheus 单实例存储，无分片策略 |

### 2.3 核心设计与实现

#### 一、Agent 端

**（1）策略模式实现多类型采集**

使用策略模式实现 5 种类型指标采集，每种采集器实现统一的 `Collector` 接口，可快速扩展接入新组件：

```go
// 统一采集器接口
type Collector interface {
    Collect(ctx context.Context) ([]*Metric, error)
    Type() CollectorType
}

// 5 种采集器实现
// 1. JMX 采集器：解析 JMX MBean 数据（Hadoop/Kafka/HBase 等）
// 2. HTTP 采集器：调用组件 HTTP API 获取指标（Spark/Flink REST API）
// 3. CMD 采集器：执行命令行获取指标（系统级指标、进程状态等）
// 4. 日志采集器：解析组件日志文件提取关键指标
// 5. 自定义采集器：用户自定义脚本采集
```

**（2）心跳与采集配置机制**

- Agent 每 **5 秒**向 Server 上报一次心跳信息
- 心跳请求中同时携带"请求当前节点需要采集哪些组件信息"的查询
- Server 端对采集配置做了**多级缓存**：Server 内存缓存 5 秒，Redis 缓存 10 秒后才回源到 DB 查询
- 通过心跳通道实现低频的"配置同步"，Agent 无需重启即可动态调整采集项

**（3）可用性保障**

| 机制 | 说明 |
|------|------|
| **失败快速重试** | 采集失败后立即重试（最多 3 次），避免因瞬时故障丢失数据 |
| **心跳上报** | 每 5s 上报心跳，Server 端检测 Agent 存活状态 |
| **挂掉自动拉起** | 通过 systemd/supervisor 守护进程，Agent 异常退出后自动重启 |
| **本地缓存** | 网络中断时数据暂存本地，恢复后自动上报 |

#### 二、Server 端

**（1）gRPC 长连接 + Protobuf 压缩**

- Agent 与 Server 之间使用 **gRPC 长连接**，避免频繁建立/断开 TCP 连接的开销
- 数据通过 **Protobuf 二进制序列化**，相比 JSON 降低 **60%** 网络传输开销
- gRPC 支持流式传输，适合大批量指标数据的持续上报

**（2）高吞吐量上报设计**

```
Agent 数据 ──gRPC──→ Server 接收层 ──→ 内存缓冲区 ──→ 批量 HTTP 上报 ──→ VictoriaMetrics
                                         │
                                    连接池复用
                                    批量聚合（减少 HTTP 请求数）
```

- Server 向 VictoriaMetrics 上报数据通过**连接池**提高吞吐量
- 通过 **HTTP 批量上报**（将多个数据点聚合为一次 HTTP 请求）降低资源消耗
- **1C2G 单 Server 节点支持每秒处理 4 万+ 数据点**

**（3）无状态微服务架构**

- Server 为**无状态服务**，不在本地存储任何持久化数据
- 基于 **Kubernetes** 部署，支持 HPA（Horizontal Pod Autoscaler）动态扩缩容
- 受限于网络带宽和 VictoriaMetrics 写入速度，最多可扩容至 **50 个实例**
- 多 Server 实例之间通过负载均衡分摊 Agent 连接

#### 三、存储层优化

**（1）一致性哈希分片存储**

原始架构指标存储服务（Prometheus）无高可用和横向扩展能力。优化方案：

- 指标上报过程中通过**一致性哈希算法**将监控指标推送到多个异构存储服务
- 实现指标存储服务**高可用**和存储容量的**横向扩展**
- 存储层从 Prometheus 迁移到 **VictoriaMetrics**，获得更好的写入性能和压缩比

**（2）VictoriaMetrics 选型理由**

| 对比维度 | Prometheus | VictoriaMetrics |
|---------|-----------|-----------------|
| 写入性能 | 单实例有限 | 高吞吐，支持集群模式 |
| 存储压缩 | 一般 | 压缩比更高，节省 50%+ 存储 |
| 横向扩展 | 不支持原生集群 | 原生支持集群模式 |
| 兼容性 | — | 完全兼容 PromQL |
| 资源消耗 | 较高 | 更低的 CPU/内存消耗 |

### 2.4 核心难点与解决方案

#### 难点一：异构数据源采集适配

**问题**：Hadoop、Kafka、Spark 等组件指标暴露方式完全不同（JMX / HTTP API / 命令行 / 日志文件），需要统一对接方案。

**方案**：设计插件化采集框架，定义统一指标模型（MetricName / Label / Value），使用策略模式实现不同类型的采集器，支持热加载插件。新增组件监控支持只需实现 `Collector` 接口即可。

**成果**：新增组件监控支持的开发周期从 **2 周缩短到 2-3 天**，插件开发效率提升 60%。

#### 难点二：Agent 资源开销控制

**问题**：大数据集群对资源极度敏感，Agent 不能影响宿主节点上的大数据组件运行。

**方案**：
- 采用 Go 协程实现异步采集，避免阻塞
- 通过 cgroup 限制 Agent 的 CPU/内存使用上限
- 设计增量采集策略（仅抓取变化指标），减少不必要的计算
- gRPC + Protobuf 压缩降低 60% 网络带宽消耗

**成果**：Agent 资源消耗控制在 **CPU < 5%、内存 < 50MB**，生产环境零故障。

#### 难点三：高并发下的数据可靠性

**问题**：网络抖动可能导致 Agent 数据丢失，Server 端突发流量可能击穿存储。

**方案**：
- Agent 端实现**本地缓存**，网络中断时数据暂存本地，恢复后自动上报
- Agent 失败快速重试机制（最多 3 次）
- Server 端通过内存缓冲区 + 批量上报平滑写入压力
- 连接池复用避免连接风暴

**成果**：系统可用性达 **99.99%**，数据完整率 **>99.99%**。

#### 难点四：数据丢失的可接受性分析

**工程判断**：在监控数据采集场景中，并非所有数据丢失都需要零容忍：

| 场景 | 是否可接受 | 原因 |
|------|-----------|------|
| Agent 采集的监控指标偶尔丢失 1-2 个采集周期 | ✅ 可接受 | 监控指标是时序数据，下一个采集周期会补上，不影响趋势判断 |
| Agent 已将数据发送到 Server 且 Server 已返回 ACK，但 Server 还没推送到 VM 就挂掉 | ✅ 可接受 | 概率极低（Server 处理延迟通常 <100ms），且监控数据本身允许少量丢失 |
| 大面积网络中断导致长时间数据丢失 | ❌ 不可接受 | 因此设计了 Agent 本地缓存 + 恢复后自动上报机制 |

> 区分"可接受的丢失"和"不可接受的丢失"，对前者不做过度设计，对后者设计多层防护。

## 三、面试高频问题准备

### Q1：为什么选择 VictoriaMetrics 而不是 Prometheus？

> Prometheus 是单实例架构，不支持原生集群模式，在万级节点场景下存储容量和写入性能都是瓶颈。VictoriaMetrics 完全兼容 PromQL，但支持集群模式、写入性能更高、存储压缩比更好（节省 50%+ 存储空间），且 CPU/内存消耗更低。迁移成本很低，因为查询语法完全兼容。

### Q2：Agent 和 Server 之间为什么用 gRPC 而不是 HTTP？

> 三个原因：
> 1. **长连接复用**：gRPC 基于 HTTP/2，支持多路复用，避免频繁建立 TCP 连接的开销。万级 Agent 场景下，连接管理成本差异巨大。
> 2. **Protobuf 压缩**：相比 JSON，Protobuf 二进制序列化体积小 60%+，在大量指标数据传输场景下显著降低网络带宽消耗。
> 3. **流式传输**：gRPC 支持 Server/Client Streaming，适合 Agent 持续上报大批量指标数据的场景。

### Q3：一致性哈希在存储分片中是怎么用的？

> 原来所有指标都写入单个 Prometheus 实例，存储容量和写入性能都有上限。优化后通过一致性哈希算法，按指标名称（MetricName）哈希到不同的 VictoriaMetrics 节点：
>
> 1. **负载均衡**：指标均匀分布到多个存储节点
> 2. **横向扩展**：新增存储节点时，一致性哈希只需迁移少量指标（约 1/N）
> 3. **高可用**：某个存储节点宕机时，其负责的指标自动漂移到相邻节点（虚拟节点机制）
>
> 相比简单取模，一致性哈希在节点扩缩容时的数据迁移量从 100% 降低到约 1/N。

### Q4：策略模式在采集器设计中是怎么体现的？

> 定义统一的 `Collector` 接口（`Collect()` 方法返回标准化的 `[]*Metric`），每种采集类型实现这个接口：
>
> - **JMX 采集器**：连接 JMX 端口，解析 MBean 数据（Hadoop NameNode/DataNode、Kafka Broker 等）
> - **HTTP 采集器**：调用组件 REST API（Spark UI API、Flink REST API）
> - **CMD 采集器**：执行 Shell 命令获取系统级指标（CPU、内存、磁盘、网络）
> - **日志采集器**：解析组件日志文件，提取关键指标（错误率、GC 频率等）
> - **自定义采集器**：用户自定义脚本，输出标准格式即可接入
>
> Agent 启动时根据 Server 下发的采集配置，动态加载对应的采集器。新增组件只需实现 `Collector` 接口 + 注册到采集器工厂即可，无需修改框架代码。

### Q5：Server 是怎么做到无状态的？

> Server 不在本地存储任何持久化数据，所有状态都外置：
>
> - **采集配置**：存储在 MySQL，通过 Redis 多级缓存加速读取
> - **Agent 注册信息**：存储在 Redis，任何 Server 实例都可以查询
> - **指标数据**：接收后直接批量推送到 VictoriaMetrics，不在 Server 本地落盘
>
> 因此 Server 可以随时扩缩容，新实例启动后立即可以接收 Agent 数据。基于 Kubernetes HPA，当 CPU 使用率超过 70% 时自动扩容。
