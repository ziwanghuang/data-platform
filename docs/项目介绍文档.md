# Data-Platform（TBDS 管控平台）项目介绍

> **定位**：大规模分布式任务调度系统，管理 6000+ 节点，单次任务编排 32 万+ Action 指令
> **技术栈**：Go · MySQL · Redis · gRPC · Kafka · Prometheus
> **核心能力**：四层任务编排 → 性能深度优化 → 可靠性保障 → Saga 集群交付引擎

---

## 一、系统概览

### 1.1 业务场景

TBDS（Tencent Big Data Service）管控平台负责大数据集群的全生命周期管理：创建集群、扩缩容、配置下发、服务启停、巡检修复。核心挑战在于将用户的一个高层操作（如"安装 YARN 集群"）拆解为数千条 Shell 指令，下发到 6000+ 节点的 Agent 并发执行，并可靠地收集结果、驱动流转。

### 1.2 四层任务编排模型

```
Job（一个操作，如"安装 YARN 集群"）
 └── Stage（操作的一个阶段，如"下发配置"/"启动服务"）
      └── Task（阶段内可并行的子任务，如"配置 HDFS"/"配置 YARN"）
           └── Action（每个节点具体执行的 Shell 命令）
```

| 层级 | 职责 | 典型数量 | 执行方式 |
|------|------|---------|---------|
| **Job** | 完整操作流程 | 1 | API 触发 |
| **Stage** | 操作阶段 | ~16 | 阶段间顺序执行（优化后支持 DAG 并行） |
| **Task** | 阶段内子任务 | ~22 | 同 Stage 内并行 |
| **Action** | 节点级 Shell 命令 | 数千～数万 | 下发到 Agent 并发执行 |

**典型规模**：安装 YARN 组件 → 1 Job → 16 Stage → 22 Task → 107 Action。全量安装 6000 节点集群 → 约 321,000 Action。

### 1.3 整体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                     woodpecker-server                            │
│                                                                  │
│  ┌───────────┐  ┌─────────────────────────────────────────────┐ │
│  │ HTTP API  │  │        调度引擎（ProcessDispatcher）          │ │
│  │ - 创建Job │  │  6 Workers: memStoreRefresher / jobWorker   │ │
│  │ - 查询状态│  │  stageWorker / taskWorker / taskCenterWorker│ │
│  │ - 取消Job │  │  cleanerWorker                              │ │
│  └───────────┘  └─────────────────────────────────────────────┘ │
│                                                                  │
│  ┌───────────┐  ┌─────────────────────────────────────────────┐ │
│  │ gRPC 服务 │  │    RedisActionLoader（DB → Redis → Agent）    │ │
│  │ - 任务拉取│  │    定时 100ms 扫描 state=init 的 Action      │ │
│  │ - 结果上报│  │    ZADD 到 Redis Sorted Set                  │ │
│  └───────────┘  └─────────────────────────────────────────────┘ │
│                       │                │                         │
│                  ┌────▼────┐     ┌─────▼─────┐                  │
│                  │  MySQL  │     │   Redis   │                  │
│                  └─────────┘     └───────────┘                  │
│                                       │ gRPC                     │
│  ┌────────────────────────────────────▼─────────────────────┐   │
│  │              woodpecker-agent × 6000 节点                 │   │
│  │  CmdFetchTask(拉取) → WorkPool(50并发) → CmdReport(上报) │   │
│  │  HeartBeat(5s) · CmdMsg(内存队列)                         │   │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

**核心数据流**：  
API 创建 Job → DB 写入 → memStoreRefresher 定时刷新 → stageWorker/taskWorker 消费队列 → 生成 Action → RedisActionLoader 扫描 → Redis Sorted Set → Agent gRPC 拉取 → 执行 Shell → gRPC 上报 → DB 状态更新 → 进度检测 → 触发下一 Stage

## 二、痛点发现与性能优化之路

### 2.1 问题全景图

通过对线上运行数据和代码走读，梳理出 7 个核心问题，按优先级分为三档：

```
┌───────────────────────────────────────────────────────────┐
│                    Server 端问题                           │
│  [P0] 6 个 Worker 高频轮询 DB ──→ 事件驱动改造            │
│  [P0] Action 全表扫描 ──→ Stage 维度查询 + 覆盖索引       │
│  [P1] 逐条 UPDATE Action ──→ 批量聚合                    │
│  [P1] 单 Leader 瓶颈 ──→ 一致性哈希亲和性                │
│  [P2] Action 表无限增长 ──→ 冷热分离                     │
├───────────────────────────────────────────────────────────┤
│                    Agent 端问题                            │
│  [P0] 每 100ms 空轮询 ──→ UDP Push + 指数退避             │
│  [P1] 无本地去重 / 结果不持久化 ──→ WAL 刷盘              │
└───────────────────────────────────────────────────────────┘
```

### 2.2 P0 优化一：Kafka 事件驱动改造（替换 Worker 轮询）

**问题**：6 个 Worker 依赖"定时扫描 DB 发现变化"，空闲时仍产生 ~33 QPS，高峰时因 Agent 逐条上报导致 10 万+ QPS。

**根因**：没有事件通知机制，所有流转都靠定时发现。

**优化方案**：

```
优化前：Worker → 定时轮询 DB → 发现新数据 → 处理
优化后：DB 变更 → 产生 Kafka Event → Worker 消费 → 处理
```

- 引入 Kafka 作为内部事件总线
- Stage/Task/Action 状态变更产生事件（`stage.created` / `task.completed` 等）
- 6 个 Worker 改为 Kafka Consumer，按需消费
- **效果**：DB 轮询 QPS 降 80%+，空闲时趋近于零

### 2.3 P0 优化二：UDP Push + 指数退避（替换 Agent 空轮询）

**问题**：6000 Agent 每 100ms 轮询一次 Server，99% 返回空列表，产生 **6 万 QPS 空请求**。

**量化浪费**：
```
6000 节点 × 10次/s = 60,000 QPS
网络带宽：60,000 × ~200B/请求 = ~12 MB/s（仅空轮询）
Redis 空查询：60,000/s
```

**优化方案**：

```
优化前：Agent 每 100ms → gRPC 拉取 → 大概率返回空
优化后：
  1. Server 有新 Action 时 → UDP Push 通知对应 Agent
  2. Agent 收到通知 → 立即 gRPC 拉取（有数据）
  3. 无通知时 → 指数退避（100ms → 200ms → ... → 30s）
  4. UDP 不可靠 → 保底 30s 一次兜底轮询
```

**效果**：QPS 从 60,000 降至 ~1,500（降 97.5%），任务下发延迟从 100ms 降至 ~10ms。

### 2.4 P0 优化三：Stage 维度查询 + 覆盖索引（替换 Action 全表扫描）

**问题**：RedisActionLoader 每 100ms 执行 `SELECT id, hostuuid FROM action WHERE state=0 LIMIT 2000`，百万级表上即使有索引仍需回表 2000 次，耗时 ~200ms。

**优化方案**：

1. **Stage 维度查询**：不再扫全量 Action，只查当前活跃 Stage 下的 Action
2. **覆盖索引**：`CREATE INDEX idx_action_cover ON action(stage_id, state, id, hostuuid)` → 纯索引扫描，零回表
3. **哈希分片**：Action 按 `hostuuid` 哈希分为 N 个 Redis Key，避免单 Key 热点

```sql
-- 优化前
SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000;
-- Rows examined: ~50000, Extra: Using index condition（回表）

-- 优化后
SELECT id, hostuuid FROM action WHERE stage_id = ? AND state = 0 LIMIT 2000;
-- Rows examined: ~2000, Extra: Using index（覆盖索引，零回表）
```

**效果**：查询耗时从 ~200ms 降至 ~20ms，扫描行数降 96%。

### 2.5 P1 优化四：Action 状态批量聚合上报

**问题**：Agent 每完成一个 Action 就单独 UPDATE，突发场景 2000 节点 × 50 Action 同时完成 = 10 万次 UPDATE + 10 万次 ZREM。

**优化方案**：

```
优化前：每个 Action 完成 → 单独 UPDATE action SET state=3 WHERE id=?

优化后：
  1. Agent 侧：结果先写入本地 buffer，200ms 聚合一批上报
  2. Server 侧：接收到批量结果 → 单条 SQL 批量更新
     UPDATE action SET state=3 WHERE id IN (?,?,?,...?)
  3. Redis 侧：PIPELINE 批量 ZREM
```

**效果**：UPDATE 次数降 99%+，DB 行锁竞争大幅缓解。

### 2.6 P1 优化五：一致性哈希打破单 Leader 瓶颈

**问题**：3 个 Server 实例只有 1 个 Leader 工作，其余完全空闲。Leader 宕机需等 30s 锁过期。

**优化方案**：
- 按 `clusterId` 一致性哈希分配到不同 Server 实例
- 每个实例负责自己的 cluster 分片
- 新增/下线实例时仅需迁移少量 cluster

**效果**：Server 可线性水平扩展，故障切换从 30s 降至秒级。

### 2.7 P2 优化六：Action 表冷热分离

**问题**：Action 表月增 ~150 万条，年增 ~1800 万条，99% 是已完成的历史数据。

**优化方案**：定时任务将 `state=success/failed` 且 `updated_at < 30天前` 的 Action 迁移至 `action_archive` 表，热表保持 < 50 万条。

### 2.8 优化效果总结

| 优化项 | 优化前 | 优化后 | 提升幅度 |
|--------|--------|--------|---------|
| Server DB 轮询 QPS | 33~100,000+ | 按需消费，趋近 0 | **↓ 80%+** |
| Agent 空轮询 QPS | 60,000 | ~1,500 | **↓ 97.5%** |
| Action 查询耗时 | ~200ms | ~20ms | **↓ 90%** |
| Action UPDATE 次数 | 100,000/批 | ~1,000/批 | **↓ 99%** |
| Server 水平扩展 | 仅 1 Leader | 线性扩展 | **×N** |
| Action 热表大小 | 无限增长 | < 50 万条 | **可控** |

## 三、四大核心新增模块

### 3.1 模块一：事务幂等性改造

**解决的问题**：三个 Worker（jobWorker / stageWorker / taskWorker）存在裸写 DB 的问题，网络抖动或重启可能导致脏状态、重复数据、流程卡死。

**核心方案**：

| 机制 | 实现方式 | 覆盖场景 |
|------|---------|---------|
| **CAS 乐观锁** | `UPDATE SET state=new WHERE id=? AND state=old` | 状态流转防重 |
| **INSERT IGNORE** | `INSERT IGNORE INTO task (deterministic_id, ...)` | 创建防重 |
| **确定性 ID** | `md5(jobId + stageId + taskIndex)` | 全局去重标识 |

**三层保障**：

```
第一层（Server 端）：确定性 ID + INSERT IGNORE → 创建防重
第二层（Redis 层）：幂等投递 Token → 下发防重
第三层（Agent 端）：本地 finished_set → 执行防重
```

**面试价值**：直击分布式系统"至少一次"到"精确一次"的经典难题，代码量小但原理深。

### 3.2 模块二：Stage DAG 并行调度

**解决的问题**：Stage 之间使用 `next_stage_id` 串行链表，16 个 Stage 只能一个接一个执行，即使某些 Stage 之间没有依赖关系。

**核心方案**：

```
优化前（串行链表）：
  Stage1 → Stage2 → Stage3 → Stage4 → ... → Stage16
  总耗时 = sum(所有 Stage 耗时)

优化后（DAG 并行）：
  Stage1 ──→ Stage2 ──→ Stage5 ──→ Stage7
         ├── Stage3 ──┘          ├── Stage8
         └── Stage4 ──→ Stage6 ──┘
  总耗时 = 关键路径耗时
```

**技术实现**：
1. 数据模型：`next_stage_id` 改为 `depends_on JSON`（如 `[2, 3]`）
2. 调度算法：Kahn 拓扑排序 + 入度计数
3. 就绪检测：Stage 完成时，检查下游 Stage 所有依赖是否满足 → 入度归零则调度
4. 环检测：启动时 BFS 检测 DAG 有无环

```go
type Stage struct {
    ID        int64
    DependsOn []int64 `gorm:"type:json"` // DAG 依赖
    // ...
}

// Kahn 拓扑排序
func (s *StageScheduler) FindReadyStages(job *Job) []*Stage {
    inDegree := map[int64]int{}
    // ... 计算入度，返回入度为 0 的 Stage
}
```

**效果**：安装耗时预估降低 ~40%，面试中可展开讲拓扑排序、死锁检测、并发控制。

### 3.3 模块三：巡检修复 + Agent WAL 崩溃恢复

**解决的问题**：
1. Server 重启后无法区分"未下发"和"已下发未完成"的 Action → 重复下发
2. Agent 崩溃重启后内存 `finished_set` 丢失 → 重复执行
3. Agent 上报失败后结果丢失 → 任务永远卡在 executing

**核心方案**：

**Server 端三层去重**：
```
┌─────────────────────────────────────────┐
│ 第一层：确定性 ID（Server 创建时）        │ → 全局唯一，INSERT IGNORE 防重
│ 第二层：Redis 幂等 Token（下发时）       │ → SETNX 保证单次下发
│ 第三层：Agent 本地去重（接收时）          │ → 本地持久化 finished_set
└─────────────────────────────────────────┘
```

**Agent 端 WAL（Write-Ahead Log）**：
```
Action 接收 → WAL 写入（fsync）→ 执行 → 结果写入 WAL → 上报成功 → WAL 标记完成
                                                              │
Agent 崩溃重启 → 扫描 WAL → 恢复 finished_set + 重试未上报结果
```

**巡检修复**：Server 端定时扫描 `state=executing AND updated_at < 10分钟前` 的 Action，触发补偿重发或标记失败。

**面试价值**：覆盖"端到端 exactly-once"、WAL、崩溃恢复三个高频面试点。

### 3.4 模块四：Saga 集群交付引擎

**业务场景**：客户购买大数据集群服务 → 系统需要完成：创建订单 → 扣款 → 购买 CVM 机器 → 部署集群组件。任何一步失败都需要反向补偿（退款、释放 CVM），保证最终一致性。

**为什么选 Saga 而不是 2PC / TCC**：
- 2PC：CVM API 不支持 XA 协议，无法做两阶段提交
- TCC：CVM API 没有"Try（预留）"接口，只有直接购买/释放
- Saga：每一步独立执行 + 失败时反向补偿，天然适合长事务 + 外部 API 场景

**Saga 步骤编排**：

```
正向流程（Forward）：
  Step 1: 创建订单（OrderService）
  Step 2: 扣款（PaymentService）
  Step 3: 购买 CVM（TencentCloud CVM API）
  Step 4: 部署集群（调度引擎 Job）

补偿流程（Compensate，任一步骤失败时反向执行）：
  Step 4 失败 → 释放 CVM → 退款 → 取消订单
  Step 3 失败 → 退款 → 取消订单
  Step 2 失败 → 取消订单
```

**核心设计**：

```go
type SagaTransaction struct {
    ID              string        `gorm:"primaryKey"`
    State           SagaState     // RUNNING / COMPENSATING / COMPLETED / FAILED
    CurrentStep     int           // 当前执行到第几步
    Steps           []SagaStep    // 所有步骤定义
    IdempotentToken string        // 幂等令牌，防止重复创建
    CreatedAt       time.Time
    UpdatedAt       time.Time
}

type SagaStep struct {
    Index           int
    Name            string        // "create_order" / "charge" / "buy_cvm" / "deploy"
    State           StepState     // PENDING / RUNNING / SUCCESS / FAILED / COMPENSATED
    ForwardFunc     string        // 正向操作函数名
    CompensateFunc  string        // 补偿操作函数名
    Input           json.RawMessage
    Output          json.RawMessage
    RetryCount      int
    MaxRetries      int
}
```

**关键机制**：

| 机制 | 实现方式 |
|------|---------|
| **幂等令牌** | CVM 购买使用 `ClientToken`，网络超时重试不会重复购买 |
| **状态持久化** | 每一步执行前后都持久化到 DB，Server 崩溃后可恢复 |
| **崩溃恢复** | 定时扫描 `state=RUNNING AND updated_at < 5分钟前` 的 Saga，重新驱动 |
| **补偿重试** | 补偿操作失败 → 指数退避重试 3 次 → 仍失败则标记为 `NEED_HUMAN_INTERVENTION` |
| **人工兜底** | 所有自动补偿都失败后，告警通知运维人员手动处理 |

**完整生命周期**：

```
                         ┌──────────────┐
                         │  INITIALIZED │
                         └──────┬───────┘
                                │ start()
                         ┌──────▼───────┐
                    ┌────│   RUNNING    │────┐
                    │    └──────┬───────┘    │
               成功 │           │            │ 步骤失败
                    │     下一步│            │
                    │           │     ┌──────▼────────┐
                    │           │     │ COMPENSATING   │
                    │           │     └──────┬─────────┘
             ┌──────▼──────┐   │    补偿成功 │  补偿失败
             │  COMPLETED  │   │            │         │
             └─────────────┘   │     ┌──────▼───┐  ┌──▼──────────────┐
                               │     │COMPENSATED│  │NEED_HUMAN_INTER.│
                               │     └──────────┘  └─────────────────┘
                               │
                         (循环执行下一步)
```

**面试价值**：覆盖分布式事务（Saga vs 2PC vs TCC 选型）、外部 API 幂等、崩溃恢复、人工兜底四个高价值话题。

## 四、Prometheus 可观测性建设

**解决的问题**：原系统零可观测性，线上问题全靠日志 grep，无法实时感知系统健康度。

**指标体系**：

| 指标类别 | 具体指标 | 类型 |
|---------|---------|------|
| **任务执行** | `job_total{status}` / `stage_duration_seconds` / `action_execution_seconds` | Counter / Histogram |
| **调度引擎** | `worker_queue_length` / `worker_process_duration` | Gauge / Histogram |
| **Agent 通信** | `grpc_requests_total{method}` / `grpc_duration_seconds` | Counter / Histogram |
| **Redis** | `redis_action_set_size` / `redis_operation_duration` | Gauge / Histogram |
| **Saga** | `saga_total{state}` / `saga_step_duration` / `saga_compensation_total` | Counter / Histogram |

**接入方式**：
```go
import "github.com/prometheus/client_golang/prometheus"

var jobCompletedTotal = prometheus.NewCounterVec(
    prometheus.CounterOpts{
        Name: "woodpecker_job_completed_total",
        Help: "Total number of completed jobs",
    },
    []string{"status"}, // success / failed / cancelled
)
```

## 五、微服务拆分设计（演进方向）

基于数据流分析，将单体拆分为 **6 个微服务 + 1 个 Agent**：

```
┌─────────────────────────────────────────────────────────────┐
│                   微服务架构总览                               │
│                                                              │
│  ① ctrl-gateway      │ API 网关，请求路由 + 鉴权             │
│  ② ctrl-orchestrator  │ 流程编排，Job → Stage → Task → Action│
│  ③ ctrl-dispatcher    │ Action 下发，DB → Redis → 通知 Agent │
│  ④ ctrl-bridge        │ Agent 通信网关，gRPC 服务            │
│  ⑤ ctrl-tracker       │ 状态聚合，批量更新 + 进度检测        │
│  ⑥ ctrl-guardian      │ 运维守护，巡检 + 告警 + 冷热分离    │
│  ⑦ ctrl-agent         │ 节点 Agent，任务执行 + 结果上报      │
│                                                              │
│  通信：Kafka 消息总线（异步解耦）+ gRPC（同步调用）          │
└─────────────────────────────────────────────────────────────┘
```

**拆分收益**：
- 独立扩缩容：gRPC 网关（IO 密集）和流程编排（CPU 密集）分开扩容
- 故障隔离：Action 下发 OOM 不影响 API 可用性
- 独立发布：修改 API 不需要重启调度引擎
