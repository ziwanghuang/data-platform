## 一、拆分原则与方法论

### 1.1 为什么要拆分？

当前系统是一个**巨型单体**（woodpecker-server），承担了以下所有职责：

```
woodpecker-server（单体）
├── HTTP API（接收用户请求）
├── 流程编排（Job → Stage → Task → Action 的生成）
├── 调度引擎（6 个 Worker 驱动任务流转）
├── Action 下发（RedisActionLoader → Redis → Agent 拉取）
├── 结果收集（gRPC 接收 Agent 上报 → 更新 DB）
├── 进度检测（定时检查 Task/Stage/Job 完成度）
├── 心跳管理（接收 Agent 心跳 → 写入 Redis）
├── 分布式锁选举（Leader Election）
└── 清理任务（超时检测、失败重试、冷热分离）
```

**单体架构的核心问题**：

| 问题 | 说明 |
|------|------|
| **扩展性差** | 只有 Leader 执行调度，增加实例无法提升吞吐 |
| **故障域大** | 任何一个模块出问题（如 Action 下发 OOM），整个 Server 不可用 |
| **资源浪费** | 流程编排（CPU 密集）和 gRPC 服务（IO 密集）混在一起，无法独立扩缩容 |
| **发布耦合** | 修改 API 接口需要重新部署整个 Server，影响正在执行的调度任务 |
| **开发效率低** | 所有功能在一个代码库中，编译慢、测试慢、冲突多 |

### 1.2 拆分原则

| 原则 | 说明 | 应用 |
|------|------|------|
| **单一职责** | 每个服务只做一件事 | 流程编排 ≠ 任务下发 ≠ 结果收集 |
| **数据自治** | 每个服务拥有自己的数据存储 | Action 表归 Action 下发服务管理 |
| **按业务边界拆分** | 而非按技术层拆分 | 不是拆成"DB 层"+"缓存层"+"API 层" |
| **高内聚低耦合** | 服务内部高度相关，服务之间松散依赖 | 流程编排和 Action 下发通过 Kafka 解耦 |
| **独立部署** | 每个服务可以独立发布、扩缩容 | gRPC 网关可以独立扩容应对 Agent 高并发 |
| **渐进式拆分** | 不一步到位，分阶段演进 | 先拆最痛的点（Action 下发），再拆其他 |

### 1.3 拆分方法：基于数据流分析

通过分析系统的核心数据流，识别天然的服务边界：

```
用户请求 ──→ [API 网关] ──→ 创建 Job
                              │
                              ▼
                        [流程编排引擎] ──→ 生成 Stage → Task → Action
                              │
                              ▼ (Kafka)
                        [Action 下发服务] ──→ DB → Redis → 通知 Agent
                              │
                              ▼ (gRPC)
                        [Agent 通信网关] ←──→ Agent（拉取/上报/心跳）
                              │
                              ▼
                        [状态聚合服务] ──→ 批量更新 DB → 检测进度 → 触发下一 Stage
```

每个 `[...]` 就是一个天然的服务边界。

### 2.2 微服务命名规范

| 序号 | 服务名 | 全称 | 简称 | 端口 | 说明 |
|------|--------|------|------|------|------|
| ① | **ctrl-gateway** | Control Gateway | gateway | 8080 | 管控网关，用户请求入口 |
| ② | **ctrl-orchestrator** | Control Orchestrator | orchestrator | 8081 | 管控编排引擎，流程/任务生成 |
| ③ | **ctrl-dispatcher** | Control Dispatcher | dispatcher | 8082 | 管控指令下发，Action 投递 |
| ④ | **ctrl-bridge** | Control Bridge | bridge | 9090 | 管控通信桥接，Agent gRPC 网关 |
| ⑤ | **ctrl-tracker** | Control Tracker | tracker | 8083 | 管控状态追踪，进度检测与聚合 |
| ⑥ | **ctrl-guardian** | Control Guardian | guardian | 8084 | 管控守护服务（清理/补偿/归档） |
| ⑦ | **ctrl-agent** | Control Agent | agent | - | 管控节点执行器（每节点一个） |

> **命名规则**：`ctrl-` 前缀（Control 缩写，体现管控平台定位）+ 英文职责名词。简洁、语义明确、团队内无歧义。

### 2.3 拆分理由总览

| 服务 | 拆分理由 | 核心收益 |
|------|---------|---------|
| **ctrl-gateway** | API 层与业务逻辑解耦，可独立做鉴权/限流/灰度 | 独立扩缩容，不影响调度 |
| **ctrl-orchestrator** | 流程编排是 CPU 密集型（模板解析、Action 生成），与 IO 密集型的下发/上报分离 | 编排逻辑独立演进，可替换 Activiti |
| **ctrl-dispatcher** | Action 下发是系统最大的性能瓶颈（百万级 Action 写入），需要独立扩展 | 多实例并行分片写入，线性扩展 |
| **ctrl-bridge** | gRPC 服务承受 6 万 QPS（Agent 轮询），是 IO 密集型，需要独立扩容 | 独立扩容应对 Agent 高并发 |
| **ctrl-tracker** | 状态聚合和进度检测是独立的数据处理逻辑，与编排/下发无关 | 批量聚合优化，减少 DB 压力 |
| **ctrl-guardian** | 清理/补偿/归档是低优先级的后台任务，不应影响核心链路 | 故障隔离，不影响主流程 |

## 三、各微服务详细设计

### 3.1 ① ctrl-gateway — API 网关

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-gateway |
| **语言** | Go |
| **框架** | Gin |
| **端口** | 8080（HTTP） |
| **实例数** | 2~3（无状态，可水平扩展） |
| **资源特征** | IO 密集型（接收请求、转发） |

#### 职责

1. **接收用户 HTTP 请求**（创建 Job、查询状态、取消 Job）
2. **鉴权与权限校验**（Token 验证、操作权限检查）
3. **请求限流与熔断**（防止突发流量打垮后端服务）
4. **请求路由**（将请求转发到对应的后端服务）
5. **响应聚合**（查询 Job 状态时聚合多个服务的数据）

#### 对外接口

```
POST   /api/v1/jobs                  → 创建 Job（转发到 ctrl-orchestrator）
GET    /api/v1/jobs/{jobId}          → 查询 Job 状态（聚合 orchestrator + tracker）
POST   /api/v1/jobs/{jobId}/cancel   → 取消 Job（转发到 ctrl-orchestrator）
GET    /api/v1/jobs/{jobId}/stages   → 查询 Stage 列表（转发到 ctrl-tracker）
GET    /api/v1/jobs/{jobId}/actions  → 查询 Action 列表（转发到 ctrl-tracker）
GET    /api/v1/clusters/{clusterId}/hosts → 查询节点列表（转发到 ctrl-guardian）
GET    /health                       → 健康检查
```

#### 设计理由

> **为什么要单独拆出 API 网关？**
>
> 1. **发布解耦**：修改 API 接口（如新增查询参数）只需重新部署 gateway，不影响正在运行的调度任务
> 2. **安全边界**：鉴权、限流、审计日志等横切关注点集中在网关层，后端服务无需关心
> 3. **灰度发布**：可以在网关层做流量分配，将部分请求路由到新版本服务
> 4. **协议转换**：未来可以在网关层支持 gRPC-Web、GraphQL 等新协议，后端服务不需要改动

#### 依赖关系

```
ctrl-gateway → ctrl-orchestrator（创建/取消 Job）
ctrl-gateway → ctrl-tracker（查询状态/进度）
ctrl-gateway → ctrl-guardian（查询节点信息）
```

### 3.2 ② ctrl-orchestrator — 流程编排引擎

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-orchestrator |
| **语言** | Go |
| **框架** | Gin + Kafka Consumer |
| **端口** | 8081（HTTP，内部调用） |
| **实例数** | 2~3（消费者组，可水平扩展） |
| **资源特征** | CPU 密集型（模板解析、Action 生成） |

#### 职责

1. **Job 创建**：接收创建请求，根据 JobCode 查找流程模板，生成 Job + Stage 列表写入 DB
2. **Stage 编排**：消费 Kafka `stage_topic`，为 Stage 生成所有 Task
3. **Task 生成**：消费 Kafka `task_topic`，调用 TaskProducer 生成 Action 列表
4. **Action 投递**：将生成的 Action 批量投递到 Kafka `action_batch_topic`（分片）
5. **Job 取消**：标记 Job 为 cancelled，停止后续 Stage 推进
6. **流程模板管理**：维护 ProcessTemplate 和 TaskProducer 注册表

#### Kafka 消费 Topic

| Topic | 分区数 | 消费方式 | 说明 |
|-------|--------|---------|------|
| `job_topic` | 4 | Consumer Group | 消费新创建的 Job，生成 Stage |
| `stage_topic` | 8~16 | Consumer Group | 消费待处理的 Stage，生成 Task |
| `task_topic` | 16~32 | Consumer Group | 消费待处理的 Task，生成 Action |

#### Kafka 生产 Topic

| Topic | 分区数 | 分区策略 | 说明 |
|-------|--------|---------|------|
| `action_batch_topic` | 16~32 | IP 哈希分片 | Action 批量写入请求 |
| `stage_topic` | 8~16 | stage_id | 新生成的 Stage |

#### 核心数据流

```
创建 Job:
  gateway → HTTP → orchestrator
  → 查找流程模板
  → 写入 Job + Stage 到 DB
  → 投递第一个 Stage 到 Kafka[stage_topic]
  → 返回 jobId

Stage 编排:
  Kafka[stage_topic] → orchestrator
  → 根据 StageCode 查找 TaskProducer 列表
  → 生成 Task 写入 DB
  → 投递 Task 到 Kafka[task_topic]

Task 生成:
  Kafka[task_topic] → orchestrator
  → 调用 TaskProducer.Produce() 生成 Action 列表
  → 按 IP 哈希分片
  → 投递到 Kafka[action_batch_topic]
```

#### 设计理由

> **为什么流程编排要独立成服务？**
>
> 1. **核心业务逻辑隔离**：流程编排是整个系统的"大脑"，包含 56 种 BPMN 流程定义和数十个 TaskProducer。将其独立出来，可以专注于编排逻辑的演进（如从 Activiti 迁移到 Go 原生实现）
> 2. **CPU 密集型**：Action 生成涉及模板解析、节点过滤、命令拼装等 CPU 密集操作。6000 节点 × 50 Action/节点 = 30 万个 Action 的生成，需要独立的计算资源
> 3. **独立扩展**：当有大量并发 Job 时（如批量扩容），可以增加 orchestrator 实例来提升 Task/Action 的生成速度
> 4. **Kafka 消费者组**：多个 orchestrator 实例组成 Consumer Group，自动负载均衡消费 stage_topic 和 task_topic

#### 数据归属

| 表 | 读/写 | 说明 |
|----|-------|------|
| `job` | **读写** | Job 的创建和状态更新 |
| `stage` | **读写** | Stage 的创建和状态更新 |
| `task` | **写** | Task 的创建 |
| `host` | **读** | 查询节点信息用于 Action 生成 |
| `cluster` | **读** | 查询集群信息 |

### 3.3 ③ ctrl-dispatcher — Action 下发服务

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-dispatcher |
| **语言** | Go |
| **框架** | Kafka Consumer |
| **端口** | 8082（HTTP，健康检查/指标） |
| **实例数** | 3~5（核心服务，需要多实例并行） |
| **资源特征** | IO 密集型（DB 批量写入 + Redis 写入） |

#### 职责

1. **Action 批量写入 DB**：消费 Kafka `action_batch_topic`，批量 INSERT Action 到 MySQL
2. **Action 加载到 Redis**：将 DB 中 state=init 的 Action 写入 Redis Sorted Set
3. **通知 Agent**：Action 写入 Redis 后，通过 UDP 通知对应节点的 Agent 拉取
4. **Action 补偿加载**：定时检查 state=cached 但 Redis 中不存在的 Action，重新加载

#### Kafka 消费 Topic

| Topic | 分区数 | 消费方式 | 说明 |
|-------|--------|---------|------|
| `action_batch_topic` | 16~32 | Consumer Group（按 IP 分片） | 批量写入 Action |

#### 核心数据流

```
Kafka[action_batch_topic] → dispatcher
  → 批量 INSERT Action 到 DB（每批 200 条）
  → 更新 state: init(0)
  → Redis Pipeline: ZADD {hostuuid} {actionId}
  → 更新 state: cached(1)
  → UDP 通知 Agent: "有新任务"
```

#### 设计理由

> **为什么 Action 下发要独立成服务？**
>
> 1. **系统最大瓶颈**：Action 下发是整个系统的性能瓶颈。6000 节点 × 50 Action = 30 万个 Action 需要写入 DB 和 Redis。原来单 Server 串行写入需要分钟级，拆分后多个 dispatcher 实例并行写入可降至秒级
> 2. **分片并行**：Action 按节点 IP 哈希分片到 Kafka 不同分区，每个 dispatcher 实例消费不同分区，实现**无锁并行写入**
> 3. **独立扩缩容**：突发大量 Action 时（如 6000 节点扩容），可以临时增加 dispatcher 实例来加速写入
> 4. **故障隔离**：Action 写入 DB 出现慢查询或死锁时，不影响流程编排和 Agent 通信

#### 数据归属

| 表 | 读/写 | 说明 |
|----|-------|------|
| `action` | **读写** | Action 的批量创建和状态更新（init → cached） |

#### Redis 操作

| 操作 | Key | 说明 |
|------|-----|------|
| ZADD | `{hostuuid}` | 写入待执行的 Action ID |
| ZREM | `{hostuuid}` | 移除已完成的 Action ID（由 tracker 触发） |

#### 关键配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `action.batch.size` | 200 | 每批写入 DB 的 Action 数量 |
| `action.load.interval` | 100ms | 补偿加载间隔 |
| `udp.notify.enabled` | true | 是否启用 UDP 通知 |

### 3.4 ④ ctrl-bridge — Agent 通信网关

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-bridge |
| **语言** | Go |
| **框架** | gRPC Server |
| **端口** | 9090（gRPC） |
| **实例数** | 3~5（核心服务，承受 Agent 高并发） |
| **资源特征** | IO 密集型（网络 IO 为主） |

#### 职责

1. **任务拉取（CmdFetchChannel）**：接收 Agent 的 gRPC 请求，从 Redis 读取 Action 列表返回
2. **结果上报（CmdReportChannel）**：接收 Agent 的执行结果，投递到 Kafka `action_result_topic`
3. **心跳接收（HeartBeatV2）**：接收 Agent 心跳，写入 Redis Hash（TTL=30s）
4. **连接管理**：管理与 Agent 的 gRPC 连接，健康检查

#### gRPC 接口

```protobuf
service WoodpeckerCmdService {
    rpc CmdFetchChannel (CmdFetchChannelRequest) returns (CmdFetchChannelResponse);
    rpc CmdReportChannel (CmdReportChannelRequest) returns (CmdReportChannelResponse);
}

service WoodpeckerCoreService {
    rpc HeartBeatV2 (HeartBeatRequest) returns (HeartBeatResponse);
}
```

#### Kafka 生产 Topic

| Topic | 分区数 | 分区策略 | 说明 |
|-------|--------|---------|------|
| `action_result_topic` | 16 | hostuuid 哈希 | Agent 上报的 Action 执行结果 |

#### 核心数据流

```
任务拉取:
  Agent → gRPC → agent-gateway
  → Redis: ZRANGE {hostuuid} 0 -1（获取 Action ID 列表）
  → DB: SELECT * FROM action WHERE id IN (...)（获取 Action 详情）
  → 返回 ActionList

结果上报:
  Agent → gRPC → agent-gateway
  → 投递到 Kafka[action_result_topic]（异步处理，快速返回）
  → 返回成功

心跳:
  Agent → gRPC → agent-gateway
  → Redis: HSET heartbeat:{hostuuid} ... EX 30
  → 返回成功
```

#### 设计理由

> **为什么 Agent 通信要独立成服务？**
>
> 1. **最高并发压力**：Agent 通信网关承受 6000 节点 × 10 次/s = **6 万 QPS**，是整个系统并发最高的服务。独立出来可以专门针对高并发场景优化（如连接池、协程池、零拷贝）
> 2. **无状态设计**：Agent 通信网关是完全无状态的（从 Redis 读、往 Kafka 写），可以任意水平扩展
> 3. **快速响应**：结果上报不再同步写 DB（原来逐条 UPDATE），而是投递到 Kafka 后立即返回，大幅降低 Agent 上报延迟
> 4. **故障隔离**：即使 DB 出现慢查询，也不影响 Agent 的心跳上报和结果投递
> 5. **协议演进**：未来可以在 agent-gateway 层支持新的通信协议（如 QUIC），而不影响后端服务

#### 数据归属

| 存储 | 读/写 | 说明 |
|------|-------|------|
| Redis（Action 缓存） | **读** | ZRANGE 获取 Action ID 列表 |
| Redis（心跳） | **写** | HSET 写入心跳信息 |
| MySQL（action 表） | **读** | SELECT Action 详情（拉取时） |
| Kafka | **写** | 投递 Action 执行结果 |

#### 关键配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `grpc.max.connections` | 10000 | 最大 gRPC 连接数 |
| `grpc.worker.pool.size` | 200 | gRPC 工作协程池大小 |
| `agent.fetch.batch.size` | 20 | Agent 每次拉取的最大 Action 数 |

### 3.5 ⑤ ctrl-tracker — 状态追踪服务

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-tracker |
| **语言** | Go |
| **框架** | Kafka Consumer + Gin |
| **端口** | 8083（HTTP，查询接口） |
| **实例数** | 2~3 |
| **资源特征** | IO 密集型（DB 批量更新 + 进度计算） |

#### 职责

1. **Action 结果聚合**：消费 Kafka `action_result_topic`，在内存中聚合（200ms 时间窗口），批量更新 DB
2. **Task 进度检测**：消费 Kafka `task_status_detect_topic`，统计 Task 下所有 Action 的完成情况
3. **Stage 进度检测**：消费 Kafka `stage_status_detect_topic`，统计 Stage 下所有 Task 的完成情况
4. **Stage 推进**：当前 Stage 完成后，投递下一个 Stage 到 Kafka `stage_topic`
5. **Job 完成检测**：最后一个 Stage 完成后，标记 Job 为 success/failed
6. **状态查询**：提供 HTTP 接口供 gateway 查询 Job/Stage/Task/Action 的状态和进度

#### Kafka 消费 Topic

| Topic | 分区数 | 消费方式 | 说明 |
|-------|--------|---------|------|
| `action_result_topic` | 16 | Consumer Group | Agent 上报的 Action 结果 |
| `task_status_detect_topic` | 8 | Consumer Group | Task 进度检测事件 |
| `stage_status_detect_topic` | 4 | Consumer Group | Stage 进度检测事件 |

#### Kafka 生产 Topic

| Topic | 分区数 | 说明 |
|-------|--------|------|
| `stage_topic` | 8~16 | 触发下一个 Stage（投递给 orchestrator） |
| `task_status_detect_topic` | 8 | Action 完成后触发 Task 进度检测 |
| `stage_status_detect_topic` | 4 | Task 完成后触发 Stage 进度检测 |

#### 核心数据流

```
Action 结果聚合:
  Kafka[action_result_topic] → tracker
  → 内存聚合（200ms 窗口 / 100 条阈值）
  → 批量 UPDATE action SET state=?, exit_code=? WHERE id IN (...)
  → Redis: ZREM {hostuuid} {actionId}（批量移除）
  → 投递 task_status_detect 事件

Task 进度检测:
  Kafka[task_status_detect_topic] → tracker
  → SELECT COUNT(*) ... FROM action WHERE task_id=? GROUP BY state
  → 计算进度 = (success + failed) / total
  → 全部完成 → UPDATE task SET state=success
  → 投递 stage_status_detect 事件

Stage 进度检测:
  Kafka[stage_status_detect_topic] → tracker
  → SELECT state FROM task WHERE stage_id=?
  → 全部 Task 完成 → UPDATE stage SET state=success
  → 是否最后一个 Stage？
    → 是 → UPDATE job SET state=success
    → 否 → 投递下一个 Stage 到 Kafka[stage_topic]
```

#### 设计理由

> **为什么状态追踪要独立成服务？**
>
> 1. **批量聚合优化**：原来 Agent 上报结果后逐条 UPDATE DB（10 万次/批），拆分后 tracker 在内存中聚合（200ms 窗口），批量 UPDATE（数百次/批），DB 压力降低 **99%+**
> 2. **进度检测解耦**：进度检测是独立的数据处理逻辑（统计 Action/Task 完成度），与编排和下发无关。独立出来后可以专门优化统计查询
> 3. **事件驱动的 Stage 推进**：Stage 推进从"定时扫描"变为"事件触发"（Action 完成 → Task 完成 → Stage 完成 → 触发下一个 Stage），延迟从秒级降至毫秒级
> 4. **查询服务**：提供统一的状态查询接口，gateway 不需要直接查 DB

#### 数据归属

| 表 | 读/写 | 说明 |
|----|-------|------|
| `action` | **读写** | 批量更新 Action 状态，查询进度统计 |
| `task` | **读写** | 更新 Task 状态和进度 |
| `stage` | **读写** | 更新 Stage 状态，查询下一个 Stage |
| `job` | **读写** | 更新 Job 状态和进度 |

### 3.6 ⑥ ctrl-guardian — 运维服务

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-guardian |
| **语言** | Go |
| **框架** | Gin + 定时任务 |
| **端口** | 8084（HTTP） |
| **实例数** | 1~2（低频后台任务，单实例即可） |
| **资源特征** | 低资源消耗（定时任务为主） |

#### 职责

1. **超时检测**：定时扫描 state=executing 且超过 120s 的 Action，标记为 TimeOutFail
2. **失败重试**：扫描 state=failed 且 retryCount < retryLimit 的 Task，重置状态触发重新执行
3. **冷热分离**：定时将已完成的 Action 迁移到归档表（保留 7 天热数据）
4. **补偿扫描**：定时扫描"超时未完成"的 Stage/Task，重新投递到 Kafka（兜底机制）
5. **节点管理**：提供节点信息查询、离线检测等功能
6. **系统指标**：收集和暴露系统运行指标（Prometheus metrics）

#### 定时任务

| 任务 | 频率 | 说明 |
|------|------|------|
| 超时检测 | 每 10s | 扫描超时 Action |
| 失败重试 | 每 30s | 扫描可重试的 Task |
| 冷热分离 | 每天凌晨 2:00 | 迁移历史 Action 到归档表 |
| 补偿扫描 | 每 5~10s | 扫描超时未完成的 Stage/Task |
| 节点离线检测 | 每 30s | 检查心跳超时的节点 |

#### 设计理由

> **为什么运维任务要独立成服务？**
>
> 1. **故障隔离**：冷热分离涉及大量数据迁移（DELETE + INSERT），可能导致 DB 锁表。独立出来后即使 maintainer 出问题，也不影响核心的编排/下发/上报链路
> 2. **低优先级**：超时检测、失败重试、补偿扫描都是"兜底"机制，不在核心路径上。独立出来可以分配较少的资源
> 3. **单实例即可**：这些定时任务不需要多实例并行（通过分布式锁保证只有一个实例执行），资源消耗低
> 4. **运维友好**：运维相关的功能集中在一个服务中，便于监控和管理

#### 数据归属

| 表 | 读/写 | 说明 |
|----|-------|------|
| `action` | **读写** | 超时检测、状态更新 |
| `action_archive` | **写** | 冷热分离目标表 |
| `task` | **读写** | 失败重试 |
| `stage` | **读** | 补偿扫描 |
| `job` | **读** | 补偿扫描 |
| `host` | **读写** | 节点管理 |

### 3.7 ⑦ ctrl-agent — 节点执行器

#### 基本信息

| 属性 | 值 |
|------|---|
| **服务名** | ctrl-agent |
| **语言** | Go |
| **框架** | 自研模块管理 |
| **端口** | 无（主动连接 Server） |
| **实例数** | 每个节点一个 |
| **资源特征** | 混合型（网络 IO + Shell 执行） |

#### 职责

1. **任务拉取**：通过 gRPC 调用 ctrl-bridge 的 CmdFetchChannel
2. **命令执行**：WorkPool 并发执行 Shell 命令（最大 50 并发）
3. **结果上报**：通过 gRPC 调用 ctrl-bridge 的 CmdReportChannel
4. **心跳上报**：定时向 ctrl-bridge 上报节点状态
5. **本地去重**：维护 finished_set + executing_set，防止任务重复执行
6. **UDP 监听**：接收 ctrl-dispatcher 的 UDP 通知，触发立即拉取

#### 优化后的拉取策略

```
正常模式（无任务）：每 2s 定时拉取（兜底）
通知模式（收到 UDP）：立即拉取
高频模式（最近有任务）：每 500ms 拉取，连续 10 次无任务后退回正常模式
```

#### 设计理由

> Agent 本身已经是独立部署的，不需要拆分。优化重点在于：
> 1. 从高频轮询改为 UDP 通知 + 自适应轮询
> 2. 增加本地去重机制（内存 + 磁盘持久化）
> 3. 批量上报（50ms 窗口 / 6 条聚合）

## 四、服务间通信设计

### 4.1 通信方式总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                    服务间通信拓扑                                     │
│                                                                      │
│  ctrl-gateway ──HTTP──→ ctrl-orchestrator（创建/取消 Job）               │
│  ctrl-gateway ──HTTP──→ ctrl-tracker（查询状态）                         │
│  ctrl-gateway ──HTTP──→ ctrl-guardian（查询节点）                      │
│                                                                      │
│  ctrl-orchestrator ──Kafka──→ ctrl-dispatcher（Action 批量写入）         │
│  ctrl-orchestrator ──Kafka──→ ctrl-orchestrator（Stage/Task 消费）       │
│                                                                      │
│  ctrl-dispatcher ──UDP──→ ctrl-agent（通知有新任务）                     │
│                                                                      │
│  ctrl-agent ──gRPC──→ ctrl-bridge（拉取/上报/心跳）              │
│                                                                      │
│  ctrl-bridge ──Kafka──→ ctrl-tracker（Action 结果）               │
│                                                                      │
│  ctrl-tracker ──Kafka──→ ctrl-orchestrator（触发下一个 Stage）           │
│                                                                      │
│  ctrl-guardian ──Kafka──→ ctrl-orchestrator（补偿触发）                │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 通信方式选型

| 通信场景 | 方式 | 选型理由 |
|---------|------|---------|
| gateway → 后端服务 | **HTTP（同步）** | 用户请求需要同步响应，延迟要求 <500ms |
| orchestrator → dispatcher | **Kafka（异步）** | Action 生成是批量操作，异步解耦提升吞吐 |
| agent → agent-gateway | **gRPC（同步）** | 任务拉取需要实时响应，gRPC 高效序列化 |
| agent-gateway → tracker | **Kafka（异步）** | 结果上报不需要同步等待 DB 更新，异步提升响应速度 |
| tracker → orchestrator | **Kafka（异步）** | Stage 推进是事件驱动，天然适合消息队列 |
| dispatcher → agent | **UDP（单向通知）** | 仅通知"有新任务"，不需要可靠传输 |

### 4.3 Kafka Topic 全景

| Topic | 生产者 | 消费者 | 分区数 | 说明 |
|-------|--------|--------|--------|------|
| `job_topic` | orchestrator | orchestrator | 4 | 新创建的 Job |
| `stage_topic` | orchestrator / tracker | orchestrator | 8~16 | 待处理的 Stage |
| `task_topic` | orchestrator | orchestrator | 16~32 | 待处理的 Task |
| `action_batch_topic` | orchestrator | dispatcher | 16~32 | Action 批量写入请求 |
| `action_result_topic` | agent-gateway | tracker | 16 | Action 执行结果 |
| `task_status_detect_topic` | tracker | tracker | 8 | Task 进度检测事件 |
| `stage_status_detect_topic` | tracker | tracker | 4 | Stage 进度检测事件 |

## 五、数据归属与存储拆分

### 5.1 数据库拆分策略

采用**逻辑拆分**（同一个 MySQL 实例，不同的 Schema/权限），而非物理拆分：

| 服务 | 数据库 | 拥有的表 | 读写权限 |
|------|--------|---------|---------|
| **ctrl-orchestrator** | `ctrl_orchestrator` | job, stage, task, host, cluster | 读写 |
| **ctrl-dispatcher** | `ctrl_dispatcher` | action | 读写 |
| **ctrl-tracker** | `ctrl_tracker` | action（读写）, task/stage/job（读写进度） | 读写 |
| **ctrl-guardian** | `ctrl_guardian` | action_archive, host | 读写 |

> **注意**：在微服务初期，可以共享同一个数据库实例，通过**读写权限**控制数据归属。
> 随着规模增长，再逐步拆分为独立的数据库实例。

### 5.2 共享数据的处理

有些表需要被多个服务访问（如 `action` 表被 dispatcher、agent-gateway、tracker 访问）。处理方式：

| 方案 | 说明 | 适用阶段 |
|------|------|---------|
| **共享数据库** | 多个服务直接访问同一张表 | 初期（快速落地） |
| **数据同步** | 通过 Kafka 事件同步数据到各服务的本地表 | 中期（数据量增大） |
| **API 调用** | 通过服务间 API 查询数据 | 长期（完全解耦） |

**推荐策略**：初期共享数据库，中期通过 Kafka 事件同步关键数据，长期逐步迁移到 API 调用。

### 5.3 Redis 使用分配

| Key 模式 | 归属服务 | 读取服务 | 说明 |
|---------|---------|---------|------|
| `{hostuuid}` (Sorted Set) | ctrl-dispatcher（写） | ctrl-bridge（读）, ctrl-tracker（删） | Action 缓存 |
| `heartbeat:{hostuuid}` (Hash) | ctrl-bridge（写） | ctrl-guardian（读） | 心跳信息 |
| `ctrl:leader:{service}` (String) | 各服务自己 | 各服务自己 | 分布式锁 |

## 七、拆分前后对比

### 7.1 架构对比

```
拆分前（单体）：

│                                               │

│ (API)    │ │ (编排)       │ │ (下发)       │

• 每个服务独立扩缩容
• 故障隔离（dispatcher OOM 不影响 agent-gateway）
• 独立发布（修改 API 不影响调度）
• 按需分配资源
```

| 指标 | 拆分前 | 拆分后 | 提升 |
|------|--------|--------|------|

## 八、拆分路径与演进策略

### 8.1 渐进式拆分路线图

**不建议一步到位拆成 6 个服务**，而是分 3 个阶段渐进式拆分：

```
Phase 1（2~3 周）：拆出 Agent 通信网关 + Action 下发
├── 拆出 ctrl-bridge（最痛的点：6 万 QPS）
├── 拆出 ctrl-dispatcher（第二痛的点：百万级 Action 写入）
└── 其余功能暂时保留在 ctrl-server（过渡态）

Phase 2（2~3 周）：拆出流程编排 + 状态追踪
├── 拆出 ctrl-orchestrator（引入 Kafka 事件驱动）
├── 拆出 ctrl-tracker（引入批量聚合）
└── ctrl-server 退化为 ctrl-gateway + ctrl-guardian

Phase 3（1~2 周）：拆出 API 网关 + 运维服务
├── 拆出 ctrl-gateway
├── 拆出 ctrl-guardian
└── 废弃 ctrl-server
```

### 8.2 Phase 1 详细方案

```
当前状态：
  woodpecker-server（单体）
  woodpecker-agent

Phase 1 目标：
  woodpecker-server（保留编排 + 追踪 + 运维 + API）
  ctrl-bridge（新，独立 gRPC 服务）
  ctrl-dispatcher（新，独立 Action 下发）
  woodpecker-agent（改造，连接 ctrl-bridge）

改造步骤：
  1. 将 AgentCmdServiceModule 从 server 中抽出，部署为 ctrl-bridge
  2. 将 RedisActionLoader 从 server 中抽出，部署为 ctrl-dispatcher
  3. Agent 的 gRPC 连接地址改为 ctrl-bridge
  4. ctrl-bridge 的结果上报改为投递 Kafka（而非直接写 DB）
  5. server 消费 Kafka 更新 Action 状态（过渡方案）
```

### 8.3 回滚策略

每个 Phase 都保留回滚能力：

| Phase | 回滚方式 | 回滚时间 |
|-------|---------|---------|
| Phase 1 | Agent 连接地址切回 server | <5 分钟 |
| Phase 2 | 关闭 Kafka 消费者，启用 server 的 Worker | <10 分钟 |
| Phase 3 | 将 gateway 和 maintainer 合并回 server | <30 分钟 |

## 九、风险评估与应对

### 9.1 技术风险

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| **Kafka 成为单点** | 低 | 高 | Kafka 集群高可用（3 Broker + 副本因子 3） |
| **服务间调用延迟** | 中 | 中 | 核心链路使用 Kafka 异步解耦，非核心链路使用 HTTP |
| **数据一致性问题** | 中 | 高 | 先写 DB 后发 Kafka + 补偿扫描兜底 |
| **分布式事务** | 低 | 高 | 避免跨服务事务，使用最终一致性 + 补偿 |
| **服务发现故障** | 低 | 高 | etcd 集群高可用 + 本地缓存 |

### 9.2 组织风险

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| **团队规模不足** | 高 | 高 | 渐进式拆分，每个 Phase 由 1~2 人负责 |
| **运维能力不足** | 中 | 中 | 使用 K8s 统一编排，减少手动运维 |
| **调试困难** | 中 | 中 | 引入 Jaeger 分布式链路追踪 + 统一日志平台 |

### 9.3 不拆分的替代方案

如果团队规模或运维能力不足以支撑微服务，可以考虑**模块化单体**：

```
woodpecker-server（模块化单体）
├── module-gateway（API 模块）
├── module-orchestrator（编排模块）
├── module-dispatcher（下发模块）
├── module-agent-gateway（gRPC 模块）
├── module-tracker（追踪模块）
└── module-maintainer（运维模块）

优势：
• 保持单体部署的简单性
• 模块间通过接口解耦（而非网络调用）
• 未来可以按需拆分为独立服务

劣势：
• 无法独立扩缩容
• 故障域仍然是整个进程
```

> **建议**：如果团队 ≤3 人，先做模块化单体；团队 ≥5 人，可以直接做微服务拆分。
