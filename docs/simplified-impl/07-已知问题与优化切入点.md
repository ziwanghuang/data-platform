## 二、问题详细分析

### 2.1 [P0] 问题一：6 个 Worker 高频轮询 DB

**问题位置**：`02-Server端核心实现.md` → 第三章 6 个 Worker

**现象**：
```
memStoreRefresher  → 每 200ms 扫描 DB（全量加载 Job/Stage/Task 到内存）
jobWorker          → 每 1s 扫描 synced=0 的 Job
taskCenterWorker   → 每 1s 扫描 Activiti 活跃 Task
stageWorker        → 消费内存队列（依赖 memStoreRefresher 的全量加载）
taskWorker         → 消费内存队列（依赖 memStoreRefresher 的全量加载）
cleanerWorker      → 每 1s 扫描超时 Action
```

**根因分析**：
```
所有任务流转都依赖"定时扫描 DB 发现变化"，没有事件通知机制。
即使没有任何任务在执行，6 个 Worker 仍然在持续扫描 DB。
```

**量化影响**：
```
DB QPS 估算（无任务时）：
  memStoreRefresher: 5次/s × 4表 = 20 QPS
  jobWorker:         1次/s × 1查询 = 1 QPS
  taskCenterWorker:  1次/s × 1查询 = 1 QPS
  cleanerWorker:     1次/s × 1查询 = 1 QPS
  RedisActionLoader: 10次/s × 1查询 = 10 QPS
  合计（空闲时）：~33 QPS

DB QPS 估算（有任务时，2000 节点）：
  memStoreRefresher: 5次/s × 4表 × 大量数据 = 20 QPS（但每次扫描数据量大）
  RedisActionLoader: 10次/s × 全表扫描 = 10 QPS（但每次扫描百万行）
  Agent 上报:        10万次/批 × 逐条 UPDATE = 10万 QPS
  合计（高峰时）：10万+ QPS
```

**→ 对应优化方案**：`优化一：事件驱动改造`

---

### 2.2 [P0] 问题二：Agent 高频空轮询

**问题位置**：`03-Agent端核心实现.md` → 第三章 CmdFetchTask

**现象**：
```go
// Agent 每 100ms 拉取一次任务
func (h *CmdModule) fetchLoop() {
    for {
        h.doCmdFetchTask()
        time.Sleep(time.Duration(h.interval) * time.Millisecond) // 100ms
    }
}
```

**根因分析**：
```
Agent 无法感知 Server 是否有新任务，只能高频轮询。
99% 的请求返回空列表（没有新任务）。
```

**量化影响**：
```
6000 节点 × 10次/s = 6 万 QPS（Agent → Server）
每次请求：
  1. gRPC 调用 Server
  2. Server 查询 Redis（ZRANGE）
  3. Redis 返回空列表
  4. Server 返回空响应

网络带宽：6万 × ~200字节/请求 = ~12 MB/s（仅空轮询）
Redis QPS：6万/s（仅空查询）
```

**→ 对应优化方案**：`优化二：UDP Push + 指数退避 Pull`

### 2.3 [P0] 问题三：Action 全表扫描

**问题位置**：`04-gRPC通信与Redis下发.md` → 第三章 Redis Action 下发

**现象**：
```go
// RedisActionLoader 每 100ms 全表扫描
func (r *RedisActionLoader) loadActions() {
    // SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000
    actions, _ := r.clusterActionModule.GetWaitExecutionActions(2000)
    // ...
}
```

**根因分析**：
```
1. 索引 idx_action_state(state) 只包含 state，查询 hostuuid 需要回表
2. 百万级 Action 表中，state=0 的行可能分散在各个数据页
3. 即使 LIMIT 2000，MySQL 仍需扫描大量行来找到 state=0 的记录
```

**EXPLAIN 分析**：
```sql
EXPLAIN SELECT id, hostuuid FROM action WHERE state = 0 LIMIT 2000;

-- 结果：
-- type: ref
-- key: idx_action_state
-- rows: ~50000（估算扫描行数）
-- Extra: Using index condition  ← 需要回表！
```

**量化影响**：
```
每 100ms 执行一次全表扫描：
  - 扫描行数：~50000 行（即使 LIMIT 2000）
  - 回表次数：2000 次（每个匹配行都需要回表获取 hostuuid）
  - 耗时：~200ms（在百万级表上）
  - 锁竞争：与 Agent 上报的 UPDATE 竞争行锁
```

**→ 对应优化方案**：`优化三：Stage 维度查询 + 哈希分片 + 覆盖索引`

### 2.4 [P1] 问题四：无幂等性保障

**问题位置**：`03-Agent端核心实现.md` → 第五章 结果上报

**现象**：
```
1. Agent 重启后，内存中的 finished_set 丢失
2. Server 重新下发已完成的 Action
3. Agent 无法判断是否已执行过，重复执行
```

**场景复现**：
```
T=0s    Agent 执行 Action-1001，结果放入 resultQueue
T=0.2s  Agent 上报结果 → 网络超时
T=0.5s  Agent 进程被 kill（OOM 或运维操作）
T=5s    Agent 重启
T=5.1s  Agent 拉取任务 → Server 返回 Action-1001（因为上报失败，Server 认为未完成）
T=5.2s  Agent 再次执行 Action-1001 ← 重复执行！
```

**影响**：
```
- 幂等操作（如 systemctl start）：影响不大
- 非幂等操作（如 format namenode）：可能导致数据丢失！
- 长耗时操作（如安装软件包）：浪费时间和带宽
```

**→ 对应优化方案**：`优化四：分布式可靠性保障`

### 2.5 [P1] 问题五：逐条 UPDATE Action 状态

**问题位置**：`04-gRPC通信与Redis下发.md` → 第二章 ClusterActionModule

**现象**：
```go
// 逐条 UPDATE，每个 Action 一次 DB 操作
func (cam *ClusterActionModule) ReportAction2(uuid string, resultList []*ActionResultV2) error {
    for _, result := range resultList {
        cam.updateActionState(result)  // UPDATE action SET ... WHERE id = ?
        cam.redisClient.ZRem(uuid, fmt.Sprintf("%d", result.Id))
    }
    return nil
}
```

**量化影响**：
```
突发场景：2000 节点 × 50 Action 同时完成
  = 10 万个 Action 结果需要上报
  = 10 万次 UPDATE action SET state=3 WHERE id=?
  = 10 万次 ZREM

DB 压力：
  - 10 万次 UPDATE × ~1ms/次 = ~100 秒
  - 期间 Action 表被大量行锁锁定
  - 与 RedisActionLoader 的 SELECT 竞争
```

**→ 对应优化方案**：`优化五：Action 状态上报批量聚合`

---

### 2.6 [P1] 问题六：单 Leader 瓶颈

**问题位置**：`02-Server端核心实现.md` → 第二章 分布式锁

**现象**：
```go
// 只有获得分布式锁的 Server 才能执行 Worker
func (w *Worker) Run() {
    if !w.distributedLock.TryLock("woodpecker:server:leader", 30*time.Second) {
        return // 非 Leader，跳过
    }
    // 只有 Leader 执行
    w.doWork()
}
```

**影响**：
```
- 3 个 Server 实例，只有 1 个在工作
- 其他 2 个 Server 完全空闲
- 无法水平扩展处理能力
- Leader 宕机后需要等待锁过期（30s）才能切换
```

**→ 对应优化方案**：`优化七：Server 实例任务亲和性（一致性哈希）`

---

### 2.7 [P2] 问题七：Action 表无限增长

**问题位置**：`05-数据库模型与SQL.md` → 第七章 数据量估算

**现象**：
```
1 个月 Action 数据量：~150 万条
6 个月 Action 数据量：~900 万条
1 年 Action 数据量：~1800 万条

其中 99% 是已完成的历史数据，但仍然在同一张表中。
```

**影响**：
```
- 索引膨胀：B+ 树层级增加，查询变慢
- 全表扫描更慢：即使有索引，扫描范围也更大
- 备份/恢复时间增长
- DDL 操作（如加索引）耗时更长
```

**→ 对应优化方案**：`优化六：Action 表冷热分离`

## 四、优化实施路径

### 4.1 推荐实施顺序

```
Phase 1（基础改造，1-2 周）
├── 优化三：Stage 维度查询（改动最小，立即见效）
├── 优化五：批量聚合上报（改动小，效果显著）
└── 优化六：冷热分离（定时任务，独立实施）

Phase 2（核心改造，2-4 周）
├── 优化一：Kafka 事件驱动（核心改造，工作量最大）
└── 优化四：分布式可靠性保障（配合事件驱动一起实施）

Phase 3（通信优化，1-2 周）
├── 优化二：UDP Push + 指数退避（需要改 Agent 和 Server）
└── 优化七：一致性哈希亲和性（需要改 Server 选主逻辑）
```

### 4.2 依赖关系

```
优化三（Stage 维度查询）──→ 无依赖，可独立实施
优化五（批量聚合）──→ 无依赖，可独立实施
优化六（冷热分离）──→ 无依赖，可独立实施

优化一（事件驱动）──→ 需要 Kafka 集群就绪
优化四（可靠性保障）──→ 依赖优化一（Kafka 消费幂等）

优化二（UDP Push）──→ 依赖优化一（Server 知道何时有新 Action）
优化七（一致性哈希）──→ 依赖优化一（替换分布式锁选主）
```
