# TBDS 简化版管控平台 — 总体设计文档

> 本文档描述 TBDS 大数据管控平台的简化版本实现方案。
> 该简化版本保留了原系统的核心架构和关键机制，去除了业务复杂度（如 50+ 种 BPMN 流程、30+ 大数据组件集成代码），
> 聚焦于**任务调度引擎**本身的实现，作为后续架构优化的基础。

---

## 一、项目概述

### 1.1 系统定位

TBDS（Tencent Big Data Service）管控平台是一个**大规模分布式任务调度系统**，核心职责是：

1. 将用户的集群操作（创建集群、扩缩容、配置下发、服务启停等）**编排为具体的执行指令**
2. 将指令**下发到数千个节点的 Agent** 执行
3. **收集执行结果**，驱动任务流转直至完成

### 1.2 核心概念：四层任务编排模型

```
Job（一个操作，如"安装集群"）
 └── Stage（操作的一个阶段，Stage 之间顺序执行）
      └── Task（阶段内可并行的子任务）
           └── Action（每个节点具体执行的 Shell 命令）
```

**层级关系说明**：

| 层级 | 含义 | 数量级 | 执行方式 |
|------|------|--------|---------|
| **Job** | 一个完整的操作流程（如安装 YARN） | 1 | 由 API 触发 |
| **Stage** | 操作的一个阶段（如"下发配置"、"启动服务"） | ~16 | Stage 之间**顺序执行** |
| **Task** | 阶段内的子任务（如"配置 HDFS"、"配置 YARN"） | ~22 | 同一 Stage 内 Task **并行执行** |
| **Action** | 每个节点的具体命令（如 `systemctl start hadoop-yarn-resourcemanager`） | 数千 | 下发到各节点 Agent 执行 |

**典型数据规模**：安装 YARN 组件 → 1 Job → 16 Stage → 22 Task → 107 Action

### 1.3 简化版本的范围

| 保留 | 去除 |
|------|------|
| ✅ 四层任务编排模型（Job/Stage/Task/Action） | ❌ 50+ 种 BPMN 流程定义 |
| ✅ Server 端 6 个 Worker 轮询机制 | ❌ 30+ 大数据组件集成代码 |
| ✅ Agent 端任务拉取、执行、上报 | ❌ Activiti 工作流引擎（简化为直接调度） |
| ✅ Redis Action 缓存下发机制 | ❌ 配置管理、拓扑管理等业务功能 |
| ✅ gRPC Server-Agent 通信 | ❌ 多集群管理、权限控制 |
| ✅ 分布式锁选举机制 | ❌ 监控告警、日志采集 |
| ✅ 数据库模型和索引设计 | ❌ 前端 UI |

### 2.2 核心数据流

```
用户操作
    │
    ▼
HTTP API ──→ 创建 Job ──→ 写入 DB
    │
    ▼
memStoreRefresher ──定时──→ 扫描 DB 中未完成的 Stage/Task
    │                        │
    │                   写入内存队列
    │                        │
    ▼                        ▼
stageWorker ──消费──→ 生成 Task ──→ 写入 DB
    │
    ▼
taskWorker ──消费──→ 生成 Action ──→ 批量写入 DB
    │
    ▼
RedisActionLoader ──定时 100ms──→ 扫描 DB(state=init)
    │                               │
    │                          写入 Redis Sorted Set
    │                               │
    ▼                               ▼
Agent ──gRPC──→ Server ──→ 从 Redis 读取 Action 列表
    │                               │
    │                          返回 Action 给 Agent
    │
    ▼
Agent WorkPool ──并发执行──→ Shell 命令
    │
    ▼
Agent ──gRPC──→ Server ──→ 更新 DB(Action 状态)
    │
    ▼
定时任务 ──→ 检查 Task/Stage/Job 进度 ──→ 触发下一个 Stage
```

## 三、模块划分

### 3.1 Server 端模块

| 模块 | 职责 | 对应原系统 |
|------|------|-----------|
| **HTTP API Module** | 提供 RESTful API（创建 Job、查询状态、取消 Job） | `woodpecker-server/pkg/server/` |
| **ProcessDispatcher** | 调度引擎，包含 6 个 Worker | `woodpecker-server/pkg/serviceproduce/` |
| **MemStore** | 内存缓存（Job/Stage/Task 状态） | `woodpecker-server/pkg/serviceproduce/memstore/` |
| **RedisActionLoader** | 定时扫描 DB → 写入 Redis | `woodpecker-server/pkg/module/redis_action_loader.go` |
| **AgentCmdService** | gRPC 服务（任务拉取、结果上报） | `woodpecker-server/pkg/module/agentcmdservice_module.go` |
| **ClusterActionModule** | Action 的 CRUD 操作 | `woodpecker-server/pkg/module/cluster_action_module.go` |
| **LeaderElection** | 分布式锁选举（只有 Leader 执行调度） | `woodpecker-common/pkg/module/leader_election.go` |
| **GormModule** | 数据库连接管理 | `woodpecker-common/pkg/module/gorm_module.go` |
| **RedisModule** | Redis 连接管理 | `woodpecker-common/pkg/module/redis_module.go` |

### 3.2 Agent 端模块

| 模块 | 职责 | 对应原系统 |
|------|------|-----------|
| **CmdModule** | 定时拉取任务 + 定时上报结果 | `woodpecker-agent/pkg/module/cmd_module.go` |
| **WorkPool** | 并发执行 Action（协程池） | `woodpecker-bootstrap/pkg/module/cmd_processor.go` |
| **HeartBeatModule** | 定时心跳上报 | `woodpecker-agent/pkg/module/heartbeat_module.go` |
| **CmdMsg** | 内存消息队列（Action 队列 + Result 队列） | `woodpecker-common/pkg/models/cmd_msg.go` |

### 3.3 公共模块

| 模块 | 职责 |
|------|------|
| **Models** | 数据模型定义（Job/Stage/Task/Action） |
| **Proto** | gRPC 协议定义（CmdFetchChannel / CmdReportChannel / HeartBeat） |
| **Config** | 配置管理（INI 格式） |
| **Logger** | 日志模块 |
| **Module Manager** | 模块生命周期管理（Create → Start → Destroy） |

## 四、技术选型

| 类别 | 技术 | 选型理由 |
|------|------|---------|
| **语言** | Go | Server 和 Agent 均用 Go，高并发、低资源消耗 |
| **数据库** | MySQL 5.7+ | 业务数据持久化，GORM 作为 ORM |
| **缓存** | Redis 5.0+ | Action 任务缓存（Sorted Set），心跳信息存储 |
| **RPC** | gRPC | Server-Agent 通信，protobuf 序列化 |
| **HTTP** | Gin | RESTful API 框架 |
| **配置** | INI | 简单的配置文件格式 |
| **日志** | logrus | 结构化日志 |
| **定时任务** | 自研 ExecutorService | 毫秒级定时任务调度 |

```ini

host = 127.0.0.1
port = 3306

password = root
database = woodpecker

[redis]
host = 127.0.0.1

[action]
# RedisActionLoader 扫描间隔（毫秒）
cache.interval = 100
# 每次从 DB 加载的 Action 批次大小
loader.batch.size = 2000
# Agent 每次拉取的 Action 批次大小

[dispatcher]

# cleanerWorker 清理间隔（毫秒）

[election]
# 分布式锁 Redis Key

ttl = 30
# 续约间隔（秒）
renew.interval = 10
```

## 七、模块生命周期管理

Server 采用**模块化架构**，所有模块实现统一的 `Module` 接口：

```go
type Module interface {
    Create(config *Config) error  // 初始化（创建连接、读取配置）
    Start() error                 // 启动（开始定时任务、监听端口）
    Destroy() error               // 销毁（关闭连接、释放资源）
}
```

**启动顺序**（有依赖关系）：

```
1. LogModule          → 日志初始化
2. GormModule         → MySQL 连接
3. RedisModule        → Redis 连接
4. LeaderElection     → 分布式锁选举
5. GRpcServerModule   → gRPC 服务启动
6. HttpApiModule      → HTTP API 启动
7. ProcessDispatcher  → 调度引擎启动（6 个 Worker）
8. RedisActionLoader  → Action 加载器启动
```

**关键设计**：只有获得分布式锁的 Server 实例（Leader）才会执行调度逻辑（Worker 轮询、Action 加载），其他实例只提供 gRPC 和 HTTP 服务。

## 八、与架构优化文档的衔接

本简化版本实现的是**优化前的架构**，即：

| 当前实现（优化前） | 对应的优化方案 |
|-------------------|---------------|
| 6 个 Worker 定时轮询 DB | → 优化一：Kafka 事件驱动改造 |
| Agent 每 100ms 轮询 Server | → 优化二：UDP Push + 指数退避 Pull |
| RedisActionLoader 全表扫描 Action | → 优化三：Stage 维度查询 + 哈希分片 |
| Action 逐条 UPDATE 状态 | → 优化五：批量聚合上报 |
| Action 表无冷热分离 | → 优化六：冷热分离 |
| 单 Leader 处理所有集群 | → 优化七：一致性哈希任务亲和性 |
| 无 Agent 本地去重 | → 优化四：三层幂等性保障 |

> 详见 [tbds-architecture-optimization.md](../tbds-architecture-optimization.md)
