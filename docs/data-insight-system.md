## 二、简历展开版（面试口述 / 项目详述页）

### 2.1 项目背景与问题定义

**TBDS 大数据平台**管控着客户的上百个大数据集群，这些集群上运行着大量的 Spark/Hive/MapReduce 等计算任务。在实际生产环境中，用户经常遇到以下问题但难以自行定位：

| 问题类型 | 典型表现 | 影响 |
|---------|---------|------|
| **数据倾斜** | 某个 Task 处理的数据量远超其他 Task | 任务运行时间成倍增长，甚至 OOM |
| **内存浪费** | Executor 申请 8G 内存但实际只用 1G | 集群资源利用率低，其他任务排队等待 |
| **小文件问题** | HDFS 上产生大量小文件 | NameNode 内存压力大，集群性能下降 |
| **Shuffle 异常** | Shuffle 数据量异常大 | 网络 IO 成为瓶颈，任务缓慢 |
| **GC 频繁** | Executor 频繁 Full GC | 任务卡顿，吞吐量下降 |

**核心目标**：构建一套自动化的智能诊断系统，实时采集集群运行数据，自动检测上述性能问题并给出诊断建议，让用户无需深入了解大数据内部机制也能快速定位和解决问题。

### 2.3 核心模块详解

#### 模块一：数据采集层（Agent）

Agent 部署在每个大数据组件节点上，负责采集多种类型的运行数据：

| 采集类型 | 数据来源 | 采集方式 | 采集频率 |
|---------|---------|---------|---------|
| **JMX 指标** | Hadoop/Spark/Kafka 等组件暴露的 JMX 端口 | JMX 协议解析 | 15s/次 |
| **事件日志** | Spark EventLog、YARN 应用日志 | 日志文件解析 | 实时/准实时 |
| **HTTP 接口** | YARN ResourceManager API、Spark History Server API | HTTP 轮询 | 30s/次 |
| **FsImage 文件** | HDFS NameNode 的 FsImage 快照 | 定时拉取 | 1 次/天 |

**关键设计**：
- 使用**策略模式**实现不同类型的采集器，新增采集类型只需实现统一接口，无需修改核心逻辑
- Agent 通过 **gRPC 长连接**向 Server 上报数据，Protobuf 序列化降低网络开销
- 采集配置由 Server 下发，Agent 每 5 秒心跳时同步最新配置

#### 模块二：数据传输层（Server）

Server 作为数据中转站，接收 Agent 上报的数据并推送到 Kafka：

**关键设计**：
```
Agent 数据 → Server 接收 → 内存缓冲区 → 异步批量提交 → Kafka
                                              │
                                    按集群 ID 哈希分区
                                    保障同集群数据顺序性
```

- **异步批量提交**：Server 不逐条发送到 Kafka，而是在内存中缓冲，达到批量阈值或时间窗口后批量提交，提高吞吐量
- **分区策略**：按 `cluster_id` 哈希路由到 Kafka 分区，保证同一集群的数据在同一分区内有序，Flink 消费时无需额外排序
- **消息持久化**：Kafka `acks=all` + `min.insync.replicas=2`，确保消息不丢失

#### 模块三：Flink 实时诊断引擎（核心）

Flink 消费 Kafka 中的指标和事件数据，在数据落表的同时进行**旁路诊断**：

**（1）广播状态 — 诊断规则实时更新**

```
诊断规则配置（如数据倾斜阈值）
    │
    ▼
Kafka 规则 Topic ──→ Flink 广播流（BroadcastStream）
                          │
                          ▼
                    广播到所有算子实例
                          │
                          ▼
                    与数据流 connect
                          │
                          ▼
                    使用最新规则进行诊断
```

- 运维人员修改诊断规则后，规则通过 Kafka 广播到 Flink 所有算子实例
- 规则生效延迟 **<10 秒**，无需重启 Flink 任务
- 典型规则示例：`数据倾斜阈值 = 某 Task 数据量 > 平均值 × 5`

**（2）滚动窗口 — 任务资源分布统计**

```
Spark 任务指标流
    │
    ▼
KeyBy(task_id) → 滚动窗口(1min) → 聚合统计
                                      │
                                      ▼
                              计算各 Task 的数据量分布
                                      │
                                      ▼
                              判断是否存在数据倾斜
                              （最大值 / 平均值 > 阈值）
                                      │
                                      ▼
                              输出诊断结果 → Kudu
```

- 使用 **1 分钟滚动窗口**统计每个 Spark Stage 中各 Task 的数据处理量
- 计算数据分布的偏度（skewness），当最大 Task 数据量超过平均值的 N 倍时判定为数据倾斜
- 同时统计 Executor 的内存使用率，申请内存与实际使用差距过大时判定为内存浪费

**（3）旁路诊断架构**

```
Kafka 数据流
    │
    ├──→ 主路径：数据清洗 → 落表 Kudu（原始数据表）
    │
    └──→ 旁路：诊断算子 → 诊断结果 → 落表 Kudu（诊断结果表）
```

- **解耦设计**：数据存储和诊断逻辑分离，诊断算子故障不影响数据落表
- **可扩展**：新增诊断类型只需添加新的旁路算子，不影响主数据流

**（4）Checkpoint 与容错**

- 开启 Flink Checkpoint（间隔 60s），保障 **Exactly-Once** 处理语义
- Checkpoint 存储到 HDFS，故障恢复时间 **<10s**
- Kafka Consumer 的 Offset 与 Checkpoint 绑定，确保故障恢复后不重复消费

#### 模块四：Spark 离线诊断（FsImage 解析）

FsImage 是 HDFS NameNode 的元数据快照文件，包含整个文件系统的目录树和文件信息。通过解析 FsImage 可以诊断：

| 诊断项 | 检测逻辑 | 意义 |
|--------|---------|------|
| **小文件问题** | 统计文件大小 < 128MB 的文件数量和占比 | 小文件过多会导致 NameNode 内存压力 |
| **热点目录** | 统计各目录下的文件数量 Top N | 识别数据分布不均的目录 |
| **空目录清理** | 检测无文件的空目录 | 释放 NameNode 内存 |
| **大文件识别** | 统计超大文件（>10GB）的分布 | 优化存储策略 |

- FsImage 文件通常很大（GB 级），使用 **Spark 分布式解析**而非单机处理
- 每天定时拉取最新 FsImage，Spark 任务解析后将诊断结果写入 Kudu

### 2.4 核心难点与解决方案

#### 难点一：异构数据源的统一采集与处理

**问题**：不同组件的数据格式、采集方式、数据量差异巨大，需要统一的数据模型。

**解决方案**：
- 定义统一的**数据模型**：`{cluster_id, component, metric_name, labels, value, timestamp, event_type}`
- Agent 端使用**策略模式**，每种采集类型实现统一的 `Collector` 接口
- Server 端对数据进行**标准化处理**后再推送到 Kafka，Flink 消费时无需关心数据来源

#### 难点二：诊断规则的动态更新

**问题**：不同集群的诊断阈值不同（如小集群数据倾斜阈值 3x，大集群 5x），且需要运维人员能实时调整。

**解决方案**：
- 使用 Flink **广播状态（Broadcast State）**：规则变更通过 Kafka 广播到所有算子实例
- 规则支持**集群级别覆盖**：全局默认规则 + 集群维度自定义规则
- 规则生效延迟 <10 秒，无需重启 Flink 任务

#### 难点三：数据倾斜检测的准确性

**问题**：简单的"最大值/平均值"判断容易产生误报（如只有 2 个 Task 时天然差异大）。

**解决方案**：
- 引入**统计学指标**：结合标准差、变异系数（CV）等多维度判断
- 设置**最小样本量**：Task 数量 < 10 时不触发数据倾斜诊断
- 结合**历史基线**：与该任务历史运行数据对比，排除正常波动

#### 难点四：Flink 任务的稳定性与性能

**问题**：Flink 任务需要 7×24 小时运行，处理高吞吐数据流的同时进行复杂诊断计算。

**解决方案**：
- **反压处理**：监控 Flink 算子的反压指标，当下游算子处理不过来时自动降低消费速率
- **状态管理**：使用 RocksDB StateBackend，避免大状态导致 OOM
- **Checkpoint 优化**：增量 Checkpoint + 异步快照，减少 Checkpoint 对吞吐量的影响
- **资源隔离**：诊断算子与数据落表算子使用不同的 Slot Sharing Group，互不影响

#### 难点五：FsImage 大文件解析性能

**问题**：FsImage 文件可达 GB 级，单机解析耗时过长。

**解决方案**：
- 使用 Spark 分布式解析，将 FsImage 转换为 XML 格式后按目录分片并行处理
- 解析结果写入 Kudu，支持后续的多维度查询和分析
- 设置合理的 Spark 资源配额，避免诊断任务抢占业务任务资源

## 三、面试高频问题准备

### Q1：这个数据洞察系统的整体架构是怎样的？

> 整体是一个 **Agent → Server → Kafka → Flink/Spark → Kudu** 的数据管道架构。
>
> Agent 部署在每个大数据组件节点上，采集 JMX 指标、事件日志、HTTP 接口数据等，通过 gRPC 上报到 Server。Server 将数据异步批量提交到 Kafka，按集群 ID 分区保证顺序性。
>
> 下游有两类消费者：
> 1. **Flink 实时诊断**：消费指标和事件数据，在数据落表 Kudu 的同时进行旁路诊断，实时检测数据倾斜、内存浪费等问题
> 2. **Spark 离线诊断**：定时解析 FsImage 文件，诊断 HDFS 小文件、热点目录等问题
>
> 诊断结果统一写入 Kudu 的诊断结果表，前端展示给用户。

### Q2：为什么用 Flink 广播状态来更新诊断规则？

> 诊断规则需要满足两个需求：
> 1. **实时生效**：运维人员修改阈值后希望立即生效，不能重启 Flink 任务
> 2. **全局一致**：所有 Flink 算子实例必须使用相同的规则版本
>
> Flink 的广播状态（Broadcast State）天然满足这两个需求：规则变更发送到 Kafka 的规则 Topic，Flink 将其作为广播流与数据流 connect，规则自动广播到所有并行实例。
>
> 我也考虑过其他方案：
> - **定时拉取配置中心**：延迟不可控，且需要额外的轮询线程
> - **Flink 参数重启更新**：需要停止任务，影响数据处理连续性
> - **外部缓存（Redis）**：每条数据都查 Redis 性能开销大
>
> 广播状态是最优方案，规则存储在 Flink 算子的本地状态中，查询零网络开销。

### Q3：什么是旁路诊断？为什么这样设计？

> 旁路诊断是指在数据落表的主路径之外，分出一条独立的诊断路径。
>
> ```
> 数据流 → 主路径：清洗 → 落表 Kudu
>        → 旁路：诊断算子 → 诊断结果 → 落表 Kudu
> ```
>
> 这样设计的好处是**解耦**：
> 1. 诊断算子出 bug 或性能问题，不影响数据落表（核心功能）
> 2. 新增诊断类型只需添加新的旁路算子，不修改主数据流
> 3. 可以对诊断算子和数据落表算子分配不同的资源（Slot Sharing Group）
>
> 如果把诊断逻辑放在主路径中，一旦诊断算子反压，会导致数据落表也变慢，影响整个系统。

### Q4：数据倾斜检测的具体逻辑是什么？怎么避免误报？

> 数据倾斜检测分三步：
>
> 1. **数据收集**：通过 1 分钟滚动窗口，收集同一个 Spark Stage 中所有 Task 的数据处理量
> 2. **统计计算**：计算平均值、最大值、标准差、变异系数（CV = 标准差/平均值）
> 3. **规则判断**：当最大 Task 数据量 > 平均值 × N（默认 5 倍），且 CV > 阈值时，判定为数据倾斜
>
> 避免误报的措施：
> - **最小样本量**：Task 数量 < 10 时不触发诊断，因为少量 Task 天然差异大
> - **多维度判断**：不只看最大值/平均值，还结合标准差和变异系数
> - **历史基线对比**：与该任务历史运行数据对比，排除正常波动
> - **集群级别阈值**：不同规模的集群使用不同的阈值，通过广播状态动态配置

### Q5：Flink Checkpoint 是怎么保障 Exactly-Once 的？

> Exactly-Once 的保障依赖 Flink 的 **两阶段提交（2PC）** 机制：
>
> 1. **Checkpoint 触发**：JobManager 定期（60s）向所有 Source 算子发送 Checkpoint Barrier
> 2. **Barrier 对齐**：Barrier 随数据流向下游传播，每个算子收到所有输入的 Barrier 后，将当前状态快照到 HDFS
> 3. **Kafka Offset 绑定**：Kafka Consumer 的 Offset 作为 Source 算子状态的一部分，与 Checkpoint 一起持久化
> 4. **故障恢复**：从最近一次成功的 Checkpoint 恢复状态，Kafka Consumer 从对应的 Offset 重新消费
>
> 这样即使 Flink 任务故障重启，也不会重复处理或遗漏数据。故障恢复时间通常 <10s。
>
> 需要注意的是，Exactly-Once 的前提是 Sink 端也支持事务或幂等写入。我们使用 Kudu 作为 Sink，通过主键去重实现幂等写入。

### Q6：为什么 FsImage 解析用 Spark 而不是 Flink？

> FsImage 解析是一个**批处理**场景，不是流处理：
>
> 1. **数据特征**：FsImage 是一个完整的快照文件（GB 级），每天只生成一次，不是持续的数据流
> 2. **处理逻辑**：需要对整个文件系统目录树进行全量统计分析，适合批处理的 MapReduce 模型
> 3. **资源效率**：Spark 批处理任务用完即释放资源，Flink 流任务需要常驻资源
>
> 所以 FsImage 解析用 Spark 批处理更合适，而实时指标和事件的诊断用 Flink 流处理。两者各司其职。
