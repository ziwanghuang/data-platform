## 二、简历展开版（面试口述 / 项目详述）

### 2.1 项目背景与问题定义

随着大数据平台微服务化演进，平台内部拆分出 **50+ 微服务**（管控服务、监控服务、数据洞察服务、配置服务等），面临以下问题：

| 问题 | 现象 | 影响 |
|------|------|------|
| **入口分散** | 每个微服务独立暴露 API，客户端需维护多个地址 | 前端对接成本高，服务地址变更需全量通知 |
| **安全防护缺失** | 各服务自行实现认证鉴权，标准不统一 | 安全漏洞多，鉴权逻辑重复开发 |
| **流量不可控** | 无统一限流、熔断机制，单个服务被打爆影响全局 | 级联故障风险高 |
| **发布风险高** | 服务更新需全量切流，无灰度能力 | 新版本 Bug 影响所有用户 |

因此需要一个**统一的 API 网关**作为所有外部请求的唯一入口，集中处理路由、安全、流量治理等横切关注点。

### 2.3 核心模块设计

#### 模块一：高性能通信架构（Netty Reactor）

**设计方案**：

基于 Netty 的**主从 Reactor 多线程模型**：

```
┌─────────────────────────────────────────────────┐
│                 Netty Server                      │
│                                                   │
│  Boss Group (1 线程)                              │
│  └── 负责接收新连接（Accept）                      │
│       │                                           │
│       ▼                                           │
│  Worker Group (CPU核数 × 2 线程)                   │
│  └── 负责 IO 读写 + 业务处理                       │
│       │                                           │
│       ▼                                           │
│  Channel Pipeline                                 │
│  ├── HttpServerCodec (HTTP 编解码)                 │
│  ├── HttpObjectAggregator (请求聚合)               │
│  ├── IdleStateHandler (空闲检测)                   │
│  └── GatewayHandler (网关核心处理器)               │
│       └── 调用 Plugin Chain 执行插件链              │
└─────────────────────────────────────────────────┘
```

**关键设计点**：
- **Boss Group** 只负责 Accept 新连接，不处理任何业务逻辑，避免连接接入被阻塞
- **Worker Group** 线程数设为 `CPU核数 × 2`，充分利用多核性能
- **IdleStateHandler** 检测空闲连接（读空闲 60s / 写空闲 30s），自动关闭僵尸连接，防止连接泄漏
- 所有 IO 操作均为**非阻塞**，单机可支撑 **2 万+ 并发长连接**

**性能数据**：
| 指标 | 数值 |
|------|------|
| 单机 QPS | **3.2 万+** |
| 单机长连接数 | **2 万+** |
| 平均响应延迟 | **<5ms**（网关自身开销） |
| P99 延迟 | **<15ms** |

#### 模块二：可扩展插件机制（责任链模式）

**设计方案**：

基于**责任链设计模式**，将请求处理流程拆分为三个阶段：

```
请求进入

    ▼
┌─────────────────────────────────┐
│         Pre-Route 阶段           │
│  ┌───────────┐  ┌───────────┐  │
│  │ IP 限流    │→│ 黑白名单   │→ │
│  └───────────┘  └───────────┘  │
│  ┌───────────┐  ┌───────────┐  │
│  │ JWT 认证   │→│ 请求日志   │  │
│  └───────────┘  └───────────┘  │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│          Route 阶段              │
│  ┌───────────┐  ┌───────────┐  │
│  │ 路由匹配   │→│ 负载均衡   │→ │
│  └───────────┘  └───────────┘  │
│  ┌───────────┐                  │
│  │ 灰度路由   │                  │
│  └───────────┘                  │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│        Post-Route 阶段           │
│  ┌───────────┐  ┌───────────┐  │
│  │ 响应改写   │→│ 指标采集   │  │
│  └───────────┘  └───────────┘  │
└─────────────────────────────────┘
               │
               ▼
          返回响应
```

**核心接口设计**：

```java
public interface GatewayPlugin {
    /** 插件名称 */
    String name();
    /** 执行阶段：PRE_ROUTE / ROUTE / POST_ROUTE */
    PluginPhase phase();
    /** 执行优先级（数值越小越先执行） */
    int order();
    /** 插件执行逻辑，返回 true 继续执行下一个插件，返回 false 中断链路 */
    boolean execute(GatewayContext context);
}
```

**动态加载机制**：
- 插件通过 **SPI（Service Provider Interface）** 机制注册
- 支持运行时**热加载**新插件，无需重启网关
- 每个插件独立配置开关，可通过管理 API 动态启用/禁用

#### 模块三：服务注册发现与负载均衡

**服务注册发现（Etcd）**：

```
┌──────────┐  注册   ┌──────┐  Watch   ┌──────────┐
│ Service A │──────→ │ Etcd │ ←──────  │ Gateway  │
│ Service B │──────→ │      │ ←──────  │          │
│ Service C │──────→ │      │          │ 本地缓存  │
└──────────┘        └──────┘          └──────────┘
```

**关键设计**：
- 服务启动时向 Etcd 注册（Key: `/services/{service_name}/{instance_id}`，Value: `ip:port`）
- 注册信息带 **TTL（10s）**，服务需定时续约（每 3s 一次），停止续约则自动摘除
- 网关通过 **Etcd Watch** 机制实时感知服务实例变化，更新本地路由表
- 故障节点剔除延迟 **<7 秒**（TTL 过期时间）

**负载均衡算法 — 加权轮询 + 响应时间自适应**：

```
基础权重（配置）+ 动态权重（响应时间）= 最终权重

动态权重计算：
  avg_rt = 最近 100 次请求的平均响应时间
  dynamic_weight = base_weight × (1000 / avg_rt)
  
示例：
  Service-A: base_weight=10, avg_rt=50ms  → final_weight = 10 × 20 = 200
  Service-B: base_weight=10, avg_rt=200ms → final_weight = 10 × 5  = 50
  → Service-A 获得 80% 流量，Service-B 获得 20% 流量
```

**优势**：
- 相比简单轮询：能自动感知后端服务的健康状态，慢节点自动降权
- 相比最少连接：不需要维护连接计数，实现更轻量
- 响应时间采用**滑动窗口**统计（最近 100 次），避免历史数据干扰

#### 模块四：JWT 认证鉴权

**认证流程**：

```
┌────────┐  1.登录请求   ┌──────────┐  2.验证凭证  ┌──────────┐
│ Client  │────────────→│ Gateway   │────────────→│ Auth Svc  │
│         │             │           │             │           │
│         │  4.返回Token │           │  3.返回用户  │           │
│         │←────────────│           │←────────────│           │
└────┬───┘             └──────────┘             └──────────┘
     │
     │  5.携带Token请求
     ▼
┌──────────┐  6.JWT验证   ┌──────────┐  7.转发请求  ┌──────────┐
│ Client    │────────────→│ Gateway   │────────────→│ Backend   │
│           │             │ (Pre-Route│             │ Service   │
│           │             │  JWT插件) │             │           │
└──────────┘             └──────────┘             └──────────┘
```

**关键设计**：
- **Token 结构**：Header（算法） + Payload（用户ID、角色、过期时间） + Signature（HMAC-SHA256 签名）
- **双 Token 机制**：Access Token（有效期 30 分钟）+ Refresh Token（有效期 7 天）
- **Token 黑名单**：用户登出或 Token 被撤销时，将 Token ID 写入 Redis 黑名单（TTL = Token 剩余有效期）
- **网关层验证**：JWT 验证在网关 Pre-Route 阶段完成，后端服务无需重复验证，解析后的用户信息通过 Header 透传

#### 模块五：灰度发布

**设计方案**：

```
请求 Header: env=gray
    │
    ▼
┌──────────────────────────────────┐
│         Route 插件                │
│                                   │
│  1. 检查 Header 中是否有灰度标签   │
│  2. 有标签 → 路由到灰度实例        │
│  3. 无标签 → 路由到正式实例        │
│                                   │
│  灰度规则：                        │
│  ├── Header 标签匹配（env=gray）   │
│  ├── 用户 ID 白名单               │
│  └── 流量百分比（如 10% 灰度）     │
└──────────────────────────────────┘
```

**灰度策略**：
| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **Header 标签** | 请求携带 `env=gray` 路由到灰度环境 | 内部测试、指定用户灰度 |
| **用户白名单** | 指定用户 ID 列表路由到灰度环境 | 核心用户优先体验 |
| **流量百分比** | 按比例（如 10%）随机路由到灰度环境 | 大规模灰度验证 |

**零中断切换**：灰度规则通过 Etcd 配置中心下发，网关 Watch 实时生效，无需重启，流量切换过程对用户完全透明。

#### 模块六：黑白名单（可删除布隆过滤器）

**问题**：传统 HashSet 存储黑名单 IP，100 万 IP 需要约 **64MB** 内存，且不支持高效的批量判断。

**方案**：采用**可删除布隆过滤器（Counting Bloom Filter）** 替代 HashSet。

**布隆过滤器 vs HashSet 对比**：

| 指标 | HashSet | 布隆过滤器 | 提升 |
|------|---------|-----------|------|
| 100 万 IP 内存占用 | ~64MB | **~6MB** | **节省 90%** |
| 查询时间复杂度 | O(1) | O(k)，k 为哈希函数数 | 相当 |
| 误判率 | 0 | **<0.01%** | 可接受 |
| 支持删除 | ✅ | ✅（Counting 变体） | — |

**为什么选择"可删除"布隆过滤器**：
- 标准布隆过滤器不支持删除操作（删除会影响其他元素的判断）
- **Counting Bloom Filter** 将每个 bit 扩展为 **4-bit 计数器**，支持安全删除
- 内存开销比标准布隆过滤器多 4 倍，但仍远小于 HashSet（~24MB vs ~64MB）

**实现要点**：
- 使用 **3 个独立哈希函数**（MurmurHash3 + CityHash + FNV-1a），降低碰撞概率
- 误判率公式：`(1 - e^(-kn/m))^k`，其中 k=3, n=100万, m=1000万 → 误判率 ≈ 0.008%
- 黑名单更新通过 Etcd 配置中心下发，网关实时重建过滤器

## 三、面试高频问题准备

### Q1：为什么要自研网关而不用 Kong / Spring Cloud Gateway？

> 主要有三个原因：
>
> 1. **性能需求**：我们的场景需要极低延迟（<5ms 网关开销），Kong 基于 OpenResty（Lua），Spring Cloud Gateway 基于 WebFlux，在我们的压测中自研 Netty 网关性能最优
>
> 2. **定制化需求**：需要深度集成平台的认证体系、灰度发布策略、以及特定的黑白名单算法（可删除布隆过滤器），现有网关的插件机制不够灵活
>
> 3. **技术栈统一**：团队主要使用 Java/Go，自研网关便于团队维护和迭代，不需要额外学习 Lua（Kong）或 Go（Traefik）

### Q2：Netty 的 Reactor 模型是怎么工作的？为什么能支撑 2 万+ 连接？

> Netty 采用**主从 Reactor 多线程模型**：
>
> - **Boss Group**（1 个线程）：只负责接收新连接（Accept），将新连接注册到 Worker Group
> - **Worker Group**（CPU核数 × 2 个线程）：负责所有已建立连接的 IO 读写和业务处理
>
> 能支撑 2 万+ 连接的核心原因是**非阻塞 IO + 事件驱动**：
> - 传统 BIO 模型：1 个连接 = 1 个线程，2 万连接需要 2 万线程，线程切换开销巨大
> - Netty NIO 模型：1 个线程通过 **Selector** 多路复用监听多个连接的 IO 事件，只在有数据可读/可写时才处理
> - 实际上 16 个 Worker 线程就能处理 2 万+ 连接，因为大部分连接在大部分时间都是空闲的
>
> 另外，**IdleStateHandler** 会自动关闭超过 60s 无数据的空闲连接，防止连接泄漏占满资源。

### Q3：责任链模式的插件机制是怎么设计的？

> 核心思路是将请求处理流程拆分为**三个阶段**，每个阶段可以挂载多个插件：
>
> 1. **Pre-Route**：请求到达后、路由转发前执行。包括 IP 限流、黑白名单检查、JWT 认证、请求日志等
> 2. **Route**：路由匹配 + 负载均衡 + 灰度路由，将请求转发到后端服务
> 3. **Post-Route**：收到后端响应后执行。包括响应改写、指标采集等
>
> 每个插件实现统一的 `GatewayPlugin` 接口，通过 `phase()` 指定执行阶段，`order()` 指定优先级。插件链在启动时按阶段和优先级排序，请求到达后依次执行。
>
> **动态加载**通过 Java SPI 机制实现：新插件只需实现接口并在 `META-INF/services` 中注册，网关通过 `ServiceLoader` 扫描加载，无需修改网关代码或重启。
>
> 如果某个插件返回 `false`（如认证失败），则**中断链路**，直接返回错误响应，不再执行后续插件。

### Q4：负载均衡的"响应时间自适应"是怎么实现的？

> 传统加权轮询的权重是静态配置的，无法感知后端服务的实时状态。我设计的算法在静态权重基础上叠加了**动态权重**：
>
> 1. 网关为每个后端实例维护一个**滑动窗口**（最近 100 次请求的响应时间）
> 2. 每次路由时，计算 `最终权重 = 基础权重 × (1000 / 平均响应时间)`
> 3. 响应时间越短的实例，最终权重越高，获得的流量越多
>
> 举个例子：Service-A 平均响应 50ms，Service-B 平均响应 200ms，那么 A 获得 80% 流量，B 获得 20%。
>
> 这样当某个实例因为 GC、磁盘 IO 等原因变慢时，流量会**自动倾斜**到健康实例，无需人工干预。
>
> 滑动窗口用的是**环形数组**（固定 100 个槽位），内存开销极小，计算复杂度 O(1)（维护累加和，新值入队时加新减旧）。

### Q5：为什么黑白名单用布隆过滤器而不是 Redis Set？

> 对比了三种方案：
>
> | 方案 | 内存（100万IP） | 查询延迟 | 支持删除 |
> |------|----------------|---------|---------|
> | 本地 HashSet | ~64MB | <1μs | ✅ |
> | Redis Set | ~64MB（Redis端） | ~1ms（网络） | ✅ |
> | Counting Bloom Filter | **~24MB** | <1μs | ✅ |
>
> 选择布隆过滤器的原因：
> 1. **内存节省**：网关是流量入口，每个请求都要检查黑名单，必须在本地内存中完成（不能走网络到 Redis），布隆过滤器比 HashSet 节省 60%+ 内存
> 2. **误判可接受**：0.01% 的误判率意味着 1 万个正常请求中只有 1 个被误拦，对于安全防护场景完全可接受（误拦比漏拦好）
> 3. **支持删除**：使用 Counting Bloom Filter（4-bit 计数器）而非标准布隆过滤器，支持安全删除 IP
>
> 黑名单更新流程：管理员通过 API 更新黑名单 → 写入 Etcd → 网关 Watch 到变更 → 重建本地布隆过滤器（100 万 IP 重建耗时 <500ms）。

### Q6：灰度发布是怎么做到零中断的？

> 灰度规则存储在 **Etcd 配置中心**，网关通过 Watch 机制实时监听规则变更：
>
> 1. 运维人员通过管理 API 更新灰度规则（如"10% 流量路由到 v2 版本"）
> 2. 规则写入 Etcd
> 3. 网关收到 Watch 事件，更新内存中的灰度规则
> 4. 后续请求立即按新规则路由
>
> 整个过程**无需重启网关**，规则生效延迟 <100ms（Etcd Watch 延迟）。
>
> 灰度路由在 Route 阶段执行，优先级高于普通路由。匹配灰度规则的请求路由到灰度实例，不匹配的走正式实例。两套实例完全隔离，互不影响。
